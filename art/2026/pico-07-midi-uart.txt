Pico — Part 7: UART MIDI Input
2026-06-01


MIDI was designed in 1983 for a world of 5-pin DIN connectors, optoisolators, and
31250-baud serial lines. The byte format has barely changed since then. Getting MIDI
into the Pico is straightforward - UART0 or UART1 configured at 31250 baud, an IRQ
handler that collects bytes as they arrive, a state machine that assembles bytes into
messages, and a ring buffer that gets those messages to the audio core without locking.
The tricky parts are running status (MIDI's byte-saving shorthand), the ring buffer
discipline between cores, and the GPIO pin assignment. This article covers all of it.


What You Will Learn


Configuring UART at 31250 baud with uart_init, GPIO assignment, and RX interrupt enable

MIDI byte-stream state machine: status bytes, data bytes, running status, System Realtime
Handling note on/off, CC, and pitch bend messages
A lock-free single-producer single-consumer ring buffer for cross-core message passing
Complete example: MIDI note on/off drives oscillator frequency and gate on core 1


UART Setup at 31250 Baud


MIDI uses 31250 baud, 8 data bits, no parity, 1 stop bit - abbreviated 8N1. The RP2040
has two UARTs. UART0 defaults to GPIO 0 (TX) and GPIO 1 (RX); UART1 defaults to GPIO 4
(TX) and GPIO 5 (RX). Any UART-capable GPIO can be reassigned using
gpio_set_function. We use UART1 on GPIO 5 (RX only - MIDI input is
receive-only in the simplest case) to avoid conflict with the USB serial UART0 debug
path.



#include "hardware/uart.h"
#include "hardware/gpio.h"
#include "hardware/irq.h"

#define MIDI_UART        uart1
#define MIDI_UART_IRQ    UART1_IRQ
#define MIDI_BAUD        31250
#define MIDI_RX_GPIO     5    /* GPIO 5 = UART1 RX, pin 7 on Pico */

static void midi_uart_init(void) {
    uart_init(MIDI_UART, MIDI_BAUD);

    /* Assign GPIO 5 to UART1 RX function */
    gpio_set_function(MIDI_RX_GPIO, GPIO_FUNC_UART);

    /* 8 data bits, 1 stop bit, no parity (MIDI spec) */
    uart_set_format(MIDI_UART, 8, 1, UART_PARITY_NONE);

    /* Disable hardware flow control (MIDI doesn't use it) */
    uart_set_hw_flow(MIDI_UART, false, false);

    /* Enable the UART RX FIFO interrupt.
       Fire when the RX FIFO reaches 1/8 full (4 bytes on RP2040's 32-byte FIFO),
       which at 31250 baud gives us ~1 ms before the FIFO could overflow.
       MIDI is slow enough that even 1/2-full is fine; 1/8 is safe margin. */
    uart_set_fifo_enabled(MIDI_UART, true);
    irq_set_exclusive_handler(MIDI_UART_IRQ, midi_uart_irq_handler);
    irq_set_enabled(MIDI_UART_IRQ, true);
    uart_set_irq_enables(MIDI_UART, true, false); /* RX only */
}



The RP2040's UART hardware contains a 32-byte receive FIFO. When you enable the FIFO
the IRQ fires at a programmable fill level - the UART_UARTIFLS register controls this.
The SDK sets it via the FIFO level parameter to uart_set_fifo_enabled. With
FIFO enabled and the interrupt at 1/8 level (4 bytes), the IRQ fires before the FIFO
fills, giving the handler time to drain it. At 31250 baud, one byte takes
320 microseconds - a 4-byte FIFO level means you have 1.28 ms of slack before overflow.
That is generous.


The UART RX IRQ Handler


The handler runs on core 0 (because midi_uart_init is called from core 0
main). It drains the hardware FIFO into the ring buffer as fast as possible and returns.
No parsing happens here.



/* Ring buffer for raw MIDI bytes from IRQ to parser.
   Power-of-two size enables cheap index masking. */
#define RAW_BUF_SIZE  128
#define RAW_BUF_MASK  (RAW_BUF_SIZE - 1)

static volatile uint8_t  raw_buf[RAW_BUF_SIZE];
static volatile uint32_t raw_head = 0;   /* written by IRQ  */
static volatile uint32_t raw_tail = 0;   /* read by parser  */

static void midi_uart_irq_handler(void) {
    /* Drain all available bytes from hardware FIFO */
    while (uart_is_readable(MIDI_UART)) {
        uint8_t byte = uart_getc(MIDI_UART);

        /* Drop byte if ring buffer is full (should not happen at MIDI rates) */
        uint32_t next_head = (raw_head + 1) & RAW_BUF_MASK;
        if (next_head != raw_tail) {
            raw_buf[raw_head] = byte;
            raw_head = next_head;  /* write index last - memory barrier below */
        }
    }
    /* Compiler barrier: prevent reordering of the raw_head store past this point.
       On Cortex-M0+ all stores are strongly ordered, but the compiler is not. */
    __dmb();
}



The __dmb() (Data Memory Barrier) is important for the cross-core visibility of
raw_head. The Cortex-M0+ memory model guarantees that writes to normal
(non-device) memory are sequentially consistent from the perspective of one core, but
when core 0 updates raw_head and core 1 reads it, a memory barrier ensures
the store of raw_buf[raw_head] is visible before the store of
raw_head itself. Without it, the compiler might reorder the writes and core 1
could see the updated head index before the byte data.


MIDI Byte-Stream State Machine


MIDI is a byte stream with two categories of byte: status bytes (bit 7 set,
0x80-0xFF) and data bytes (bit 7 clear, 0x00-0x7F). A status byte begins a
new message and encodes the message type in bits 6-4 and the MIDI channel in bits 3-0.
Data bytes follow with the message's parameters.



Running status: if a second message of the same type and channel follows without
a status byte, the status byte is omitted. A data byte arriving when the previous
message has been completed is treated as the first data byte of the same message type.
This is running status. A parser that ignores it will miss most real-world note-on
streams.



System Realtime: bytes 0xF8 (Clock), 0xFA (Start), 0xFB (Continue), 0xFC (Stop),
0xFE (Active Sensing), and 0xFF (Reset) are single-byte messages that can appear
anywhere in the stream, even in the middle of a two-byte message. They must be
processed immediately without disrupting the current parse state.



/* Parsed MIDI message */
typedef struct {
    uint8_t type;     /* 0x80=note_off, 0x90=note_on, 0xB0=CC, 0xE0=pitch_bend */
    uint8_t channel;  /* 0-15 */
    uint8_t data1;
    uint8_t data2;
} MidiMsg;

/* Expected data byte counts per status nybble (high 4 bits of status byte) */
static const uint8_t midi_data_len[8] = {
    2, /* 0x80 Note Off          */
    2, /* 0x90 Note On           */
    2, /* 0xA0 Poly Aftertouch   */
    2, /* 0xB0 Control Change    */
    1, /* 0xC0 Program Change    */
    1, /* 0xD0 Channel Pressure  */
    2, /* 0xE0 Pitch Bend        */
    0, /* 0xF0 System (variable) */
};

typedef struct {
    uint8_t  status;      /* current status byte (running status) */
    uint8_t  data[2];     /* accumulated data bytes */
    int      data_count;  /* how many data bytes accumulated */
    int      data_needed; /* how many we need for this message */
} MidiParser;

/* Feed one byte into the parser.
   If a complete message is assembled, writes it to *out and returns 1.
   Returns 0 otherwise. */
static int midi_parser_feed(MidiParser *p, uint8_t byte, MidiMsg *out) {
    /* System Realtime (0xF8-0xFF): single byte, ignore here */
    if (byte = 0xF8) {
        /* Handle clock, transport etc. if needed - for now discard */
        return 0;
    }

    if (byte & 0x80) {
        /* Status byte: begin new message */
        /* Ignore SysEx (0xF0-0xF7) for simplicity */
        if (byte >= 0xF0) {
            p->status     = 0;   /* clear running status on system common */
            p->data_count = 0;
            return 0;
        }
        p->status      = byte;
        p->data_count  = 0;
        p->data_needed = midi_data_len[(byte >> 4) & 0x07];
        return 0;
    }

    /* Data byte */
    if (p->status == 0) return 0;   /* no running status, ignore */

    p->data[p->data_count++] = byte;

    if (p->data_count >= p->data_needed) {
        /* Complete message */
        out->type    = p->status & 0xF0;
        out->channel = p->status & 0x0F;
        out->data1   = p->data[0];
        out->data2   = (p->data_needed > 1) ? p->data[1] : 0;
        /* Running status: reset count but keep status byte */
        p->data_count = 0;
        return 1;
    }
    return 0;
}


Lock-Free Ring Buffer: Core 0 to Core 1


The MIDI parser runs on core 0 (where the UART IRQ fires). The synthesiser runs on
core 1. We need to pass MidiMsg structs from core 0 to core 1 without
using a mutex - mutexes can stall the audio core. A single-producer single-consumer
(SPSC) ring buffer is safe without locks when one core writes and the other reads,
provided the head and tail indices are updated atomically (they are, for 32-bit aligned
words on Cortex-M0+) and memory barriers are used at the publish/consume points.



#define MSG_BUF_SIZE  32
#define MSG_BUF_MASK  (MSG_BUF_SIZE - 1)

static MidiMsg          msg_buf[MSG_BUF_SIZE];
static volatile uint32_t msg_head = 0;   /* written by core 0 */
static volatile uint32_t msg_tail = 0;   /* written by core 1 */

/* Push one message from core 0 (parser thread).
   Returns 0 if buffer full (drop the message). */
static int midi_msg_push(const MidiMsg *m) {
    uint32_t next = (msg_head + 1) & MSG_BUF_MASK;
    if (next == msg_tail) return 0;   /* full */
    msg_buf[msg_head] = *m;
    __dmb();                          /* ensure data written before head update */
    msg_head = next;
    return 1;
}

/* Pop one message from core 1 (audio thread).
   Returns 1 if a message was available, 0 if empty. */
static int midi_msg_pop(MidiMsg *m) {
    if (msg_tail == msg_head) return 0;   /* empty */
    *m = msg_buf[msg_tail];
    __dmb();                              /* ensure data read before tail update */
    msg_tail = (msg_tail + 1) & MSG_BUF_MASK;
    return 1;
}


Core 0: Parse Loop


/* Run on core 0 - call this in the core 0 main loop */
static void midi_poll(void) {
    static MidiParser parser = {0};
    MidiMsg msg;

    /* Drain raw byte ring buffer and feed parser */
    while (raw_tail != raw_head) {
        uint8_t byte = raw_buf[raw_tail];
        __dmb();
        raw_tail = (raw_tail + 1) & RAW_BUF_MASK;

        if (midi_parser_feed(&parser, byte, &msg)) {
            midi_msg_push(&msg);
        }
    }
}


Core 1: Handling MIDI Messages


/* MIDI note number to frequency (Hz), standard equal temperament */
static float midi_note_to_freq(uint8_t note) {
    /* A4 = note 69 = 440 Hz */
    return 440.0f * powf(2.0f, (note - 69) / 12.0f);
}

/* Call this from the core 1 fill loop, once per buffer, before DSP */
static void midi_apply_events(Oscillator *osc, bool *gate) {
    MidiMsg msg;
    while (midi_msg_pop(&msg)) {
        switch (msg.type) {
        case 0x90:  /* Note On */
            if (msg.data2 > 0) {
                /* velocity > 0: genuine note on */
                osc_set_freq(osc, midi_note_to_freq(msg.data1), SAMPLE_RATE);
                *gate = true;
            } else {
                /* velocity 0 = note off (MIDI spec) */
                *gate = false;
            }
            break;
        case 0x80:  /* Note Off */
            *gate = false;
            break;
        /* CC and pitch bend handling can be added here */
        }
    }
}



Note that midi_note_to_freq calls powf - roughly 200 cycles.
This is fine because it happens at most once per event, never per sample. If you want
to avoid all soft-float in the event handler, pre-compute a 128-entry frequency table
during init:



static float midi_freq_table[128];

static void midi_freq_table_init(void) {
    for (int i = 0; i < 128; i++)
        midi_freq_table[i] = 440.0f * powf(2.0f, (i - 69) / 12.0f);
}

Summary


UART MIDI on the RP2040 is four pieces that compose cleanly: UART hardware configured at
31250 baud, an IRQ handler that drains the hardware FIFO into a raw byte buffer, a MIDI
state machine that handles running status and assembles messages on core 0, and a
lock-free SPSC ring buffer that carries those messages to the audio core without ever
stalling it. The memory barrier (__dmb()) at each ring buffer publish and
consume point is the only synchronisation primitive needed between the cores for this
pattern.



Part 8 adds USB MIDI using TinyUSB, so the Pico can present itself as a class-compliant
MIDI device and receive note events over USB without a physical UART cable.
