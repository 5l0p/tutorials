Go - Part 19: Vertical Slice Architecture

The conventional way to organise a backend service is by technical layer: handlers in one package, services (business logic) in another, repositories (storage) in a third. Every feature cuts horizontally across all three layers. Adding a new endpoint means touching three packages, three test files, and three sets of interfaces. The alternative, called vertical slice architecture, flips the axis: you organise by feature, and each feature package contains its own handler, logic, and storage. This article explains the idea, shows a concrete Go HTTP service organised as vertical slices, and discusses when it is the right choice and when it is not.

What You Will Learn

  • What vertical slice architecture is and how it differs from layered architecture.
  • How to identify the boundaries of a slice in a real Go HTTP service.
  • What a complete slice looks like: handler, logic, storage in one package.
  • How shared code (database connection, middleware, error types) is handled.
  • When vertical slices beat layered architecture - and when they do not.
  • How this relates to Go's package model from Part 18.

Layered Architecture: The Starting Point

Most developers start with a layered layout because it matches how frameworks are taught. In C# ASP.NET projects the template scaffolds Controllers, Services, and Repositories. The same convention appears in Go tutorials: handler/, service/, repository/. It looks tidy on day one.

The problems emerge as the project grows. Suppose you have a URL shortener with two features: creating short links and looking up the original URL. A layered layout looks like this: 

internal/
    handler/
        create.go      ← handler for POST /links
        resolve.go     ← handler for GET /:code
    service/
        links.go       ← CreateLink, ResolveLink business logic
    repository/
        links.go       ← SQL queries for links table

When you add a third feature - user accounts - you add to all three packages. When you read the code for the "create link" feature, you jump between three directories. When you delete a feature, you hunt through three packages to find everything to remove.

The deeper issue: packages in a layered layout share names (handler, service) that describe their technical role, not their purpose. If service/links.go and service/users.go both grow large, you cannot split them into separate packages without restructuring the whole layer.

Vertical Slices: The Alternative

A vertical slice is one package per feature, containing everything needed to implement that feature end-to-end: the HTTP handler, the business logic, and the storage queries. The name of the package describes the feature, not the technical role. 

internal/
    links/
        handler.go     ← HTTP handlers for /links
        service.go     ← CreateLink, ResolveLink
        store.go       ← SQL queries for links table
        handler_test.go
        service_test.go
    users/
        handler.go
        service.go
        store.go
    shared/
        db.go          ← *sql.DB shared across slices
        middleware.go  ← auth, logging middleware
        errors.go      ← common error types

Everything for the "links" feature is in links/. When a product manager asks "what happens when someone creates a short link?" you read one package. When you delete the feature, you delete one directory.

A Worked Example

Let us build the links slice of a URL shortener as a complete, runnable example.

First, the shared infrastructure - the database wrapper and common error type: 

// internal/shared/db.go
package shared

import (
    "database/sql"
    _ "github.com/lib/pq"
)

func OpenDB(dsn string) (*sql.DB, error) {
    db, err := sql.Open("postgres", dsn)
    if err != nil {
        return nil, err
    }
    if err := db.Ping(); err != nil {
        return nil, err
    }
    return db, nil
}

// internal/shared/errors.go
package shared

import (
    "errors"
    "net/http"
)

// AppError is an error that carries an HTTP status code.
type AppError struct {
    Code    int
    Message string
    Err     error
}

func (e *AppError) Error() string { return e.Message }
func (e *AppError) Unwrap() error { return e.Err }

var ErrNotFound = &AppError{Code: http.StatusNotFound, Message: "not found"}
var ErrConflict = &AppError{Code: http.StatusConflict, Message: "already exists"}

Now the links slice. Start with the storage layer - just the SQL it needs: 

// internal/links/store.go
package links

import (
    "context"
    "database/sql"
    "errors"
    "fmt"
    "example.com/shortener/internal/shared"
)

// Link represents a stored short link.
type Link struct {
    Code string
    URL  string
}

// store wraps *sql.DB with link-specific queries.
type store struct{ db *sql.DB }

func (s *store) insert(ctx context.Context, code, url string) error {
    _, err := s.db.ExecContext(ctx,
        `INSERT INTO links (code, url) VALUES ($1, $2)`, code, url)
    if err != nil {
        // detect Postgres unique violation
        if isDuplicateKey(err) {
            return shared.ErrConflict
        }
        return fmt.Errorf("links.store.insert: %w", err)
    }
    return nil
}

func (s *store) findByCode(ctx context.Context, code string) (Link, error) {
    var l Link
    err := s.db.QueryRowContext(ctx,
        `SELECT code, url FROM links WHERE code = $1`, code).
        Scan(&l.Code, &l.URL)
    if errors.Is(err, sql.ErrNoRows) {
        return Link{}, shared.ErrNotFound
    }
    if err != nil {
        return Link{}, fmt.Errorf("links.store.findByCode: %w", err)
    }
    return l, nil
}

func isDuplicateKey(err error) bool {
    // simplified; real code inspects the pq error code
    return err != nil && len(err.Error()) > 0
}

The business logic in the same package: 

// internal/links/service.go
package links

import (
    "context"
    "crypto/rand"
    "encoding/base64"
    "fmt"
    "net/url"
)

// Service handles link creation and resolution.
type Service struct {
    store *store
}

// NewService creates a Service backed by the given database.
func NewService(db interface{ ExecContext(context.Context, string, ...any) (interface{}, error) }) *Service {
    // In practice, pass *sql.DB directly
    return &Service{}
}

func newService(s *store) *Service {
    return &Service{store: s}
}

// CreateLink generates a unique short code and stores the link.
func (svc *Service) CreateLink(ctx context.Context, rawURL string) (string, error) {
    if _, err := url.ParseRequestURI(rawURL); err != nil {
        return "", fmt.Errorf("invalid URL: %w", err)
    }
    code, err := randomCode(6)
    if err != nil {
        return "", err
    }
    if err := svc.store.insert(ctx, code, rawURL); err != nil {
        return "", err
    }
    return code, nil
}

// ResolveLink looks up the original URL for the given short code.
func (svc *Service) ResolveLink(ctx context.Context, code string) (string, error) {
    link, err := svc.store.findByCode(ctx, code)
    if err != nil {
        return "", err
    }
    return link.URL, nil
}

func randomCode(n int) (string, error) {
    b := make([]byte, n)
    if _, err := rand.Read(b); err != nil {
        return "", err
    }
    return base64.URLEncoding.EncodeToString(b)[:n], nil
}

The HTTP handler in the same package, registering its routes on an *http.ServeMux: 

// internal/links/handler.go
package links

import (
    "database/sql"
    "encoding/json"
    "errors"
    "net/http"
    "strings"
    "example.com/shortener/internal/shared"
)

// Register mounts the links feature's routes onto mux.
// This is the only symbol the feature exports to main.
func Register(mux *http.ServeMux, db *sql.DB) {
    svc := newService(&store{db: db})
    h := &handler{svc: svc}
    mux.HandleFunc("POST /links", h.create)
    mux.HandleFunc("GET /{code}", h.resolve)
}

type handler struct{ svc *Service }

type createRequest struct {
    URL string `json:"url"`
}

type createResponse struct {
    Code string `json:"code"`
}

func (h *handler) create(w http.ResponseWriter, r *http.Request) {
    var req createRequest
    if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
        http.Error(w, "bad request", http.StatusBadRequest)
        return
    }
    code, err := h.svc.CreateLink(r.Context(), req.URL)
    if err != nil {
        writeError(w, err)
        return
    }
    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(http.StatusCreated)
    json.NewEncoder(w).Encode(createResponse{Code: code})
}

func (h *handler) resolve(w http.ResponseWriter, r *http.Request) {
    code := strings.TrimPrefix(r.PathValue("code"), "/")
    url, err := h.svc.ResolveLink(r.Context(), code)
    if err != nil {
        writeError(w, err)
        return
    }
    http.Redirect(w, r, url, http.StatusFound)
}

func writeError(w http.ResponseWriter, err error) {
    var appErr *shared.AppError
    if errors.As(err, &appErr) {
        http.Error(w, appErr.Message, appErr.Code)
        return
    }
    http.Error(w, "internal server error", http.StatusInternalServerError)
}

And the entry point, which knows about every slice but nothing about their internals: 

// cmd/api/main.go
package main

import (
    "log"
    "net/http"
    "os"
    "example.com/shortener/internal/links"
    "example.com/shortener/internal/shared"
)

func main() {
    db, err := shared.OpenDB(os.Getenv("DATABASE_URL"))
    if err != nil {
        log.Fatal(err)
    }
    defer db.Close()

    mux := http.NewServeMux()
    links.Register(mux, db)
    // users.Register(mux, db) - adding a new slice is one line

    log.Println("listening on :8080")
    log.Fatal(http.ListenAndServe(":8080", mux))
}

Each slice exports exactly one function: Register. Main knows the feature exists; it does not know how the feature is implemented. Adding a new feature is adding a new directory and one line in main.

Shared Code in a Vertical Slice Architecture

The objection most often raised against vertical slices is: "what about code that is shared between features?" Shared code belongs in internal/shared/ (or internal/platform/, or whatever name fits). The rule is that slices import shared, but shared never imports a slice. The shared package contains infrastructure, not business logic.

What belongs in shared:

  • The database connection and transaction helpers.
  • Common error types and HTTP error writers.
  • Middleware (authentication, request logging, tracing).
  • Configuration loading.

What does not belong in shared:

  • Domain types that belong to one feature (Link, User).
  • Business rules (ValidateURL, HashPassword).
  • Anything that would require shared to import a slice.

When Vertical Slices Beat Layered Architecture

Vertical slices shine when:

  • Features are largely independent. A URL shortener's "create link" and "user accounts" share infrastructure but not logic.
  • The team works on features in parallel. Each slice is a self-contained unit; two developers can work on different slices without merge conflicts.
  • Features are added and removed regularly. Deleting a feature is deleting a directory.
  • The service boundary matches a feature boundary (microservice-ready structure).

Layered architecture is better when:

  • Features share complex business logic that genuinely belongs in a service layer.
  • Multiple features use the same data access pattern in a way that warrants a shared repository.
  • The team is small and the codebase is well-understood - then the navigation cost of jumping between layers is low.

In C# the equivalent of vertical slice architecture is the MediatR + feature folder pattern, popularised by Jimmy Bogard. Each request (command or query) lives in its own class with its own handler. The idea is the same - organise by feature, not by technical role - and the trade-offs are identical. Go's version uses packages and Register functions instead of MediatR's pipeline, but the philosophy is the same.

Summary

Vertical slice architecture organises code by feature rather than technical layer. Each feature package contains its own handler, service, and storage. The entry point wires slices together by calling their Register functions. Shared infrastructure lives in a shared package that slices import but never export to.

The pattern fits Go's package model naturally: Go's import cycle prevention encourages thinking about dependency direction from the start, and vertical slices make that direction explicit - shared goes down, slices go up to main. Combined with the project layout principles from Part 18, vertical slices give you a codebase that grows gracefully: each new feature is a new directory, and the rest of the code stays unchanged.
