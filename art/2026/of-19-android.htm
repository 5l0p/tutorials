openFrameworks - Part 19: Android

Android is the most complex target in the openFrameworks ecosystem. You are cross-compiling C++ through the NDK, navigating a Java/Kotlin activity lifecycle wrapper, dealing with an audio stack that changed substantially between Android 8 and Android 12, and producing an APK that must pass Google Play's 64-bit requirement. It is not a weekend afternoon - but the payoff is a creative-coding sketch that runs on a billion devices. This article maps the full path from NDK setup to a signed APK on a real device.

What You Will Learn

  • NDK and Android Studio setup for oF Android development.
  • The Android Studio project structure oF generates and where your C++ code lives.
  • The ofxAndroid activity lifecycle and how it maps to oF's setup/update/draw.
  • Touch input versus mouse events, and multi-touch handling.
  • OpenSL ES and AAudio backends and how oF selects between them.
  • Building an APK from the command line and deploying with adb.
  • Common pitfalls and how to diagnose them with adb logcat.

Prerequisites: NDK and Android Studio

Install Android Studio from developer.android.com. Inside Android Studio, open SDK Manager and install:

  • Android SDK Platform (API level 33 or newer for current store requirements)
  • NDK (Side by side) - version r25c is well-tested with oF 0.12.x
  • CMake 3.22.x

Set environment variables so oF's build scripts can locate the tools: 

export ANDROID_SDK_ROOT=$HOME/Library/Android/sdk   # macOS/Linux path
export ANDROID_NDK_ROOT=$ANDROID_SDK_ROOT/ndk/25.2.9519653
export PATH=$PATH:$ANDROID_SDK_ROOT/platform-tools  # adds adb

Generating an Android Project

The oF Android package ships with a project template. The project generator does not support Android; use the template directly: 

cp -r $OF_ROOT/apps/myApps/androidEmptyExample apps/myApps/mySketch

The resulting project layout: 

mySketch/
  app/
    src/
      main/
        cpp/
          ofApp.h
          ofApp.cpp
          main.cpp
        java/
          cc/openframeworks/OFActivity.java    (subclass this for Java-side code)
        AndroidManifest.xml
        res/
          values/
            strings.xml
  build.gradle
  local.properties     (path to SDK, auto-generated)
  CMakeLists.txt       (references the oF Android CMake build)

Your C++ code goes in app/src/main/cpp/. Assets go in app/src/main/assets/ (which maps to bin/data/). Use ofToDataPath() to access them - oF on Android extracts assets to a cache directory and ofToDataPath() returns that path.

The ofxAndroid Activity Lifecycle

Android apps are controlled by an Activity. When the user presses Home, your activity pauses; when they return, it resumes. The screen can be rotated, another app can steal audio focus, the system can kill your process entirely. oF maps this lifecycle to virtual methods you can override: 

// ofApp.h
#pragma once
#include "ofMain.h"
#include "ofxAndroid.h"

class ofApp : public ofxAndroidApp {
public:
    void setup();
    void update();
    void draw();

    // Android lifecycle events
    void pause();    // app moved to background - stop audio, release GPU resources
    void resume();   // app returned to foreground
    void reloadTextures();  // called after an OpenGL context loss (screen rotation)
    void deviceOrientationChanged(int newOrientation);
};

void ofApp::pause() {
    soundStream.stop();   // release audio focus
}

void ofApp::resume() {
    soundStream.start();
}

void ofApp::reloadTextures() {
    // OpenGL context was destroyed and recreated.
    // Re-upload all textures, reload all shaders.
    myShader.load("shaders/plasma");
    myTexture.load("images/bg.png");
}

Context loss is the most common Android gotcha. Rotating the screen, switching apps, or answering a call can destroy the OpenGL context. Any GPU resource (texture, FBO, shader, VBO) is invalidated. reloadTextures() is your chance to recreate them. If you do not implement it, you get black screens and GPU garbage after any context loss.

Touch vs Mouse Events

oF on Android maps single-finger touch to the standard mouse event methods (mousePressed, mouseDragged, mouseReleased). For multi-touch you use the ofxAndroid-specific callbacks: 

void ofApp::touchDown(int x, int y, int id) {
    // id is the touch point index (0 = first finger, 1 = second, etc.)
}

void ofApp::touchMoved(int x, int y, int id) {
}

void ofApp::touchUp(int x, int y, int id) {
}

void ofApp::touchDoubleTap(int x, int y, int id) {
}

Touch coordinates are in screen pixels. The screen resolution on a modern Android device is typically 2× or 3× the logical resolution reported by display metrics. Use ofGetWidth() and ofGetHeight() for the logical size; these are what oF's coordinate system uses, and they match what ofDrawCircle and your shaders expect.

Audio: OpenSL ES and AAudio

Android has had two C-level audio APIs: OpenSL ES (available since Android 2.3) and AAudio (available since Android 8.0, API level 26). oF uses RTAudio under ofSoundStream, and RTAudio selects AAudio on API 26+ and falls back to OpenSL ES on older devices.

AAudio supports lower latency (FAST track, down to about 5 ms on modern hardware) when you use a buffer size that matches the device's preferred burst size: 

// Request the device's optimal buffer size
ofSoundStreamSettings ss;
ss.setOutListener(this);
ss.sampleRate        = 48000;    // Android devices prefer 48000 Hz
ss.numOutputChannels = 2;
// Set bufferSize to 0 to let oF/RTAudio negotiate the optimal size with AAudio:
ss.bufferSize = 0;
soundStream.setup(ss);

Audio focus: Android requires apps to request audio focus before playing sound and to release it when backgrounded. oF handles this internally via ofxAndroid when you call soundStream.stop() in pause() and soundStream.start() in resume().

Microphone permission: Add to AndroidManifest.xml: 

<uses-permission android:name="android.permission.RECORD_AUDIO" />

Then request it at runtime (required on Android 6.0+). oF provides a helper: 

// In setup() or in response to a button press:
ofxAndroidRequestPermission("android.permission.RECORD_AUDIO");

Building an APK from the Command Line

Android Studio wraps Gradle. You can build from a terminal without opening the IDE: 

# From the project root (where build.gradle lives):
./gradlew assembleDebug       # debug APK (signed with debug key, faster)
./gradlew assembleRelease     # release APK (needs signing, see below)
./gradlew installDebug        # build debug + push to connected device/emulator

The debug APK is at app/build/outputs/apk/debug/app-debug.apk.

For a release APK you need a keystore: 

# Create a keystore (one-time):
keytool -genkey -v -keystore mysketch.keystore \
        -alias mysketch -keyalg RSA -keysize 2048 -validity 10000

# Sign the release APK:
./gradlew assembleRelease
zipalign -v 4 app/build/outputs/apk/release/app-release-unsigned.apk \
              app-release-aligned.apk
apksigner sign --ks mysketch.keystore --out mysketch-release.apk \
               app-release-aligned.apk

Deploying with adb

# List connected devices:
adb devices

# Install APK:
adb install -r mysketch-release.apk   # -r replaces existing install

# Uninstall:
adb uninstall cc.openframeworks.mysketch   # package name from AndroidManifest

# Push a file to the device (e.g. a preset):
adb push preset.yaml /sdcard/Android/data/cc.openframeworks.mysketch/files/

# Pull a file from the device:
adb pull /sdcard/Android/data/cc.openframeworks.mysketch/files/recording.wav .

Debugging with adb logcat

All ofLog output and C++ crashes appear in the Android log. Filter to your app: 

adb logcat --pid=$(adb shell pidof cc.openframeworks.mysketch)
# or by tag:
adb logcat -s openframeworks

Native crashes (null pointer dereference, stack overflow) produce a tombstone in the log. Look for lines starting with backtrace:. Symbolicate the addresses with ndk-stack: 

adb logcat | ndk-stack -sym app/build/intermediates/cmake/debug/obj/arm64-v8a/

This replaces hex addresses with source file names and line numbers, making native crashes as readable as C++ exceptions.

Common Pitfalls

Black screen after rotation. OpenGL context was destroyed. Implement reloadTextures() and recreate all GPU resources.

No audio on Android 6+. RECORD_AUDIO permission not granted at runtime. Call ofxAndroidRequestPermission before opening the sound stream.

Assets not found. On Android, bin/data/ maps to the APK's assets/ folder. oF extracts assets to a cache directory on first run. Always use ofToDataPath(); never hardcode paths.

High latency audio. Request the device's native sample rate (usually 48000 Hz) and set bufferSize = 0 to let RTAudio/AAudio negotiate the burst size. Mismatched sample rates cause resampling in the OS, which adds latency.

64-bit requirement. Google Play requires 64-bit (arm64-v8a) APKs. The oF Android CMake build targets armeabi-v7a and arm64-v8a by default. Confirm in CMakeLists.txt; do not remove arm64-v8a.

Summary

Android development with oF requires NDK r25, Android Studio, and a Gradle-based project from the oF template. The ofxAndroidApp base class extends the standard ofBaseApp interface with lifecycle methods for pause/resume and context loss - implement reloadTextures() or face black screens. Touch is handled per-finger via touchDown/Moved/Up. Audio via AAudio on API 26+ gives low latency when you match the device's native sample rate. Build with ./gradlew assembleDebug, deploy with adb install, diagnose with adb logcat | ndk-stack. Part 20 unifies the platform knowledge from Parts 15-19 into portable coding patterns.
