openFrameworks - Part 17: Windows Builds

Windows is the platform most oF beginners start on, and it is also the platform with the most friction: DLL hell, path separator ambiguity, WASAPI's exclusive-mode quirks, and Visual Studio's project file format all add up to a day of setup before you write a single line of creative code. This article maps out the entire workflow - from project generation through release distribution - so that day becomes an hour.

What You Will Learn

  • The Visual Studio project generator workflow for oF on Windows.
  • The WASAPI audio backend and how to select devices and buffer sizes.
  • Path separator and Unicode pitfalls in C++ on Windows.
  • Which DLLs a release build needs, and how to find and bundle them.
  • Debugging with Visual Studio vs running a release build.
  • Common Windows-specific compile errors in oF projects and how to fix them.

Project Generator Workflow

oF on Windows uses the graphical project generator to create Visual Studio solution ( .sln) and project (.vcxproj) files. Download oF for Windows from openframeworks.cc - choose the Visual Studio variant, not the MSYS2 or MinGW variant, unless you specifically need GCC.

Launch projectGenerator/projectGenerator.exe. Set:

  • Name: your project name.
  • Path: the apps\myApps\ directory inside the oF root.
  • Addons: any addons you need.

Click Generate. Open the resulting .sln in Visual Studio 2022 (Community edition is free and sufficient). Set the build configuration to Release x64 for distribution, or Debug x64 for development. Press F5 to build and run.

Important: the oF root must not contain spaces in its path. Visual Studio handles spaces in paths badly in some Makefile-style custom build steps. Place oF at C:\of_v0.12.0\, not C:\Program Files\openframeworks\.

WASAPI Audio Backend

On Windows, ofSoundStream uses RTAudio's WASAPI backend. WASAPI (Windows Audio Session API) is the modern Windows audio API and supports two modes:

  • Shared mode (default): the audio engine mixes your stream with others. The engine enforces its own buffer size (typically 10–20 ms). You cannot request arbitrary buffer sizes below that floor.
  • Exclusive mode: your application has direct access to the hardware. Latency can reach 3–5 ms. The device is unavailable to other applications while yours is running. Enable with the RTAudio flag RTAUDIO_HOG_DEVICE in the stream options, which RTAudio exposes and oF passes through if you set it in ofSoundStreamSettings.
ofSoundStreamSettings ss;
ss.setInListener(this);
ss.sampleRate        = 48000;   // Windows devices often prefer 48000 Hz
ss.numInputChannels  = 2;
ss.numOutputChannels = 2;
ss.bufferSize        = 512;     // shared mode: actual size may be larger
soundStream.setup(ss);

Device enumeration on Windows returns names like "Microphone Array (Intel SST Audio)". Print the device list in setup() to find the index you need: 

auto devices = soundStream.getDeviceList();
for (const auto &d : devices) {
    ofLogNotice() << d.deviceID << ": " << d.name
                  << " in:" << d.inputChannels
                  << " out:" << d.outputChannels;
}

Path Separator Pitfalls

The Win32 API accepts forward slashes in most contexts, and oF's own code uses forward slashes throughout. The pitfalls arise in two specific places:

Concatenating paths with +: Never use string concatenation to build paths. Use ofToDataPath() for assets, and the std::filesystem (C++17) API for anything else: 

#include <filesystem>
namespace fs = std::filesystem;

fs::path base = ofToDataPath("recordings");
fs::path full = base / "take01.wav";   // operator/ appends with the platform separator
std::string s = full.string();         // "C:\...\recordings\take01.wav" on Windows

Passing paths to shell commands: system() and ShellExecute() invoke cmd.exe, which requires backslashes. Convert explicitly: 

fs::path p = ofToDataPath("tool.exe");
std::string cmdPath = p.string();
// Replace forward slashes with backslashes for cmd.exe:
std::replace(cmdPath.begin(), cmdPath.end(), '/', '\\');
std::system(("\"" + cmdPath + "\"").c_str());

Unicode and Wide Strings

Windows internally uses UTF-16. The char-based C++ API is either ANSI (system code page, no Unicode) or UTF-8 depending on settings. oF and most of its dependencies use std::string with UTF-8 encoding. This works correctly for asset names when you set the application's execution character set to UTF-8, which Visual Studio does since VS 2022 with the /utf-8 compiler flag.

The project generator sets /utf-8 for you. If you add source files that were generated by another tool and see garbled filenames or YAML parse errors on paths containing non-ASCII characters, check that the source file is saved as UTF-8 without BOM in your editor.

If you call Windows APIs directly (e.g., CreateFile), use the W (wide-string) variants and convert with std::wstring_convert or the C++17 std::filesystem::path constructor, which handles the conversion internally: 

#ifdef TARGET_WIN32
#include <windows.h>
fs::path p = ofToDataPath("data.bin");
HANDLE h = CreateFileW(p.wstring().c_str(), GENERIC_READ, 0, nullptr,
                       OPEN_EXISTING, FILE_ATTRIBUTE_NORMAL, nullptr);
#endif

Distributing an EXE: Which DLLs Are Needed?

A Release build of an oF app on Windows depends on a set of DLLs that must travel with the .exe. There are two categories:

Visual C++ runtime DLLs (MSVCP140.dll, VCRUNTIME140.dll, VCRUNTIME140_1.dll): required by any program compiled with MSVC. Install on the target machine with the free Visual C++ Redistributable package from Microsoft. This is the preferred approach for distribution (it keeps the system runtime up to date), but you can also copy the DLLs next to the .exe if you need a fully standalone package.

oF library DLLs: oF links several libraries dynamically on Windows. After a Release build, look in bin\ - the project generator copies the needed DLLs there: 

bin\
  myProjectRelease.exe
  freeimage.dll
  freetype.dll
  glew32.dll
  openal32.dll
  portaudio_x64.dll
  ...

To find any remaining missing DLLs, run Dependencies (the modern replacement for Dependency Walker, available at https://github.com/lucasg/Dependencies): 

Dependencies.exe -imports bin\myProjectRelease.exe

Any DLL listed as not found must be located and copied into bin\. The complete distribution archive is: 

myProject\
  myProjectRelease.exe
  *.dll               (all required DLLs)
  data\               (copy of bin\data\)

Debugging with Visual Studio

Visual Studio's debugger is excellent. Key workflow:

  • Set breakpoints by clicking the left margin of a source line. Press F5 to run with the debugger attached.
  • The Output window shows oF's ofLog messages and shader compile errors. Open it with View → Output.
  • The Immediate Window (Debug → Windows → Immediate) evaluates C++ expressions while paused at a breakpoint.
  • For audio callback crashes: the audio thread runs separately from the main thread. Set a breakpoint inside audioIn() or audioOut() - Visual Studio will pause correctly on the audio thread.

A Release build with /DEBUG:FULL generates a PDB (program database) file alongside the EXE. This lets the debugger resolve symbols in release builds, which is useful for crashes that only occur in optimised code.

Common Windows-Specific Compile Errors

C4251: class needs to have dll-interface. This is a warning from MSVC about exporting STL types across DLL boundaries. If you see it from oF or addon headers, suppress it: 

// At the top of the affected header:
#pragma warning(push)
#pragma warning(disable: 4251)
#include "ofxSomeAddon.h"
#pragma warning(pop)

min/max macro conflicts. <windows.h> defines min and max as macros, breaking std::min and std::max. Fix: 

#define NOMINMAX        // before including windows.h
#include <windows.h>
// or define it in your project preprocessor definitions in Visual Studio

Linker error LNK2019: unresolved external symbol. Usually means a library is not being linked. Check Project Properties → Linker → Input → Additional Dependencies. Addon libraries should be added by the project generator; if you added a DLL manually, you need the corresponding .lib import library.

Error C2338 or C7510 with std::filesystem on older Windows SDK. Requires Windows SDK 10.0.17763 or later and /std:c++17 or later. Set in Project Properties → General → C++ Language Standard.

Summary

The Windows workflow is: project generator creates the .sln, open in Visual Studio 2022, set x64 configuration, press F5. WASAPI shared mode is the default audio backend; exclusive mode gives lower latency. Use forward slashes in all paths except direct shell invocations. Set /utf-8 for Unicode correctness. Distribute the .exe alongside its DLLs from bin\ and the data\ folder. Suppress C4251 from third-party headers and define NOMINMAX before including windows.h. Part 18 covers macOS Xcode builds.
