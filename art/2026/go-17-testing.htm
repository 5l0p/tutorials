Go - Part 17: Testing in Go

Go ships with a testing framework in its standard library and a test runner built into the go command. There is no separate test project, no NuGet package to install, no choice between xUnit and NUnit - you write test functions in _test.go files and run go test. The result is that testing in Go has one canonical style, and reading another team's tests requires no mental translation. This article covers that style from the ground up: how the test runner works, the table-driven pattern that replaces parameterised tests, subtests, benchmarks, fuzz testing, and a handful of patterns that experienced developers reach for in practice.

What You Will Learn

  • How go test discovers and runs tests, and the key flags to know.
  • The table-driven test pattern - Go's idiomatic equivalent of xUnit parameterised tests.
  • Subtests with t.Run for structured, filterable test hierarchies.
  • Writing and interpreting benchmarks with testing.B.
  • Fuzz testing with go test -fuzz.
  • Test helpers, t.Helper(), and golden file patterns.

How go test Works

go test compiles each package together with any *_test.go files in that package and links a test binary. It then runs the binary. Test functions must have the signature func TestXxx(t *testing.T) where Xxx starts with an uppercase letter. There is no base class to extend, no attribute to attach - just the naming convention. 

// add.go
package calc

func Add(a, b int) int { return a + b }

// add_test.go
package calc

import "testing"

func TestAdd(t *testing.T) {
    got := Add(2, 3)
    if got != 5 {
        t.Errorf("Add(2, 3) = %d, want 5", got)
    }
}

$ go test ./...
ok      example.com/calc  0.002s

Key flags you will use daily: 

go test ./...                   # all packages
go test -v ./...                # verbose: print each test name and PASS/FAIL
go test -run TestAdd ./...      # run only tests matching the regex
go test -count=1 ./...          # disable test caching (always re-run)
go test -race ./...             # enable the data race detector
go test -timeout 30s ./...      # fail if tests take more than 30 seconds

In C# you run tests with dotnet test. The mechanics are similar but each test class carries [TestClass] / [TestFixture] attributes and each test method carries [TestMethod] / [Test]. Go's convention-based discovery requires less ceremony.

Test packages. By default, a test file is in the same package as the code under test (package calc), giving it access to unexported identifiers. For black-box testing, use package calc_test - the _test suffix is special: the Go toolchain allows it even though the directory's package is calc. Black-box tests can only use exported identifiers, which tests your public API as a user would.

Table-Driven Tests

The most distinctive Go testing pattern is the table-driven test. Rather than writing a separate test function for each case (as in xUnit's [TestCase] or NUnit's [TestCase(...)]), you define a slice of anonymous structs, each describing one case, and iterate: 

func TestAdd(t *testing.T) {
    tests := []struct {
        name string
        a, b int
        want int
    }{
        {"positive", 2, 3, 5},
        {"negative", -1, -1, -2},
        {"zero", 0, 0, 0},
        {"large", 1000000, 999999, 1999999},
    }

    for _, tc := range tests {
        got := Add(tc.a, tc.b)
        if got != tc.want {
            t.Errorf("%s: Add(%d, %d) = %d, want %d",
                tc.name, tc.a, tc.b, got, tc.want)
        }
    }
}

This pattern has several advantages over separate test functions: adding a case is one line, the test name appears in failure output, and the structure makes it obvious what the inputs and expected outputs are. In C# [Theory] with [InlineData] (xUnit) achieves the same goal, but the data is scattered across attributes on the function, which is harder to read when there are many cases.

Subtests with t.Run

Combining table-driven tests with t.Run gives you subtests - each case runs as a named, independently addressable test: 

for _, tc := range tests {
    tc := tc // capture loop variable (required before Go 1.22)
    t.Run(tc.name, func(t *testing.T) {
        got := Add(tc.a, tc.b)
        if got != tc.want {
            t.Errorf("Add(%d, %d) = %d, want %d", tc.a, tc.b, got, tc.want)
        }
    })
}

Now you can run a single subtest by name: 

go test -run TestAdd/negative ./...

Subtests can also be run in parallel. Mark the outer test and each subtest: 

func TestAdd(t *testing.T) {
    t.Parallel() // this test runs in parallel with other top-level tests

    tests := []struct{ name string; a, b, want int }{
        {"positive", 2, 3, 5},
        {"negative", -1, -1, -2},
    }

    for _, tc := range tests {
        tc := tc
        t.Run(tc.name, func(t *testing.T) {
            t.Parallel() // subtests also run in parallel with each other
            got := Add(tc.a, tc.b)
            if got != tc.want {
                t.Errorf("Add(%d, %d) = %d, want %d", tc.a, tc.b, got, tc.want)
            }
        })
    }
}

Note on loop variable capture: Before Go 1.22, the loop variable tc is shared across iterations. Without tc := tc inside the loop, all parallel subtests would use the last value of tc. Go 1.22 changed the semantics so that each iteration creates a new variable, making the re-declaration unnecessary - but it is still common in code that targets older versions.

Test Helpers and t.Helper()

When you extract repeated assertion logic into a helper function, failures report the line number inside the helper - not the line in the test that called it. t.Helper() fixes this: it marks the calling function as a test helper so the failure line points to the caller, not the helper body. 

// assertEqual is a reusable helper
func assertEqual(t *testing.T, got, want int) {
    t.Helper() // must be the first call
    if got != want {
        t.Errorf("got %d, want %d", got, want)
    }
}

func TestAdd(t *testing.T) {
    assertEqual(t, Add(2, 3), 5)    // failure points here
    assertEqual(t, Add(-1, -1), -2) // or here
}

In C# the equivalent is the Assertion methods in xUnit/NUnit - they use stack-walking to report the caller's line. Go's approach is explicit (t.Helper()) rather than implicit (stack inspection), which is consistent with Go's preference for visible control flow.

Using testify

The standard library's testing primitives are intentionally minimal. The most popular third-party addition is testify, which provides assertion functions and mock support: 

go get github.com/stretchr/testify

import (
    "testing"
    "github.com/stretchr/testify/assert"
    "github.com/stretchr/testify/require"
)

func TestSomething(t *testing.T) {
    result, err := DoSomething()

    require.NoError(t, err)         // stops the test immediately on failure
    assert.Equal(t, 42, result)     // continues even if this fails
    assert.Contains(t, "hello world", "hello")
}

The distinction between assert (continues on failure) and require (stops immediately, equivalent to t.Fatalf) mirrors xUnit's Assert.Equal vs the pattern of checking preconditions first. Use require for things that make subsequent assertions meaningless if they fail (e.g., "if this is nil, everything below will panic").

Benchmarks

A benchmark function has the signature func BenchmarkXxx(b *testing.B). The test runner calls it repeatedly, adjusting b.N until the results are stable: 

func BenchmarkAdd(b *testing.B) {
    for i := 0; i < b.N; i++ {
        Add(2, 3)
    }
}

$ go test -bench=BenchmarkAdd -benchmem ./...
BenchmarkAdd-8    1000000000    0.3 ns/op    0 B/op    0 allocs/op

The columns mean: the test ran 1,000,000,000 iterations; each iteration took 0.3 ns; each iteration allocated 0 bytes in 0 allocations. -benchmem adds the allocation columns, which are essential for catching unexpected heap escapes.

For benchmarks that need setup, use b.ResetTimer() after setup to exclude setup time from the measurement: 

func BenchmarkProcess(b *testing.B) {
    data := generateTestData(10000) // expensive setup
    b.ResetTimer()                  // start timing here

    for i := 0; i < b.N; i++ {
        Process(data)
    }
}

In C# microbenchmarking typically uses BenchmarkDotNet, a third-party package. Go's built-in benchmark runner is more basic but sufficient for most needs. Both avoid the common mistake of benchmarking without warmup - BenchmarkDotNet handles it automatically, while Go adjusts b.N until timing is stable.

Compare benchmarks across commits with benchstat: 

go install golang.org/x/perf/cmd/benchstat@latest

go test -bench=. -count=5 ./... > old.txt
# make your change
go test -bench=. -count=5 ./... > new.txt
benchstat old.txt new.txt

Fuzz Testing

Go 1.18 added native fuzz testing. A fuzz test explores random inputs automatically, looking for panics, unexpected errors, or violations of invariants. It is especially useful for parsers, serialisers, and any code that processes untrusted input. 

// fuzz_test.go
func FuzzParseURL(f *testing.F) {
    // Seed corpus - start from known good inputs
    f.Add("https://example.com/path?q=1")
    f.Add("http://localhost:8080")
    f.Add("")

    f.Fuzz(func(t *testing.T, s string) {
        // Must not panic on any input
        u, err := url.Parse(s)
        if err != nil {
            return // errors are OK, panics are not
        }
        // Round-trip invariant: parsing the string form must give the same result
        u2, err := url.Parse(u.String())
        if err != nil {
            t.Fatalf("round-trip parse failed: %v", err)
        }
        if u.String() != u2.String() {
            t.Fatalf("round-trip not idempotent: %q -> %q -> %q",
                s, u.String(), u2.String())
        }
    })
}

# Run fuzzer for 30 seconds
go test -fuzz=FuzzParseURL -fuzztime=30s ./...

# Run the seed corpus only (like a regular test - fast, deterministic)
go test -run=FuzzParseURL ./...

When the fuzzer finds a failing input it writes it to testdata/fuzz/FuzzParseURL/. Subsequent go test runs replay all saved corpus entries - so once you find a bug and fix it, the failing input becomes a permanent regression test.

C# has no built-in fuzzer. SharpFuzz wraps libFuzzer via P/Invoke. Go's integration is more first-class: the corpus is stored in the testdata directory, integrated with go test, and maintained alongside the code.

Golden Files

When the expected output of a function is large - a rendered template, a generated configuration, a serialised document - embedding it as a string literal in the test is unwieldy. The golden file pattern stores the expected output in a file: 

// render_test.go
package render

import (
    "flag"
    "os"
    "testing"
)

var update = flag.Bool("update", false, "update golden files")

func TestRenderInvoice(t *testing.T) {
    got := RenderInvoice(testInvoice)

    const golden = "testdata/invoice.golden"
    if *update {
        // Run with -update to regenerate when output intentionally changes
        if err := os.WriteFile(golden, []byte(got), 0644); err != nil {
            t.Fatal(err)
        }
        return
    }

    want, err := os.ReadFile(golden)
    if err != nil {
        t.Fatalf("missing golden file %s: run with -update", golden)
    }
    if got != string(want) {
        t.Errorf("output does not match golden file %s\ngot:\n%s", golden, got)
    }
}

Run go test -update ./... to regenerate all golden files at once after an intentional change. Check the golden files into version control - they are both the expected output and the documentation of what the function produces.

TestMain: Setup and Teardown

If a package's tests need one-time setup (starting a database, creating a temp directory), use TestMain: 

func TestMain(m *testing.M) {
    // setup
    db := startTestDatabase()

    // run all tests in this package
    code := m.Run()

    // teardown (runs even if tests fail)
    db.Stop()

    os.Exit(code)
}

TestMain is the package-level equivalent of xUnit's IClassFixture or NUnit's [OneTimeSetUp]. One key difference: in Go there is only one TestMain per package. Per-test setup goes inside the test function or in a helper called from each test.

Summary

Go testing is intentionally simple: conventions over configuration, one runner, one style. Table-driven tests replace parameterised test attributes; subtests with t.Run provide structure and filterability; t.Helper() makes helpers report errors at the right line; benchmarks and fuzzing ship with the toolchain.

The standard library gives you everything you need for unit tests. For richer assertions testify is the de facto standard third-party addition. For golden files, store expected output in testdata/ and update with a flag. For package-level setup and teardown, use TestMain.

Testing connects directly to the tooling article (Part 14): go test -race runs the race detector on every test, and the CI pipeline described there includes go test -race -count=1 ./... as its final step. Good test coverage, combined with the race detector and fuzz testing, gives Go programs a strong safety net without a separate test framework.
