DSP - Part 5: Biquad Filters
2026-08-29


The biquad filter is the single most useful building block in audio DSP. Almost every
useful filter response - lowpass, highpass, bandpass, notch, shelving EQ, peaking EQ -
can be realised as a biquad or a cascade of biquads. This article covers the mathematics,
the two main computational forms, numerical considerations, and a clean C implementation
you can use as the foundation for everything in the filter cookbook articles that follow.


What You Will Learn


The biquad transfer function H(z) and what the b and a coefficients mean
Direct Form I: conceptually clear, requires four state variables
Direct Form II Transposed: numerically superior, requires two state variables — the recommended form
How to store coefficients and state in a C struct and update them safely at runtime
Numerical stability concerns: coefficient precision and denormal contamination of filter state


The Transfer Function


A biquad is a second-order IIR (Infinite Impulse Response) filter. Its z-domain transfer
function is a ratio of two second-degree polynomials in z-1:



H(z) = (b0 + b1*z^-1 + b2*z^-2)
       / (1 + a1*z^-1 + a2*z^-2)



The five coefficients {b0, b1, b2, a1, a2} fully describe the filter. Different choices
of these coefficients give different frequency responses. The filter cookbook articles
show how to derive them for specific shapes.



Note the sign convention: some textbooks write the denominator as
1 - a1*z^-1 - a2*z^-2 with negated a coefficients. The Audio EQ Cookbook
(Robert Bristow-Johnson) and this article use the positive denominator convention
where a1 and a2 appear with negative signs in the difference equation.


Direct Form I


The most intuitive form is the direct translation from the difference equation to code.
For each output sample y[n]:



y[n] = b0*x[n] + b1*x[n-1] + b2*x[n-2]
             - a1*y[n-1] - a2*y[n-2]



This requires storing two past inputs (x1, x2) and two past
outputs (y1, y2). The implementation in C:



#include <math.h>

typedef struct {
    /* Coefficients */
    float b0, b1, b2;
    float a1, a2;
    /* State: Direct Form I */
    float x1, x2;  /* delayed inputs  */
    float y1, y2;  /* delayed outputs */
} BiquadDF1;

void biquad_df1_reset(BiquadDF1 *f) {
    f->x1 = f->x2 = 0.0f;
    f->y1 = f->y2 = 0.0f;
}

float biquad_df1_process(BiquadDF1 *f, float x) {
    float y = f->b0 * x
            + f->b1 * f->x1
            + f->b2 * f->x2
            - f->a1 * f->y1
            - f->a2 * f->y2;
    f->x2 = f->x1;
    f->x1 = x;
    f->y2 = f->y1;
    f->y1 = y;
    return y;
}



Direct Form I uses four multiply-accumulate operations and four state variable updates
per sample. It is straightforward to reason about and numerically behaves well for most
audio applications.


Direct Form II Transposed


Direct Form II Transposed (TDF-II) is preferred in professional audio because it uses
only two state variables instead of four, and the state is of smaller magnitude (making
it more robust to finite-precision arithmetic). It rearranges the computation as:



y[n]  = b0*x[n] + s1
s1   = b1*x[n] - a1*y[n] + s2
s2   = b2*x[n] - a2*y[n]



In C:



typedef struct {
    float b0, b1, b2;
    float a1, a2;
    float s1, s2;  /* two state variables */
} BiquadTDF2;

void biquad_tdf2_reset(BiquadTDF2 *f) {
    f->s1 = f->s2 = 0.0f;
}

float biquad_tdf2_process(BiquadTDF2 *f, float x) {
    float y  = f->b0 * x + f->s1;
    f->s1    = f->b1 * x - f->a1 * y + f->s2;
    f->s2    = f->b2 * x - f->a2 * y;
    return y;
}



Same five multiply-adds per sample, but only two state values to load and store. The
memory access pattern is friendlier to CPU cache when you have many filter instances.


Block Processing


For performance, always process audio in blocks rather than calling the per-sample
function through a function pointer:



void biquad_tdf2_render(BiquadTDF2 *f, const float *in,
                        float *out, int n) {
    float b0 = f->b0, b1 = f->b1, b2 = f->b2;
    float a1 = f->a1, a2 = f->a2;
    float s1 = f->s1, s2 = f->s2;

    for (int i = 0; i < n; i++) {
        float x  = in[i];
        float y  = b0 * x + s1;
        s1       = b1 * x - a1 * y + s2;
        s2       = b2 * x - a2 * y;
        out[i]   = y;
    }

    f->s1 = s1;
    f->s2 = s2;
}



Loading the coefficients into locals at the top of the function lets the compiler keep
them in registers for the inner loop, avoiding five structure-member loads per sample.


Cascading Biquads


Higher-order filters are built by cascading biquad sections in series. A 4th-order
lowpass uses two biquad stages; an 8-band parametric EQ uses eight biquad stages. The
output of each stage feeds the input of the next:



#define MAX_STAGES 8

typedef struct {
    BiquadTDF2 stage[MAX_STAGES];
    int        n_stages;
} BiquadChain;

void chain_render(BiquadChain *chain, const float *in,
                  float *out, int n) {
    /* First stage reads from in[], subsequent stages read from out[] */
    biquad_tdf2_render(&chain->stage[0], in, out, n);
    for (int s = 1; s < chain->n_stages; s++)
        biquad_tdf2_render(&chain->stage[s], out, out, n);
}



In-place processing (in == out for stages 2+) is safe here because we read
and write sequentially.


Coefficient Storage and Updates


A complete biquad coefficient set is just five floats. A good practice is to keep a
BiquadCoeffs struct that can be computed from human-readable parameters
(cutoff Hz, Q, gain dB) and then copied into a BiquadTDF2 for processing.
This separates the DSP from the parameter computation:



typedef struct {
    float b0, b1, b2, a1, a2;
} BiquadCoeffs;

void biquad_set_coeffs(BiquadTDF2 *f, const BiquadCoeffs *c) {
    f->b0 = c->b0;
    f->b1 = c->b1;
    f->b2 = c->b2;
    f->a1 = c->a1;
    f->a2 = c->a2;
    /* State is intentionally NOT reset - this enables smooth coefficient
       updates mid-stream without a click. */
}



Updating coefficients without resetting state is safe as long as the pole positions do
not jump drastically. If you need to sweep a filter continuously (e.g., a filter
envelope), compute new coefficients each block or even each sample and write them
directly - the TDF-II form handles this gracefully.


Numerical Stability


At very low cutoff frequencies (below ~20 Hz) or very high Q values, biquad coefficients
can become numerically problematic in 32-bit float. The b1 coefficient of a lowpass
filter approaches -2, and the difference between b0 and b1/2 becomes tiny - a classic
catastrophic cancellation scenario. Mitigation strategies:



Use 64-bit double for coefficient computation even if state variables stay as float.
Clamp minimum cutoff to a sane value (e.g., 10 Hz).
For very high-Q or very-low-frequency filters, use a specialised form such as the
    State Variable Filter (SVF) which is inherently more stable.


Denormals in Filter State


When the signal fades to silence, the filter state variables approach zero and can enter
the denormal float range, causing dramatic CPU slowdowns on x86. Flush the state to zero
if it drops below a threshold, or enable Flush-to-Zero mode on the FPU. The denormals
article covers this in detail.



/* Simple DC-offset trick to keep state out of denormal range */
void biquad_tdf2_process_block_safe(BiquadTDF2 *f,
                                    const float *in,
                                    float *out, int n) {
    static const float DC_OFFSET = 1.0e-25f;
    for (int i = 0; i < n; i++) {
        float x = in[i] + DC_OFFSET;
        float y = f->b0 * x + f->s1;
        f->s1   = f->b1 * x - f->a1 * y + f->s2;
        f->s2   = f->b2 * x - f->a2 * y;
        out[i]  = y;
    }
}


Summary


The biquad is a second-order IIR filter with five coefficients (b0, b1, b2, a1,
a2). Direct Form II Transposed requires only two state variables and has better
numerical behaviour than Direct Form I. The coefficient formulas come from the
Audio EQ Cookbook — the next two articles derive them for every useful filter
type. Guard filter state against denormals by adding a small DC offset or
checking the state variables for subnormal values.
