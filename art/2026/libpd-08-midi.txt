libpd - Part 8: MIDI with libpd
2026-06-01

Pure Data has a rich set of MIDI objects - [notein], [noteout], [ctlin], [ctlout], and a dozen more. libpd maps these to a C API that lets you inject MIDI events into the patch and collect MIDI output from it without touching any MIDI hardware directly. Your application handles the physical MIDI device; libpd routes the data to and from the Pd scheduler.

The Architecture

In desktop Pd, MIDI devices are selected in the preferences dialog. In libpd, Pd's MIDI I/O is replaced by the C API: you call libpd_noteon to send a Note On, the patch processes it through [notein], and the result (if any) comes back through hooks you register with libpd_set_noteonhook. Physical MIDI I/O stays entirely in your code.

Sending MIDI to Pd

All MIDI send functions are in z_libpd.h. Channel numbers are 0-indexed (channel 0 = MIDI channel 1).


#include <z_libpd.h>

/* Note On: channel 0, MIDI pitch 60 (middle C), velocity 100 */
libpd_noteon(0, 60, 100);

/* Note Off: velocity 0 = note off in standard MIDI */
libpd_noteon(0, 60, 0);

/* Control Change: channel 0, CC 74 (filter cutoff), value 64 */
libpd_controlchange(0, 74, 64);

/* Program Change: channel 0, program 12 */
libpd_programchange(0, 12);

/* Pitch Bend: channel 0, value -8192 to +8191 (0 = centre) */
libpd_pitchbend(0, 512);

/* Aftertouch (channel pressure): channel 0, value 80 */
libpd_aftertouch(0, 80);

/* Polyphonic aftertouch: channel 0, pitch 60, pressure 64 */
libpd_polyaftertouch(0, 60, 64);

/* Raw MIDI byte (for SysEx or other messages) */
libpd_midibyte(0, 0xF8); /* MIDI clock tick */


These functions are synchronous - the event enters the Pd scheduler immediately and is processed during the next call to libpd_process_float.

The [notein] and [noteout] Pd Objects

On the Pd side, MIDI input objects receive what you send via the C API:


[notein]       - outputs: pitch, velocity, channel
[notein 1]     - channel 1 only (channel number is 1-indexed in Pd)
[ctlin]        - outputs: value, controller-number, channel
[pgmin]        - outputs: program-number, channel
[bendin]       - outputs: bend-value (-8192..8191), channel
[touchin]      - outputs: pressure, channel


And MIDI output objects emit what your hooks will receive:


[noteout]      - takes: pitch, velocity, channel
[ctlout]       - takes: value, controller, channel
[pgmout]       - takes: program, channel
[bendout]      - takes: bend-value, channel


Receiving MIDI from Pd

Register hook callbacks to receive MIDI generated by the patch's output objects:


static void on_noteon(int channel, int pitch, int velocity) {
    printf("Note: ch=%d pitch=%d vel=%d\n", channel, pitch, velocity);
    /* Forward to a hardware MIDI port, synthesiser, etc. */
}

static void on_controlchange(int channel, int ctrl, int value) {
    printf("CC: ch=%d ctrl=%d val=%d\n", channel, ctrl, value);
}

static void on_programchange(int channel, int program) {
    printf("PC: ch=%d prog=%d\n", channel, program);
}

static void on_pitchbend(int channel, int bend) {
    printf("PB: ch=%d bend=%d\n", channel, bend);
}

static void on_aftertouch(int channel, int pressure) {
    printf("AT: ch=%d press=%d\n", channel, pressure);
}

static void on_polyaftertouch(int channel, int pitch, int pressure) {
    printf("PAT: ch=%d pitch=%d press=%d\n", channel, pitch, pressure);
}

libpd_set_noteonhook(on_noteon);
libpd_set_controlchangehook(on_controlchange);
libpd_set_programchangehook(on_programchange);
libpd_set_pitchbendhook(on_pitchbend);
libpd_set_aftertouchhook(on_aftertouch);
libpd_set_polyaftertouchhook(on_polyaftertouch);


As with all libpd hooks, these callbacks fire on the audio thread. Do not block, allocate, or touch the UI from inside them. Use a ring buffer to forward events to a non-audio thread (see article 06).

Complete MIDI-Driven Synth Example


#include <z_libpd.h>
#include <portaudio.h>
#include <stdio.h>

#define SR         44100
#define BLOCK      512
#define TICKS      (BLOCK / 64)
#define CHANNELS   2

static int pa_callback(const void *in, void *out,
                       unsigned long frames,
                       const PaStreamCallbackTimeInfo *ti,
                       PaStreamCallbackFlags flags,
                       void *ud)
{
    libpd_process_float(TICKS, (float *)in, (float *)out);
    return paContinue;
}

static void on_noteon(int ch, int pitch, int vel) {
    /* Route outgoing MIDI back to a hardware port if needed */
    printf("Pd output: note ch=%d p=%d v=%d\n", ch, pitch, vel);
}

int main(void) {
    libpd_init();
    libpd_init_audio(0, CHANNELS, SR);

    libpd_set_noteonhook(on_noteon);

    libpd_start_message(1);
    libpd_add_float(1.0f);
    libpd_finish_message("pd", "dsp");

    void *patch = libpd_openfile("midi_synth.pd", "patches");
    if (!patch) return 1;

    Pa_Initialize();
    PaStream *stream;
    Pa_OpenDefaultStream(&stream, 0, CHANNELS, paFloat32,
                         SR, BLOCK, pa_callback, NULL);
    Pa_StartStream(stream);

    /* Simulate incoming MIDI: play a C major arpeggio */
    int notes[] = {60, 64, 67, 72};
    for (int i = 0; i < 4; i++) {
        libpd_noteon(0, notes[i], 100);
        Pa_Sleep(250);
        libpd_noteon(0, notes[i], 0); /* note off */
        Pa_Sleep(50);
    }

    Pa_Sleep(1000);
    Pa_StopStream(stream);
    Pa_CloseStream(stream);
    Pa_Terminate();

    libpd_closefile(patch);
    return 0;
}


Bridging a Physical MIDI Device

To connect real MIDI hardware, use a MIDI I/O library (RtMidi, PortMidi, ALSA sequencer) alongside libpd. The library calls your MIDI input callback; your callback forwards the bytes to libpd:


/* RtMidi callback - called on a separate thread by RtMidi */
void rtmidi_callback(double stamp,
                     std::vector<unsigned char> *msg,
                     void *userData)
{
    if (msg->size() < 2) return;
    int status  = (*msg)[0] & 0xF0;
    int channel = (*msg)[0] & 0x0F;

    switch (status) {
    case 0x90:
        libpd_noteon(channel, (*msg)[1], (*msg)[2]);
        break;
    case 0x80:
        libpd_noteon(channel, (*msg)[1], 0);
        break;
    case 0xB0:
        libpd_controlchange(channel, (*msg)[1], (*msg)[2]);
        break;
    case 0xC0:
        libpd_programchange(channel, (*msg)[1]);
        break;
    case 0xE0: {
        int bend = ((*msg)[2] << 7) | (*msg)[1];
        libpd_pitchbend(channel, bend - 8192);
        break;
    }
    }
}


Note the thread safety issue: RtMidi calls your callback on its own thread, but libpd's MIDI functions should be called from the audio thread. In practice, for most applications the MIDI injection functions are safe to call from any thread at the cost of potential glitches under high load. For production code, queue MIDI events in a lock-free ring buffer and drain them at the start of each audio callback.

Summary


libpd replaces Pure Data's hardware MIDI layer with direct C function calls:
libpd_noteon(channel, pitch, velocity), libpd_controlchange,
libpd_pitchbend, and friends send events to the patch as if they arrived from
a MIDI device. The patch's [noteout] and [ctlout] objects route
output back through registered hooks. Physical MIDI devices are your application's
responsibility — integrate RtMidi or ALSA rawmidi, then queue events in a lock-free
buffer and inject them at the start of each audio callback.
