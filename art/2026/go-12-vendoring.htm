Go - Part 12: Vendoring

Vendoring means copying your project's dependencies into a subdirectory of the project itself, so builds never reach the network and the source of every dependency is under your direct control. Go has first-class support for this with go mod vendor. It is not the default and it is not always the right choice - but in certain environments (air-gapped build systems, regulated industries requiring audit trails, corporate proxies that intercept HTTPS) it is the only practical choice. This article explains the mechanics and the trade-offs.

What You Will Learn

  • How to create the vendor directory with go mod vendor and what it contains
  • How to build from the vendor directory and verify it is up to date
  • When vendoring is the right choice: air-gapped builds, audit requirements, corporate proxies
  • How the vendor directory interacts with the module cache and CI pipelines
  • How to patch a vendored dependency without forking it

Creating the vendor Directory

With a valid go.mod and go.sum already in place, one command copies all dependencies into a vendor/ subdirectory: 

go mod vendor

The result is a vendor/ directory alongside your go.mod: 

myproject/
  go.mod
  go.sum
  vendor/
    modules.txt
    github.com/
      gorilla/
        mux/
          ...
    golang.org/
      x/
        crypto/
          ...
  main.go

vendor/modules.txt is a manifest generated by the go command listing each vendored module, its version, and which packages from it are used. Do not edit it by hand.

There is no direct equivalent in C. In C# projects with NuGet, you can configure RestorePackagesWithLockFile and commit the lock file, but the packages themselves live in a global per-user cache (~/.nuget/packages), not in the project. Committing the actual package sources is not standard practice in C#. Go's vendor approach is closer to npm's vendor or Cargo's vendor support.

Building from the vendor Directory

Once the vendor directory exists, the go command uses it automatically if the directory is present in the module root. You can also be explicit: 

# Use vendor directory explicitly
go build -mod=vendor ./...
go test -mod=vendor ./...

# Force the go command to ignore vendor even if it exists
go build -mod=mod ./...

# Read-only: fail if modules are missing or wrong (good for CI)
go build -mod=readonly ./...

In Go 1.14+, if a vendor/ directory is present, the default is -mod=vendor. If you are in CI and want to be sure no network access happens, set GOFLAGS=-mod=vendor or pass -mod=vendor explicitly.

Keeping vendor in Sync

Whenever you add, remove, or update a dependency, re-run both commands in sequence: 

go mod tidy       # clean go.mod and go.sum
go mod vendor     # rebuild the vendor directory

Then commit the changes. In CI, verify that neither command produces a diff: 

go mod tidy
go mod vendor
git diff --exit-code  # fails the build if anything changed

This check ensures that what is committed is exactly reproducible. If a developer adds a dependency without running go mod vendor, the CI build fails loudly rather than silently fetching from the network.

Verifying the vendor Directory

The go command can verify that the vendor directory matches the go.sum hashes: 

go mod verify

This reads go.sum and checks that the vendored source matches. Run this in CI to detect tampering. If someone edits a vendored file directly (a pattern sometimes used for emergency patches), this check will fail and flag the discrepancy.

When to Use Vendor

Vendoring is appropriate when:

  • Air-gapped builds: your build servers have no internet access. Without vendoring, go build requires either the module cache to be pre-populated or network access to proxy.golang.org.
  • Audit requirements: your organisation requires that all source code - including dependencies - be reviewed and stored in your own version control system.
  • Hermetic builds: you want a guarantee that the same source always produces the same binary, even if a dependency's repository disappears or its content changes.
  • Slow or unreliable proxy: in some corporate environments, the external module proxy is blocked or unreliable, and setting up an internal proxy (Athens, Artifactory, Nexus) is more work than vendoring.

Vendoring is not appropriate when:

  • Your repository is already large and committing gigabytes of dependencies would make it unwieldy.
  • You have a reliable CI environment with a module cache and network access - the module cache already provides reproducible builds.
  • You are developing a library (not a binary) - library authors should not vendor because the consuming project controls which dependency versions to use.

Vendor vs Module Cache

The module cache ($GOPATH/pkg/mod by default, or $GOMODCACHE) is a per-machine cache shared between all modules. It is write-protected once written - go sets files to read-only to prevent accidental editing. The vendor directory is per-project and editable (though you should not edit it manually).

In a CI environment without vendoring, the module cache must either be pre-populated (cached between CI runs using the CI system's cache mechanism) or refetched on each run. Fetching on each run is slow and depends on the network; caching the module cache between runs is common and works well. Vendoring avoids this entirely at the cost of repository size. 

# In a Dockerfile, pre-populate the module cache before copying source
# This takes advantage of Docker layer caching
FROM golang:1.22 AS build
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download   # downloads but does not vendor
COPY . .
RUN go build -o myapp ./cmd/myapp

# With vendor:
FROM golang:1.22 AS build
WORKDIR /app
COPY . .              # vendor/ is included
RUN go build -mod=vendor -o myapp ./cmd/myapp

Patching a Vendored Dependency

One reason teams choose vendoring is to apply emergency patches to a dependency without waiting for the upstream maintainer. With vendor you can edit the source directly. The downside: go mod verify will fail, and go mod vendor will overwrite your changes. Document the patch, use a replace directive pointing to a fork instead, and upstream the fix as soon as possible.

Summary

go mod vendor copies all dependencies into a vendor/ directory alongside go.mod. Builds with -mod=vendor (or GOFLAGS=-mod=vendor) use that directory instead of the module cache — no network required. Commit the vendor directory to version control. Run go mod verify in CI to detect tampering. To patch a vendored dependency, edit the source directly in vendor/; document the patch prominently so it is not accidentally overwritten.
