Containers - Part 17: Linux Capabilities

For most of Linux history, privilege was binary: you were root (UID 0) or you were not. Root could do everything. Non-root could do almost nothing sensitive. Linux capabilities, added in kernel 2.2, break root into approximately forty independent privilege bits. A process can hold the capability to bind to privileged ports (CAP_NET_BIND_SERVICE) without being able to load kernel modules (CAP_SYS_MODULE). Container runtimes exploit this: they start a process with the full root privilege needed to set up the container, then drop every capability that the workload does not need before executing it. A compromised container process with a stripped capability set has far less power to damage the host than a full-root process would.

The Five Capability Sets

The kernel tracks capabilities in five sets per thread. Understanding the sets is essential to understanding how privilege flows when a process calls execve or passes capabilities to a child.

  • Permitted (P) - the ceiling of capabilities the thread may ever hold. A thread can never raise a capability that is not in its permitted set.
  • Effective (E) - the capabilities that are actually checked when the kernel evaluates a permission. A permitted capability must be in the effective set to be active.
  • Inheritable (I) - capabilities preserved across execve(). After exec, the new process's permitted set is calculated as (P & I_new) | (file_permitted & bounding).
  • Bounding (B) - a one-way-drop set that limits what can ever enter the permitted set via exec. Once a capability is removed from the bounding set (with prctl(PR_CAPBSET_DROP,...)) it can never be regained by this process or its children, regardless of file capabilities on the executed binary.
  • Ambient (A) - introduced in Linux 4.3. Capabilities in this set are inherited across exec for non-privileged binaries (binaries with no file capabilities). Simplifies setups where you want a child process to hold a specific capability without needing a setuid binary.

Inspect the capability sets of any process: 

# /proc/PID/status shows all five sets as hex bitmasks
grep Cap /proc/self/status
# CapInh: 0000000000000000
# CapPrm: 000001ffffffffff
# CapEff: 000001ffffffffff
# CapBnd: 000001ffffffffff
# CapAmb: 0000000000000000

# Decode a hex bitmask with capsh
capsh --decode=000001ffffffffff
# 0x000001ffffffffff=cap_chown,cap_dac_override,...,cap_sys_admin,...

Key Capability Constants

There are currently 41 defined capabilities (Linux 5.8+). The most significant for container security:

  • CAP_CHOWN (0) - change file ownership. Needed by package managers inside the container.
  • CAP_DAC_OVERRIDE (1) - bypass file read/write/execute permission checks. Very powerful - a root process inside a container normally needs this to write to files owned by other UIDs.
  • CAP_NET_BIND_SERVICE (10) - bind to ports below 1024. Useful for web servers; harmless if the container is network-isolated.
  • CAP_NET_RAW (13) - use raw sockets. Needed by ping. Also allows crafting arbitrary packets and sniffing all traffic on a network interface - docker drops this by default because it is a significant attack vector on shared networks.
  • CAP_SYS_ADMIN (21) - a catch-all for dozens of operations: mount filesystems, set hostnames, ptrace arbitrary processes, configure namespaces, manage devices. Described as "the new root". Containers avoid it strenuously.
  • CAP_SYS_BOOT (22) - reboot or load a new kernel with kexec. No legitimate container workload needs this.
  • CAP_SYS_MODULE (16) - load and unload kernel modules. Immediate full system compromise if abused.
  • CAP_SYS_PTRACE (19) - trace arbitrary processes. A container holding this can inspect and manipulate every process visible to it.
  • CAP_SETUID (7) / CAP_SETGID (6) - change UID/GID arbitrarily. A container process with these can escalate to UID 0 even if it started as a non-root user inside the container.

Reading Capabilities from C

#define _GNU_SOURCE
#include <stdio.h>
#include <sys/capability.h>
#include <unistd.h>

int main(void) {
    cap_t caps = cap_get_proc();
    if (!caps) { perror("cap_get_proc"); return 1; }

    char *text = cap_to_text(caps, NULL);
    printf("capabilities: %s\n", text);
    cap_free(text);
    cap_free(caps);

    /* Check a specific capability */
    cap_flag_value_t val;
    cap_get_flag(caps, CAP_NET_RAW, CAP_EFFECTIVE, &val);
    printf("CAP_NET_RAW effective: %s\n", val == CAP_SET ? "yes" : "no");
    return 0;
}

Link with -lcap: gcc -o cap_read cap_read.c -lcap. The libcap library (apt install libcap-dev) wraps the raw capget() and capset() syscalls with a human-friendly API.

Dropping Capabilities with prctl and capset in C

A container runtime drops capabilities in two stages. First it drops them from the bounding set using prctl(PR_CAPBSET_DROP, ...) so that even after execve the child cannot regain them. Then it drops them from the effective and permitted sets using capset(). 

#define _GNU_SOURCE
#include <linux/capability.h>
#include <stdio.h>
#include <stdlib.h>
#include <sys/prctl.h>
#include <sys/types.h>
#include <unistd.h>

/*
 * Capabilities that Docker drops by default.
 * Source: github.com/moby/moby - oci/defaults.go
 */
static int caps_to_drop[] = {
    CAP_AUDIT_CONTROL,
    CAP_AUDIT_READ,
    CAP_AUDIT_WRITE,
    CAP_BLOCK_SUSPEND,
    CAP_DAC_READ_SEARCH,
    CAP_FSETID,
    CAP_IPC_LOCK,
    CAP_IPC_OWNER,
    CAP_LINUX_IMMUTABLE,
    CAP_MAC_ADMIN,
    CAP_MAC_OVERRIDE,
    CAP_MKNOD,
    CAP_NET_ADMIN,
    CAP_NET_BROADCAST,
    CAP_NET_RAW,
    CAP_SETFCAP,
    CAP_SYSLOG,
    CAP_SYS_ADMIN,
    CAP_SYS_BOOT,
    CAP_SYS_MODULE,
    CAP_SYS_NICE,
    CAP_SYS_PACCT,
    CAP_SYS_PTRACE,
    CAP_SYS_RAWIO,
    CAP_SYS_RESOURCE,
    CAP_SYS_TIME,
    CAP_SYS_TTY_CONFIG,
    CAP_WAKE_ALARM,
    -1  /* sentinel */
};

int drop_capabilities(void) {
    /* Step 1: Drop from the bounding set */
    for (int i = 0; caps_to_drop[i] != -1; i++) {
        if (prctl(PR_CAPBSET_DROP, caps_to_drop[i], 0, 0, 0) == -1) {
            perror("PR_CAPBSET_DROP");
            return -1;
        }
    }

    /* Step 2: Build a capset that keeps only what we allow */
    struct __user_cap_header_struct hdr = {
        .version = _LINUX_CAPABILITY_VERSION_3,
        .pid     = 0,  /* 0 means current thread */
    };
    struct __user_cap_data_struct data[2] = {0};

    /*
     * Set all bits, then clear the ones we are dropping.
     * We keep: CAP_CHOWN, CAP_DAC_OVERRIDE, CAP_FOWNER,
     *          CAP_KILL, CAP_NET_BIND_SERVICE, CAP_SETGID,
     *          CAP_SETUID, CAP_SETPCAP.
     * Everything else is cleared.
     *
     * For simplicity in this example we just clear everything:
     */
    data[0].permitted  = 0;
    data[0].effective  = 0;
    data[0].inheritable= 0;
    data[1].permitted  = 0;
    data[1].effective  = 0;
    data[1].inheritable= 0;

    if (capset(&hdr, data) == -1) {
        perror("capset");
        return -1;
    }

    /* Step 3: Prevent regaining capabilities after setuid exec */
    if (prctl(PR_SET_NO_NEW_PRIVS, 1, 0, 0, 0) == -1) {
        perror("PR_SET_NO_NEW_PRIVS");
        return -1;
    }

    return 0;
}

int main(void) {
    printf("before drop: ");
    system("grep CapEff /proc/self/status");

    if (drop_capabilities() != 0) {
        fprintf(stderr, "failed to drop capabilities\n");
        return 1;
    }

    printf("after drop: ");
    system("grep CapEff /proc/self/status");
    printf("capabilities dropped. process is now highly restricted.\n");
    return 0;
}

Compile and run as root: gcc -o cap_drop cap_drop.c && sudo ./cap_drop. The CapEff line will change from 000001ffffffffff to 0000000000000000.

The NO_NEW_PRIVS Flag

prctl(PR_SET_NO_NEW_PRIVS, 1, ...) sets a flag that prevents any future execve in this process or its descendants from gaining new privileges through setuid bits or file capabilities. It is inherited across fork and exec and cannot be unset. Container runtimes always set this after dropping capabilities. seccomp filters require it on some kernel versions. It is also set by systemd for services marked NoNewPrivileges=yes.

Dropping Capabilities from Go

package main

import (
    "fmt"
    "log"
    "os"
    "unsafe"

    "golang.org/x/sys/unix"
)

// capabilitiy constants from linux/capability.h
const (
    CAP_CHOWN            = 0
    CAP_DAC_OVERRIDE     = 1
    CAP_FOWNER           = 3
    CAP_KILL             = 5
    CAP_SETGID           = 6
    CAP_SETUID           = 7
    CAP_SETPCAP          = 8
    CAP_NET_BIND_SERVICE = 10
    CAP_NET_RAW          = 13
    CAP_SYS_ADMIN        = 21
    CAP_SYS_MODULE       = 16
    CAP_SYS_PTRACE       = 19
    CAP_SYS_BOOT         = 22
)

// allowedCaps is the set of capabilities we keep (Docker's default retained set).
var allowedCaps = map[uint]bool{
    CAP_CHOWN:            true,
    CAP_DAC_OVERRIDE:     true,
    CAP_FOWNER:           true,
    CAP_KILL:             true,
    CAP_SETGID:           true,
    CAP_SETUID:           true,
    CAP_SETPCAP:          true,
    CAP_NET_BIND_SERVICE: true,
}

func dropBoundingCap(cap uint) error {
    return unix.Prctl(unix.PR_CAPBSET_DROP, uintptr(cap), 0, 0, 0)
}

// dropCapabilities drops all capabilities not in allowedCaps from both
// the bounding set and the effective/permitted/inheritable sets.
func dropCapabilities() error {
    // Drop from bounding set
    for cap := uint(0); cap <= 40; cap++ {
        if allowedCaps[cap] {
            continue
        }
        if err := dropBoundingCap(cap); err != nil {
            // EINVAL means the capability does not exist on this kernel
            if err == unix.EINVAL {
                continue
            }
            return fmt.Errorf("PR_CAPBSET_DROP cap %d: %w", cap, err)
        }
    }

    // Build capset data - permitted and effective only contain allowed caps
    type capHeader struct {
        version uint32
        pid     int32
    }
    type capData struct {
        effective   uint32
        permitted   uint32
        inheritable uint32
    }

    hdr := capHeader{
        version: 0x20080522, // _LINUX_CAPABILITY_VERSION_3
        pid:     0,
    }
    var data [2]capData

    var permitted uint64
    for cap := range allowedCaps {
        permitted |= 1 << cap
    }
    data[0].permitted  = uint32(permitted & 0xffffffff)
    data[0].effective  = uint32(permitted & 0xffffffff)
    data[0].inheritable= 0
    data[1].permitted  = uint32(permitted >> 32)
    data[1].effective  = uint32(permitted >> 32)
    data[1].inheritable= 0

    if _, _, errno := unix.Syscall(unix.SYS_CAPSET,
        uintptr(unsafe.Pointer(&hdr)),
        uintptr(unsafe.Pointer(&data[0])),
        0); errno != 0 {
        return fmt.Errorf("capset: %w", errno)
    }

    // Set NO_NEW_PRIVS
    if err := unix.Prctl(unix.PR_SET_NO_NEW_PRIVS, 1, 0, 0, 0); err != nil {
        return fmt.Errorf("PR_SET_NO_NEW_PRIVS: %w", err)
    }

    return nil
}

func main() {
    if os.Getuid() != 0 {
        log.Fatal("must run as root")
    }

    before, _ := os.ReadFile("/proc/self/status")
    for _, line := range splitLines(string(before)) {
        if len(line) > 6 && line[:6] == "CapEff" {
            fmt.Println("before:", line)
            break
        }
    }

    if err := dropCapabilities(); err != nil {
        log.Fatal(err)
    }

    after, _ := os.ReadFile("/proc/self/status")
    for _, line := range splitLines(string(after)) {
        if len(line) > 6 && line[:6] == "CapEff" {
            fmt.Println("after: ", line)
            break
        }
    }

    fmt.Println("capabilities restricted to container default set")
}

func splitLines(s string) []string {
    var out []string
    start := 0
    for i := 0; i < len(s); i++ {
        if s[i] == '\n' {
            out = append(out, s[start:i])
            start = i + 1
        }
    }
    return out
}

What Docker Drops by Default

Docker's default capability set keeps a small whitelist and drops everything else. The retained capabilities are: 

CHOWN, DAC_OVERRIDE, FSETID, FOWNER, MKNOD,
NET_RAW, SETGID, SETUID, SETFCAP, SETPCAP,
NET_BIND_SERVICE, SYS_CHROOT, KILL, AUDIT_WRITE

Notably dropped: CAP_SYS_ADMIN, CAP_SYS_MODULE, CAP_SYS_RAWIO, CAP_SYS_PTRACE, CAP_NET_ADMIN, CAP_NET_BROADCAST, CAP_SYS_BOOT, CAP_MAC_ADMIN, and many others. CAP_NET_RAW is kept by default (it is needed for ping) but Docker provides --cap-drop=NET_RAW to remove it.

You can override with --cap-add and --cap-drop: 

# Run a container with only NET_BIND_SERVICE added beyond Docker's default
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE nginx

# Inspect a running container's capabilities
docker inspect my-container | grep -A5 CapAdd

CAP_SYS_ADMIN: The New Root

CAP_SYS_ADMIN is mentioned separately because it is categorically dangerous. The man page lists over thirty distinct operations it permits, including: mounting filesystems, setting hostnames, loading BPF programs, calling CLONE_NEWNS/CLONE_NEWNET/CLONE_NEWUSER in some configurations, configuring loop devices, attaching eBPF programs to cgroups, and many more. A container with CAP_SYS_ADMIN can almost certainly escape to the host given a kernel bug or a misconfigured namespace. It is always dropped.

Summary

Linux capabilities decompose root privilege into ~40 independent bits. The kernel maintains five per-thread capability sets: permitted, effective, inheritable, bounding, and ambient. Container runtimes drop capabilities from the bounding set first (making them unrecoverable across exec) and then from the effective and permitted sets. PR_SET_NO_NEW_PRIVS ensures no future execve can regain privileges via setuid binaries or file capabilities. Docker's default policy retains fourteen capabilities and drops everything else, reducing the blast radius of a container escape significantly.
