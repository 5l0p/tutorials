Go - Part 14: Formatting, Linting & Tooling

Go has an unusually strong tooling culture. Formatting is not a matter of team preference - gofmt enforces a single canonical style, and code that is not gofmt'd is considered wrong. Beyond formatting, the ecosystem includes static analysis, a language server, documentation tools, and a profiler, all either shipped with Go or installable in one command. If you are coming from C (where the tooling situation is fragmented and optional) or from C# (where formatting and analysis depend on which IDE you use), the uniformity here is a welcome change.

What You Will Learn

  • gofmt: canonical formatting, how to run it, and why non-gofmt'd code is not accepted
  • go vet: built-in static checks for common mistakes
  • staticcheck: a production-grade analyser that catches subtle correctness issues
  • golangci-lint: aggregated linting with many analysers configured in one file
  • gopls: the official language server — what it provides and how to configure it
  • go doc and godoc: generating and reading package documentation
  • go tool pprof: CPU and memory profiling from first principles
  • Wiring all tools into a CI pipeline

gofmt: Non-Negotiable Formatting

gofmt is the Go formatter. It is not configurable - there are no options for tab width, brace placement, or line length. Every Go project in the world uses the same formatting. This eliminates formatting debates, reduces diff noise in code review, and means any Go code you read anywhere looks the same. 

# Format all Go files in the current directory tree (in place)
gofmt -w .

# Show what would change without writing
gofmt -d .

# Also simplifies certain expressions (use this in CI)
gofmt -s -w .

In CI, verify that code is properly formatted: 

# Fails if any file needs reformatting
if [ -n "$(gofmt -l .)" ]; then
    echo "The following files are not gofmt'd:"
    gofmt -l .
    exit 1
fi

Most editors run gofmt (or its superset goimports) on save. goimports does everything gofmt does, plus automatically adds and removes import statements: 

go install golang.org/x/tools/cmd/goimports@latest
goimports -w .

C has clang-format which is configurable (which is both its strength and its weakness). C# has the editorconfig + Roslyn formatter, which is configurable per project. Go's non-configurable approach produces more uniform code across the ecosystem.

go vet: Built-in Static Analysis

go vet is included with Go and runs a set of static analysis checks that catch common mistakes not caught by the compiler: 

go vet ./...

Examples of what it catches:

  • fmt.Printf("%d", "string") - wrong format verb for the argument type.
  • Unreachable code after a return.
  • Passing a lock by value (copying a mutex, which makes it useless).
  • Incorrect struct tags.
  • Calling sync/atomic functions on misaligned 64-bit values.

Always run go vet in CI. It costs almost nothing in CI time and catches real bugs. In C the equivalent is compiler warnings (-Wall -Wextra) plus tools like Clang Static Analyzer. In C# it is Roslyn analyzers and the compiler's diagnostic rules.

staticcheck: Production-Grade Static Analysis

staticcheck is the most widely used third-party Go static analyzer. It catches a superset of what go vet catches, plus:

  • Deprecated API usage with guidance on what to use instead.
  • Redundant code (unused function results, unnecessary type conversions).
  • Incorrect use of the standard library (e.g., wrong usage of sync.Mutex, passing a copied value to a method expecting a pointer).
  • Performance issues (e.g., unnecessary string allocations in hot paths).
go install honnef.co/go/tools/cmd/staticcheck@latest
staticcheck ./...

golangci-lint: Aggregated Linting

golangci-lint is a linter aggregator that runs many linters in parallel and reports unified output. It is the de facto standard for Go CI linting. It includes go vet, staticcheck, errcheck (detects ignored errors), gosec (security checks), and dozens more: 

# Install
go install github.com/golangci/golangci-lint/cmd/golangci-lint@latest

# Run with default settings
golangci-lint run ./...

# Run with a config file (.golangci.yml)
golangci-lint run --config .golangci.yml ./...

A minimal .golangci.yml: 

linters:
  enable:
    - errcheck
    - gosimple
    - govet
    - ineffassign
    - staticcheck
    - unused

linters-settings:
  errcheck:
    check-type-assertions: true

issues:
  exclude-rules:
    - path: "_test.go"
      linters:
        - errcheck

In C# projects using MSBuild, the equivalent is configuring Roslyn analyzers via .editorconfig and enabling specific diagnostic IDs. Go's approach is more explicit but the concept is the same.

gopls: The Language Server

gopls is the official Go language server, implementing the Language Server Protocol (LSP). Any LSP-compatible editor (VS Code, Neovim, Emacs, Helix, etc.) uses it for autocompletion, go-to-definition, find-all-references, rename, inlay hints, and diagnostics. 

go install golang.org/x/tools/gopls@latest

You do not invoke gopls directly - your editor does. Install the Go extension for your editor and it will handle gopls. The experience is comparable to the Roslyn language server in C# (OmniSharp or the built-in VS C# extension).

go doc: Documentation

go doc prints documentation for packages, types, functions, and methods directly in the terminal: 

go doc fmt                    # package overview
go doc fmt.Println            # specific function
go doc -all fmt               # everything in the package
go doc net/http Server        # type in a package
go doc -src fmt.Println       # show source

The web-based equivalent is pkg.go.dev, which renders documentation for any public Go module. In C, documentation is typically in comments and man pages. In C# it is XML doc comments rendered by Visual Studio or DocFX.

go tool pprof: Profiling

Go has built-in support for CPU and memory profiling. Add the net/http/pprof package to an HTTP server and you get profiling endpoints for free: 

import _ "net/http/pprof"

// Start the HTTP server (even for non-HTTP programs)
go func() {
    log.Println(http.ListenAndServe("localhost:6060", nil))
}()

Then collect a profile: 

# CPU profile for 30 seconds
go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30

# Memory profile
go tool pprof http://localhost:6060/debug/pprof/heap

# Interactive pprof shell - try 'top', 'web', 'list funcname'

Alternatively, write profiles directly from a benchmark or test: 

func BenchmarkMyFunc(b *testing.B) {
    for i := 0; i < b.N; i++ {
        myFunc()
    }
}

# Run benchmark and write CPU profile
go test -bench=BenchmarkMyFunc -cpuprofile=cpu.prof ./...
go tool pprof cpu.prof

In C, profiling typically requires perf, gprof, or Valgrind/callgrind - external tools with more setup. In C#, you use dotnet-trace and the performance profiler built into Visual Studio. Go's profiler is part of the standard toolchain and is designed to work on live production services.

Making Tooling Part of CI

A complete CI pipeline for a Go project runs these steps in order: 

go mod tidy
git diff --exit-code go.mod go.sum  # fail if tidy changed anything

go mod verify                        # verify module cache integrity

gofmt -s -l .
# fail if output is non-empty

go vet ./...

golangci-lint run ./...

go test -race -count=1 ./...

Keep CI fast by running these in parallel where possible. go test -race is the most expensive step; on a large codebase you may run it only on pull requests, not every push.

Summary

gofmt enforces a single canonical style — run it on save, enforce it in CI with gofmt -l .. go vet catches structural bugs the compiler misses. staticcheck and golangci-lint add deeper analysis; configure golangci-lint in .golangci.yml and run it in CI. gopls provides IDE features over LSP. Profile with go test -cpuprofile or the net/http/pprof handler in a running server, then analyse with go tool pprof.
