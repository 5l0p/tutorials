Go - Part 9: The Context Package

Once you look at enough Go code, you notice that almost every function that does I/O takes a context.Context as its first argument. This is not a convention - it is the mechanism by which Go propagates cancellation, timeouts, and deadlines down through a tree of function calls without coupling the callers to the specifics of how work gets cancelled. If you have used C#'s CancellationToken, you already understand the problem this solves; the Go design is cleaner but more pervasive.

What You Will Learn

  • What a context.Context is and what it carries
  • context.Background() and context.TODO(): the two root contexts
  • context.WithCancel: propagating cancellation through a call tree
  • context.WithTimeout and context.WithDeadline: automatic cancellation after a time limit
  • How to pass context through every layer of a call tree and check ctx.Done()
  • context.WithValue: request-scoped values and why to use them sparingly
  • Common mistakes: storing contexts in structs, passing nil contexts, using value keys that collide

What a Context Is

context.Context is an interface: 

type Context interface {
    Deadline() (deadline time.Time, ok bool)
    Done() <-chan struct{}
    Err() error
    Value(key any) any
}

Done() returns a channel that is closed when the context is cancelled or expires. Err() returns the reason: context.Canceled if the context was explicitly cancelled, or context.DeadlineExceeded if a deadline was set and it passed.

You never implement this interface yourself. You use the constructors in the context package to create instances.

The Root Context

Every context tree starts at a root. There are two root contexts provided by the package: 

ctx := context.Background() // never cancelled, no deadline, no values
ctx := context.TODO()       // placeholder - use when unsure what context to pass

context.Background() is the root of every context tree in a production program. context.TODO() is a signal to your future self or to reviewers that you know a context should go here but have not wired it up yet.

context.WithCancel

WithCancel returns a child context and a cancel function. Calling the cancel function closes the child's Done channel, which closes all of its children's Done channels recursively: 

ctx, cancel := context.WithCancel(context.Background())
defer cancel() // always call cancel to release resources

go func(ctx context.Context) {
    for {
        select {
        case <-ctx.Done():
            fmt.Println("worker cancelled:", ctx.Err())
            return
        default:
            fmt.Println("working...")
            time.Sleep(100 * time.Millisecond)
        }
    }
}(ctx)

time.Sleep(300 * time.Millisecond)
cancel() // signal the goroutine to stop
time.Sleep(100 * time.Millisecond) // give it time to exit

Always call the cancel function - it releases resources even if the context was not cancelled for an error reason. The idiomatic pattern is defer cancel() immediately after creating the context.

In C# this is CancellationTokenSource.Cancel(). The Go version is slightly simpler because the cancel function is a plain function, not a method on a separate source object.

context.WithTimeout and context.WithDeadline

WithTimeout creates a context that cancels after a duration. WithDeadline creates one that cancels at an absolute time. Both return a cancel function that you should still defer to clean up immediately if the work finishes before the deadline: 

// Cancel after 5 seconds
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()

req, err := http.NewRequestWithContext(ctx, "GET", "https://example.com", nil)
if err != nil {
    return err
}

resp, err := http.DefaultClient.Do(req)
if err != nil {
    if errors.Is(err, context.DeadlineExceeded) {
        return fmt.Errorf("request timed out")
    }
    return fmt.Errorf("request failed: %w", err)
}
defer resp.Body.Close()

The context is passed to the HTTP request via http.NewRequestWithContext. The HTTP client monitors the context; if it is cancelled before the response arrives, the client aborts the request and returns a context error. This works all the way through the network stack because the context is threaded through every I/O call.

In C, there is no equivalent. You would set a socket timeout with setsockopt, which is per-connection and cannot be coordinated across multiple concurrent I/O operations. In C# you use HttpClient.Timeout for simple cases or pass a CancellationToken to cancellation-aware methods.

Propagating Context Through Function Calls

The key discipline is that context must be propagated through every layer of the call stack explicitly. No function in the call chain can accidentally ignore a cancellation signal unless it deliberately does not accept a context. 

func handleRequest(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context() // already carries request lifetime

    user, err := loadUser(ctx, r.Header.Get("User-ID"))
    if err != nil {
        http.Error(w, "user not found", http.StatusNotFound)
        return
    }

    data, err := fetchData(ctx, user)
    if err != nil {
        http.Error(w, "internal error", http.StatusInternalServerError)
        return
    }

    json.NewEncoder(w).Encode(data)
}

func loadUser(ctx context.Context, id string) (*User, error) {
    // ctx is passed to all database calls
    row := db.QueryRowContext(ctx, "SELECT * FROM users WHERE id = $1", id)
    // ...
}

func fetchData(ctx context.Context, user *User) (*Data, error) {
    req, _ := http.NewRequestWithContext(ctx, "GET", dataURL, nil)
    resp, err := http.DefaultClient.Do(req)
    // ...
}

If the client disconnects, r.Context() is cancelled by the HTTP server. The cancellation flows through loadUser and fetchData automatically because they both accept and use the same context. In-flight database queries and HTTP requests are aborted. No manual cleanup is needed.

context.WithValue

A context can carry arbitrary key-value pairs that are retrieved anywhere in the call tree. This is used for request-scoped values that would otherwise require passing an extra parameter through every function: request IDs, authenticated user info, trace spans. 

// Define a key type to avoid collisions (unexported key type is idiomatic)
type contextKey int

const requestIDKey contextKey = 0

// Add a value
ctx = context.WithValue(ctx, requestIDKey, "req-abc-123")

// Retrieve it anywhere below
reqID, ok := ctx.Value(requestIDKey).(string)
if ok {
    log.Printf("[%s] handling request", reqID)
}

The key type must be unexported (or at least package-specific) to avoid key collisions between packages. Using a plain string as a key is a common mistake - two packages using "user" as a key would stomp on each other.

context.WithValue should not be used as a general-purpose parameter-passing mechanism. Pass parameters that are needed for computation as function arguments. Reserve context values for truly cross-cutting concerns: tracing, logging IDs, authentication tokens that are needed in many places but clutter every function signature.

What Not to Do

Do not store context in a struct. The Go documentation is explicit about this. A context is associated with a single operation (a request, a task); a struct lives longer. If you need a long-lived object to start an operation and need to cancel it, pass the context when starting the operation, not when constructing the object. 

// Wrong:
type Server struct {
    ctx context.Context // don't do this
}

// Right: accept context in the method that does the work
func (s *Server) ProcessRequest(ctx context.Context, req *Request) error {
    // ...
}

Do not pass a nil context. If you do not know which context to use, use context.TODO(). A nil context causes a panic inside the context package.

Always cancel contexts you create. Even if you know the context will naturally expire, the internal timer goroutine created by WithTimeout is not freed until either the timer fires or cancel is called. In a hot path this causes a goroutine leak.

Summary

context.Context is an immutable value carrying a cancellation signal, an optional deadline, and optional key-value pairs. Child contexts are derived from parents with WithCancel, WithTimeout, or WithDeadline; cancelling a parent cancels all its children. Pass context as the first parameter to every I/O function. Check ctx.Err() or ctx.Done() in loops. Use WithValue only for request-scoped metadata (trace IDs, auth tokens) not for passing ordinary function parameters.
