LLM - Part 13: Local Models with llama.cpp

Every article in this series has called https://api.openai.com. That endpoint sends your data to OpenAI's servers and costs money per token. Sometimes you want to run inference locally: for privacy, for offline use, for cost control, or to experiment with models that are not available via API. llama.cpp runs quantized open models locally and exposes an OpenAI-compatible REST API. Your Go agent code changes by one line.

What You Will Learn

  • What llama.cpp is and how to install it from source or pre-built binaries
  • How to download a quantized model and start the OpenAI-compatible server
  • How to point your Go agent at the local server with a single environment variable
  • Quantisation trade-offs: 4-bit vs 8-bit vs full precision, quality vs memory vs speed
  • Performance benchmarks: tokens per second by model size and hardware

What llama.cpp Is

llama.cpp is a C/C++ library for running large language model inference on commodity hardware. It supports CPU inference (slow but functional) and GPU acceleration via CUDA, Metal, and Vulkan. It can run quantized models (4-bit, 8-bit) which reduces memory requirements dramatically: a 7B parameter model at 4-bit quantization fits in about 4GB of RAM.

The project includes a server binary that exposes the OpenAI-compatible /v1/chat/completions and /v1/embeddings endpoints.

Installing llama.cpp

# Build from source (Linux/macOS)
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
cmake -B build -DLLAMA_CURL=ON
cmake --build build --config Release -j$(nproc)

# The server binary is at:
./build/bin/llama-server

On a machine with an NVIDIA GPU, add -DGGML_CUDA=ON to the cmake flags. On macOS with Apple Silicon, Metal acceleration is enabled by default.

Downloading a Model

Models are distributed as GGUF files on Hugging Face. A few practical choices:

  • Qwen2.5-7B-Instruct-Q4_K_M.gguf - 7B params, 4GB RAM, strong at following instructions. Good all-round choice.
  • Llama-3.2-3B-Instruct-Q8_0.gguf - 3B params, 3.5GB RAM, fast, less capable.
  • Mistral-7B-Instruct-v0.3-Q4_K_M.gguf - reliable instruction following, good for tool calling.
# Download a model (requires huggingface-cli or wget)
huggingface-cli download Qwen/Qwen2.5-7B-Instruct-GGUF \
  Qwen2.5-7B-Instruct-Q4_K_M.gguf \
  --local-dir ./models

Running the Server

./build/bin/llama-server \
  --model ./models/Qwen2.5-7B-Instruct-Q4_K_M.gguf \
  --port 8080 \
  --ctx-size 8192 \
  --n-gpu-layers 99 \
  --host 127.0.0.1

Key flags:

  • --ctx-size - context window size in tokens. Larger uses more RAM. Match this to your model's supported window.
  • --n-gpu-layers - number of transformer layers to offload to GPU. Use 99 to offload everything; reduce if you run out of VRAM.
  • --host 127.0.0.1 - bind to localhost only. Do not expose this to the network without auth.

Swapping the Base URL in Go

This is the key insight: llama.cpp's server speaks the same JSON API as OpenAI. Change one constant and your agent talks to a local model: 

// Before:
const apiURL = "http://localhost:8001/v1/chat/completions"

// After:
const apiURL = "http://127.0.0.1:8080/v1/chat/completions"

Make this configurable via an environment variable: 

func apiBaseURL() string {
    if base := os.Getenv("LLM_BASE_URL"); base != "" {
        return base
    }
    return "https://api.openai.com"
}

const chatPath = "/v1/chat/completions"

func chatURL() string {
    return apiBaseURL() + chatPath
}

When using a local model you may not need an API key. The server accepts the key field but ignores it: 

func apiKey() string {
    if key := os.Getenv("OPENAI_API_KEY"); key != "" {
        return key
    }
    return "local" // placeholder; llama.cpp server ignores it
}

Model Name Handling

Local models do not have the same names as OpenAI models. The model field in the request is typically ignored by llama.cpp (it always uses whatever model was loaded at startup), but you should still send something. Use an environment variable: 

func modelName() string {
    if m := os.Getenv("LLM_MODEL"); m != "" {
        return m
    }
    return "gpt-4o-mini" // default for OpenAI
}

Tool Calling with Local Models

Tool calling support varies by model. Well-supported options as of 2026:

  • Qwen2.5-7B/14B/72B-Instruct - excellent tool calling, follows complex schemas.
  • Llama-3.1/3.2-Instruct - good tool calling, built-in function call format.
  • Mistral-7B-Instruct - adequate, less consistent than the above.

llama.cpp translates the OpenAI tools format to the model's native function-calling template automatically, using the model's chat template from its GGUF metadata. If your model supports tool calling, the translation is transparent.

If tool calling is unreliable with a smaller model, add explicit instructions in the system prompt: 

systemPrompt := `You are a helpful assistant. You have access to tools.
When you need to use a tool, you MUST use the tool_calls mechanism.
Do NOT describe what you would do - actually call the tool.`

Embeddings with a Local Model

llama.cpp server exposes /v1/embeddings. Change the embedding URL to match: 

func embedURL() string {
    return apiBaseURL() + "/v1/embeddings"
}

Use a model designed for embeddings, not a chat model. nomic-embed-text and mxbai-embed-large are both available as GGUF files and perform well on retrieval benchmarks. Chat models can produce embeddings but their quality is lower. 

./build/bin/llama-server \
  --model ./models/nomic-embed-text-v1.5.Q8_0.gguf \
  --port 8081 \
  --embedding \
  --pooling mean

Performance Expectations

On a machine with a recent consumer GPU (RTX 3080 or better):

  • 7B model, 4-bit: ~50–80 tokens/second generation speed. Acceptable for interactive use.
  • 3B model, 8-bit: ~100–150 tokens/second. Fast.
  • 70B model, 4-bit (needs 40GB+ VRAM or split across GPUs): ~8–15 tokens/second. Slow but capable.

On CPU only: 3–7 tokens/second for a 7B model. Usable for batch jobs, too slow for interactive agents.

Summary

llama.cpp exposes an OpenAI-compatible API at http://localhost:8080 so the only change needed in your Go agent is the base URL. Download a GGUF quantized model from Hugging Face, start the server with --model and --port, and your existing code works. 4-bit quantization fits a 7B model in 4 GB of VRAM with minimal quality loss. Use local models for privacy, cost control, or offline use; use cloud models when you need the latest capabilities or reliable speed.
