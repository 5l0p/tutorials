openFrameworks - Part 22: OpenGL Core Profile
2026-10-13


Every draw call you have made in this series so far has gone through openFrameworks' compatibility
profile: a layer that works on any OpenGL 2.1 driver and automatically manages vertex state
behind the scenes. The core profile removes those compatibility shims and requires you to manage
vertex array objects, buffer objects, and shader programs yourself. That sounds like more work,
and it is - but it is also the foundation of modern GPU programming, gives you access to
compute shaders (Part 24), and removes driver overhead that adds up in scenes with many draw
calls. This article walks the complete path from core profile setup to a procedural mesh drawn
with your own VAO and VBO, without oF touching a single state variable.


What You Will Learn


  How to request an OpenGL 3.3 core profile in an oF application.
  What Vertex Array Objects (VAOs) and Vertex Buffer Objects (VBOs) are and why they exist.
  How to write a complete vertex and fragment shader for the programmable pipeline.
  How to upload geometry to the GPU with glBufferData.
  The difference between glDrawArrays and glDrawElements.
  A complete minimal example: a coloured triangle, then a procedural mesh.


Requesting a Core Profile


In main.cpp, replace the standard ofSetupOpenGL call with a version
that requests a core profile window:



// main.cpp
#include "ofMain.h"
#include "ofApp.h"
#include "ofGLWindowSettings.h"

int main() {
    ofGLWindowSettings settings;
    settings.setSize(1280, 720);
    settings.windowMode = OF_WINDOW;
    settings.setGLVersion(3, 3);    // Request OpenGL 3.3 core profile
    // Core profile is implied by setGLVersion(3, 3) in oF 0.12.x

    auto window = ofCreateWindow(settings);
    auto app    = std::make_shared<ofApp>();
    ofRunApp(window, app);
    return ofRunMainLoop();
}



In a core profile, gl_FragColor, ftransform(),
gl_MultiTexCoord0, and other GLSL 1.20 built-ins are gone. Use
#version 330 core in shaders and declare all inputs/outputs explicitly.


What VAOs and VBOs Are


A Vertex Buffer Object (VBO) is a block of GPU memory. You fill it with vertex data
(positions, normals, colours, UVs) by calling glBufferData. Subsequent draw
calls read directly from GPU memory - fast, with no CPU-to-GPU transfer per frame.



A Vertex Array Object (VAO) is a small GPU object that records how to
interpret a VBO's bytes: which attributes exist, what type they are, how many components
per attribute, and how many bytes from the start of one vertex to the start of the next
(the stride). You configure it once during setup; at draw time you just bind the VAO
and call glDrawArrays. The driver replays the recorded attribute configuration
automatically.



The practical effect: in compatibility mode, oF sets up vertex attribute state on every draw
call. With a VAO bound, the driver does it in one glBindVertexArray call. For
100,000 draw calls per frame the difference is measurable.


Core Profile Shader Pair


In GLSL 3.30 core:



  Vertex inputs are declared with in (replacing attribute).
  Data passed to the fragment shader is declared out in the vertex shader and in in the fragment shader (replacing varying).
  The fragment output colour is a declared out vec4 (replacing gl_FragColor).
  The model-view-projection matrix must be passed as a uniform; ftransform() is gone.



// data/shaders/core.vert
#version 330 core

layout(location = 0) in vec3 aPosition;
layout(location = 1) in vec3 aColor;

out vec3 vColor;

uniform mat4 uMVP;

void main() {
    vColor      = aColor;
    gl_Position = uMVP * vec4(aPosition, 1.0);
}



// data/shaders/core.frag
#version 330 core

in  vec3 vColor;
out vec4 fragColor;

void main() {
    fragColor = vec4(vColor, 1.0);
}


Complete Example: A Coloured Triangle


// ofApp.h
#pragma once
#include "ofMain.h"

class ofApp : public ofBaseApp {
public:
    void setup();
    void draw();
    void exit();

private:
    GLuint vao = 0;
    GLuint vbo = 0;
    ofShader shader;
};



// ofApp.cpp
#include "ofApp.h"

struct Vertex {
    float x, y, z;   // position
    float r, g, b;   // colour
};

void ofApp::setup() {
    ofBackground(20, 20, 30);
    ofSetFrameRate(60);

    shader.load("shaders/core");

    // Three vertices of a triangle
    const Vertex verts[] = {
        { 0.0f,  0.5f, 0.0f,   1.0f, 0.2f, 0.2f },   // top - red
        {-0.5f, -0.5f, 0.0f,   0.2f, 1.0f, 0.2f },   // bottom-left - green
        { 0.5f, -0.5f, 0.0f,   0.2f, 0.2f, 1.0f },   // bottom-right - blue
    };

    // Create and bind a VAO - it will record the attribute setup below
    glGenVertexArrays(1, &vao);
    glBindVertexArray(vao);

    // Create a VBO and upload vertex data
    glGenBuffers(1, &vbo);
    glBindBuffer(GL_ARRAY_BUFFER, vbo);
    glBufferData(GL_ARRAY_BUFFER, sizeof(verts), verts, GL_STATIC_DRAW);

    // Tell the VAO how to interpret the VBO bytes:
    // Attribute 0: position (3 floats, offset 0 bytes, stride = sizeof(Vertex))
    glEnableVertexAttribArray(0);
    glVertexAttribPointer(0, 3, GL_FLOAT, GL_FALSE, sizeof(Vertex),
                          (void*)offsetof(Vertex, x));

    // Attribute 1: colour (3 floats, offset = 3 floats * 4 bytes = 12 bytes)
    glEnableVertexAttribArray(1);
    glVertexAttribPointer(1, 3, GL_FLOAT, GL_FALSE, sizeof(Vertex),
                          (void*)offsetof(Vertex, r));

    // Unbind - configuration is now stored in the VAO
    glBindVertexArray(0);
    glBindBuffer(GL_ARRAY_BUFFER, 0);
}

void ofApp::draw() {
    // Build an identity MVP matrix (triangle in NDC space, no camera transform needed)
    glm::mat4 mvp(1.0f);   // identity = no transform

    shader.begin();
        shader.setUniformMatrix4f("uMVP", mvp);

        glBindVertexArray(vao);
        glDrawArrays(GL_TRIANGLES, 0, 3);   // 3 vertices, starting at index 0
        glBindVertexArray(0);
    shader.end();
}

void ofApp::exit() {
    glDeleteVertexArrays(1, &vao);
    glDeleteBuffers(1, &vbo);
}



GL_STATIC_DRAW tells the driver the data will be uploaded once and read many
times - appropriate for geometry that does not change. Use GL_DYNAMIC_DRAW for
buffers you update each frame (particle positions, animated meshes).


glDrawArrays vs glDrawElements


glDrawArrays(mode, first, count) draws count vertices starting at
first in the currently bound VBO. No index buffer; each vertex is used exactly
once. Suitable for small meshes where vertex sharing is not a concern.



glDrawElements(mode, count, type, offset) reads indices from a buffer bound to
GL_ELEMENT_ARRAY_BUFFER and uses them to look up vertices in the VBO. This lets
vertices be shared between triangles - essential for complex meshes where the index buffer
is smaller than duplicating vertices would be.



// Setup: add an index buffer alongside the VBO
const uint32_t indices[] = {0, 1, 2,   // first triangle
                             2, 1, 3};  // second triangle (shared edge 1-2)

GLuint ibo;
glGenBuffers(1, &ibo);
glBindBuffer(GL_ELEMENT_ARRAY_BUFFER, ibo);   // binds into the active VAO
glBufferData(GL_ELEMENT_ARRAY_BUFFER, sizeof(indices), indices, GL_STATIC_DRAW);

// Draw: use the index buffer
glBindVertexArray(vao);   // ibo binding is stored in the VAO
glDrawElements(GL_TRIANGLES, 6, GL_UNSIGNED_INT, 0);
glBindVertexArray(0);



Binding an GL_ELEMENT_ARRAY_BUFFER while a VAO is bound stores that binding
inside the VAO. Unbinding the VAO preserves it; subsequent glDrawElements calls
after glBindVertexArray(vao) automatically use the stored index buffer.


Procedural Mesh: Grid with Camera Transform


// Extended draw() using ofEasyCam for a 3D MVP matrix:
void ofApp::draw() {
    cam.begin();  // sets up oF's internal view and projection matrices

    // Retrieve the combined MVP from oF
    glm::mat4 model      = glm::mat4(1.0f);
    glm::mat4 view       = cam.getModelViewMatrix();
    glm::mat4 projection = cam.getProjectionMatrix();
    glm::mat4 mvp        = projection * view * model;

    shader.begin();
        shader.setUniformMatrix4f("uMVP", mvp);
        glBindVertexArray(vao);
        glDrawElements(GL_TRIANGLES, indexCount, GL_UNSIGNED_INT, 0);
        glBindVertexArray(0);
    shader.end();

    cam.end();
}


Interoperating with ofMesh in Core Profile


oF's ofMesh::draw() in core profile mode uses VAOs internally. You can mix
ofMesh for convenience and raw VAO/VBO code for performance-critical geometry
in the same sketch. The rule: whenever you call raw GL state-change functions (like
glBindVertexArray), ensure you call glBindVertexArray(0) afterwards
so oF's own draw calls start from a clean state.


Common Core Profile Errors


Black screen, no errors. The most common cause is that the VAO was bound during
glVertexAttribPointer but the wrong VBO was bound. Check that
glBindBuffer(GL_ARRAY_BUFFER, vbo) comes before
glVertexAttribPointer.



GLSL compile error: use of undeclared identifier 'gl_FragColor'. You are using a
GLSL 1.20 fragment shader with a 3.30 core context. Update to #version 330 core
and declare out vec4 fragColor;.



Error: 0:1(10): error: 'ftransform' undeclared. ftransform() is
compatibility-profile only. Pass the MVP matrix as a uniform and multiply manually.



Geometry drawn correctly in Debug, garbage in Release. Likely a stride or offset
calculation error that the debug driver tolerates but the release driver does not. Double-check
sizeof(Vertex) and offsetof(Vertex, field) with
static_assert:



static_assert(sizeof(Vertex) == 24, "Vertex layout changed unexpectedly");


Summary


The core profile removes the fixed-function compatibility layer, requiring explicit VAOs,
VBOs, and GLSL 3.30 shaders with declared inputs and outputs. A VAO records the vertex
attribute layout once; subsequent draw calls bind it and issue a single GL call.
glBufferData uploads geometry to GPU memory with a usage hint that drives driver
optimisation. glDrawArrays draws non-indexed geometry; glDrawElements
draws from an index buffer and enables vertex sharing. Always upload the MVP matrix as a
uniform in core profile - there is no ftransform(). Part 23 builds on this
foundation with framebuffer objects for render-to-texture, ping-pong buffers, and
post-processing shaders.
