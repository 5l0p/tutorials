Go - Part 5: Error Handling Best Practices

Error handling is the most discussed aspect of Go among developers coming from other languages. In C# you throw and catch exceptions; in C you check return codes or inspect errno; in Go errors are ordinary values returned from functions alongside the result. This is not a limitation - it is a deliberate design that makes control flow explicit and failure handling impossible to accidentally skip. Understanding how to use it well takes practice; this article covers the full toolkit.

What You Will Learn

  • The error interface: one method, Error() string
  • Creating errors with errors.New and fmt.Errorf
  • Wrapping errors with %w and unwrapping with errors.Is and errors.As
  • Sentinel errors: when to use package-level error values and their trade-offs
  • Custom error types that carry structured context
  • When to return an error up the stack versus log and discard it
  • The trade-offs of errors-as-values versus exceptions as seen from C and C#

The error Interface

The error type is a built-in interface with one method: 

type error interface {
    Error() string
}

Any type that implements Error() string is an error. The standard idiom is to return (result, error) and check the error immediately: 

f, err := os.Open("data.txt")
if err != nil {
    // handle the error - do not proceed
    return err
}
defer f.Close()
// use f

In C you check return codes (if (result < 0) { perror(...); }) but there is nothing stopping you from ignoring the return value entirely. In Go, if a function returns two values and you only capture one, the compiler forces you to use _ to explicitly discard the other. Ignoring errors is a visible act, not an invisible one.

In C# exceptions unwind the stack automatically. You can write a function that calls ten other functions without checking return values, and if any of them throw, the exception bubbles up. This is convenient but makes reasoning about which line actually failed harder. Go requires you to propagate errors explicitly - more code, but the execution path is always clear.

Creating Errors

The simplest way to create an error is errors.New: 

import "errors"

var ErrNotFound = errors.New("not found")

func findUser(id int) (*User, error) {
    if id <= 0 {
        return nil, errors.New("invalid user id")
    }
    // ...
}

For errors that include context, use fmt.Errorf: 

func readConfig(path string) (*Config, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        return nil, fmt.Errorf("readConfig: %w", err)
    }
    // ...
}

The %w verb wraps the error - the new error contains the original error and reports the original error's message as part of its own. This builds an error chain.

Wrapping and Unwrapping: errors.Is and errors.As

As errors propagate up the call stack, each layer adds context by wrapping. The errors.Is function checks whether any error in the chain matches a target: 

var ErrNotFound = errors.New("not found")

func getUser(id int) (*User, error) {
    // ...
    return nil, fmt.Errorf("getUser %d: %w", id, ErrNotFound)
}

func handleRequest(id int) {
    _, err := getUser(id)
    if errors.Is(err, ErrNotFound) {
        // This correctly matches even though err is wrapped
        fmt.Println("user not found")
    }
}

errors.As is used when you need access to the concrete error type, not just check identity: 

type ValidationError struct {
    Field   string
    Message string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("validation failed on %s: %s", e.Field, e.Message)
}

func process(input string) error {
    if input == "" {
        return fmt.Errorf("process: %w", &ValidationError{
            Field:   "input",
            Message: "must not be empty",
        })
    }
    return nil
}

err := process("")
var ve *ValidationError
if errors.As(err, &ve) {
    fmt.Println("Field:", ve.Field)     // "input"
    fmt.Println("Message:", ve.Message) // "must not be empty"
}

errors.As walks the error chain looking for a value assignable to the target type. In C# you would use catch (ValidationException e) or check ex.InnerException in a loop. Go's approach is more verbose but the chain structure is explicit.

Sentinel Errors

A sentinel error is a package-level error value used for comparison. The standard library uses these extensively: 

// From package io:
var EOF = errors.New("EOF")

// From package os:
var ErrNotExist = errors.New("file does not exist")
// (simplified - the real one is more complex)

Check for sentinel errors with errors.Is, not with == - because errors may be wrapped: 

data, err := os.ReadFile("config.json")
if errors.Is(err, os.ErrNotExist) {
    // file does not exist - create a default config
    data = []byte("{}")
} else if err != nil {
    return fmt.Errorf("reading config: %w", err)
}

Naming convention: sentinel errors defined in a package should be exported and named with the Err prefix (ErrNotFound, ErrTimeout). Do not name them just NotFound - the Err prefix distinguishes them from functions or types of similar names.

Custom Error Types

When you need to carry structured data with an error, implement the error interface on a struct. This is Go's equivalent of a C# exception class: 

type HTTPError struct {
    StatusCode int
    Status     string
    Body       string
}

func (e *HTTPError) Error() string {
    return fmt.Sprintf("HTTP %d: %s", e.StatusCode, e.Status)
}

func fetchURL(url string) ([]byte, error) {
    resp, err := http.Get(url)
    if err != nil {
        return nil, fmt.Errorf("fetchURL %s: %w", url, err)
    }
    defer resp.Body.Close()

    if resp.StatusCode >= 400 {
        body, _ := io.ReadAll(resp.Body)
        return nil, &HTTPError{
            StatusCode: resp.StatusCode,
            Status:     resp.Status,
            Body:       string(body),
        }
    }

    return io.ReadAll(resp.Body)
}

// Caller:
data, err := fetchURL("https://example.com/api")
var httpErr *HTTPError
if errors.As(err, &httpErr) {
    fmt.Println("Status:", httpErr.StatusCode)
}

When to Return vs When to Log

A common mistake is to both log an error and return it, leading to duplicate log lines at every level of the call stack. The convention is:

  • At a low level, return the error (possibly wrapped with context).
  • At the top level (main, HTTP handler, background worker), log the error and stop or respond.
  • Log once. Return or swallow, not both.
// Correct: add context and return
func readConfig(path string) (*Config, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        return nil, fmt.Errorf("readConfig %s: %w", err)
    }
    // ...
}

// At the boundary:
cfg, err := readConfig("config.json")
if err != nil {
    log.Fatalf("startup failed: %v", err) // log once, here
}

The Trade-Offs

Go's error handling is genuinely more verbose than C#'s exception model. A C# function that calls five operations needs no extra code for the error paths - exceptions propagate automatically. The Go equivalent requires five if err != nil checks. This is acknowledged even by Go's designers; proposals to improve the syntax have been discussed repeatedly.

The advantages are real though. Stack traces in Go errors are opt-in (using packages like github.com/pkg/errors or the newer golang.org/x/exp/errors with stack capture). Error paths are visible in code review. No hidden control flow means the performance profile is predictable - there is no cost for the happy path analogous to setting up exception tables in a C# binary.

From a C perspective, Go errors are much safer than errno (which is global, thread-local only in recent C standards, and easily forgotten). Go errors are explicit values tied to the operation that produced them.

Summary

Go errors are values — just return them alongside the result. Wrap with fmt.Errorf("...: %w", err) to add context; unwrap with errors.Is (identity check) and errors.As (type extraction). Sentinel errors (var ErrNotFound = errors.New(...)) are fine for stable public APIs; custom types are better when callers need structured data. Return errors upward; log only at boundaries where you have the full context. The verbosity compared to exceptions is intentional — it makes every failure path visible.
