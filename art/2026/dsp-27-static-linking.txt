DSP - Part 27: Static Linking and Distribution
2026-06-01


A binary that depends on shared libraries installed on the user's machine is fragile:
wrong version, wrong path, or no package manager. This article covers how to produce
a self-contained binary from the C/C++ code in this series — one that a user can
copy to any compatible machine and run without installing anything.

Static vs Dynamic Linking: A Recap


A dynamically linked binary records the names of shared libraries it needs
(libsndfile.so.1, libm.so.6). The dynamic linker resolves
those names at runtime by searching standard paths. If a library is missing or the
wrong version, the binary fails to start.



A statically linked binary has all library code copied into the executable at
link time. It carries its own libc, libm, and any other dependency. It is larger but
runs on any compatible CPU without installation.


Auditing Dependencies Before You Ship


Always audit what your binary links against before distribution.


Linux — ldd:


$ ldd build/release/synth
        linux-vdso.so.1 (0x00007ffd8b9f3000)
        libsndfile.so.1 => /usr/lib/x86_64-linux-gnu/libsndfile.so.1
        libm.so.6 => /lib/x86_64-linux-gnu/libm.so.6
        libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
        /lib64/ld-linux-x86-64.so.2



Any line that is not linux-vdso or ld-linux is a dependency
you need to either vendor or accept as a requirement.


macOS — otool:


$ otool -L build/release/synth
build/release/synth:
        /usr/local/lib/libsndfile.1.dylib
        /usr/lib/libSystem.B.dylib


Windows — dumpbin (MSVC tools):


dumpbin /dependents synth.exe


Static Linking on Linux


Pass -static to the linker. Every library on the link line must have a
static version (.a file) available:



CFLAGS  = -O3 -DNDEBUG -ffast-math
LDFLAGS = -static
LDLIBS  = third_party/libsndfile/build/src/.libs/libsndfile.a \
          -lm -lc

$(TARGET): $(OBJS)
	$(CC) $(LDFLAGS) -o $@ $^ $(LDLIBS)



After linking, verify with ldd:



$ ldd build/release/synth
        not a dynamic executable



If ldd still lists shared libraries, a dependency pulled in a transitive
shared-only library. Find the culprit with ldd -v and replace it with a
static build.


Using musl libc for smaller, more portable static binaries:


glibc static binaries have version requirements — a binary statically linked against
glibc 2.35 will not run on a system with glibc 2.17. musl libc static binaries have
no such requirement. Install the musl toolchain and compile:



CC      = musl-gcc
CFLAGS  = -O2 -static
LDFLAGS = -static



Or use a Docker image based on Alpine Linux (which uses musl by default) to produce
portable static binaries for distribution.


Partial Static Linking


Fully static binaries can be problematic with glibc (NSS lookups for DNS, users, etc.
require dynamic loading). A common compromise is to statically link everything except
glibc and libpthread:



LDFLAGS = -Wl,-Bstatic
LDLIBS_STATIC  = third_party/libsndfile/build/src/.libs/libsndfile.a \
                 -lm
LDLIBS_DYNAMIC = -Wl,-Bdynamic -lpthread

$(TARGET): $(OBJS)
	$(CC) $(LDFLAGS) -o $@ $^ $(LDLIBS_STATIC) $(LDLIBS_DYNAMIC)



-Wl,-Bstatic tells the linker to prefer static archives; -Wl,-Bdynamic
switches back to preferring shared libraries for the libraries listed after it.


macOS: Fixing Dylib Paths


macOS dynamic libraries embed their install name — the path the dynamic linker uses
to find them. If a dependency was installed in a non-standard path, the install name
may be an absolute path that does not exist on the user's machine.



Check the install name of a dylib:



$ otool -D /usr/local/lib/libsndfile.1.dylib
/usr/local/lib/libsndfile.1.dylib



To bundle the dylib alongside the binary and use a relative path, use
@executable_path:



# Copy the dylib into the app bundle:
cp /usr/local/lib/libsndfile.1.dylib ./dist/

# Change the install name embedded in the dylib:
install_name_tool -id "@executable_path/libsndfile.1.dylib" ./dist/libsndfile.1.dylib

# Change the reference in the binary:
install_name_tool -change \
    /usr/local/lib/libsndfile.1.dylib \
    @executable_path/libsndfile.1.dylib \
    ./dist/synth

# Verify:
otool -L ./dist/synth



For a macOS app bundle, use @rpath and set the rpath to
@executable_path/../Frameworks so the dylib lives in the
Frameworks/ directory inside the .app bundle.


Windows: Bundling DLLs


On Windows, the simplest distribution strategy is to place all required DLLs in the
same directory as the .exe. The Windows loader searches the executable
directory first.



# List required DLLs with dumpbin or Dependency Walker, then copy them:
mkdir dist
copy build\release\synth.exe dist\
copy C:\msys64\mingw64\bin\libsndfile-1.dll dist\
copy C:\msys64\mingw64\bin\libgcc_s_seh-1.dll dist\
copy C:\msys64\mingw64\bin\libstdc++-6.dll dist\
copy C:\msys64\mingw64\bin\libwinpthread-1.dll dist\



When using MinGW on Windows, statically link the MinGW runtime to avoid bundling
libgcc_s_seh-1.dll and friends:



LDFLAGS = -static-libgcc -static-libstdc++ -static -lpthread


Stripping Debug Symbols


A release binary compiled with -g (or left with default debug info)
contains symbol names, source file paths, and line number tables. Strip them to reduce
binary size:



strip --strip-all build/release/synth        # Linux
strip build/release/synth                    # macOS (same command)
strip synth.exe                              # Windows (MinGW strip)



On Linux, --strip-all removes all symbols including the dynamic symbol
table (safe for executables, not for shared libraries). For shared libraries, use
--strip-unneeded to preserve the symbols the dynamic linker needs.



Integrate stripping into the Makefile release target:



ifeq ($(MODE),release)
POST_BUILD = strip --strip-all $(TARGET)
endif

$(TARGET): $(OBJS)
	$(CC) $(LDFLAGS) -o $@ $^ $(LDLIBS)
	$(POST_BUILD)



Keep an unstripped copy or the debug build for post-crash analysis. Stripped binaries
still produce crash addresses that can be mapped to source lines with addr2line
if you retain the debug build alongside the stripped release.


UPX Compression


UPX is an executable packer that compresses the binary and adds a small self-extracting
stub. It can reduce binary size by 50–70% at the cost of a small startup delay
(typically 10–50 ms) while the stub decompresses:



sudo apt install upx-ucl   # or: brew install upx

upx --best --lzma build/release/synth



Verify the binary still runs after compression. Some antivirus software flags UPX-packed
binaries; this is a trade-off to consider for software distributed to non-technical users.



For plugin formats like VST3 (.vst3 bundle on macOS/Linux, .vst3 directory on Windows),
do not compress the plugin binary — plugin hosts often do their own inspection of the
binary's sections and will reject a UPX stub.


Cross-Compilation for Linux on macOS or Windows


To produce a Linux binary from a macOS or Windows developer machine, use a Docker
container with the target toolchain:



# Build a static Linux x86_64 binary using a Debian container:
docker run --rm -v $(pwd):/work -w /work debian:bookworm-slim \
  bash -c "apt-get update -qq && \
           apt-get install -y gcc make libsndfile1-dev && \
           make MODE=release LDFLAGS=-static"



For ARM (e.g. Raspberry Pi), install the cross-compiler:



sudo apt install gcc-aarch64-linux-gnu

make CC=aarch64-linux-gnu-gcc MODE=release LDFLAGS=-static


Distribution Checklist


Run ldd / otool -L / dumpbin on the release binary and confirm all dependencies are either static or bundled.
Test the binary on a clean machine (a VM or Docker container) without development libraries installed.
Strip debug symbols from the release binary; keep the unstripped build for debugging.
Record the compiler version, OS, and build flags used to produce the release binary.
Include licence files for all statically linked dependencies in the distribution archive.
Test the binary on the oldest OS version you intend to support.
If using glibc, check the minimum required glibc version with objdump -T synth | grep GLIBC.
For VST3/audio plugins: verify the plugin loads in at least one host on each target platform.


Summary


Audit runtime dependencies before shipping with ldd, otool -L,
or dumpbin. On Linux, pass -static to the linker and provide
static archives for all dependencies; use musl libc for binaries that must run on a
wide range of kernels. On macOS, use install_name_tool to rewrite dylib
paths to @executable_path for bundled dylibs. On Windows, copy DLLs into
the same directory as the .exe, or statically link the MinGW runtime.
Strip debug symbols from the release binary to reduce size. Consider UPX compression
for command-line tools but not for plugin formats. Test on a clean machine before
distribution.
