Containers - Part 4: PID Namespaces
2026-08-02


Every process on a Linux system has a PID. PID namespaces let the kernel
maintain a separate numbering scheme per namespace, so the first process in a
container is PID 1 even though the host kernel knows it as (say) PID 48372.
This article covers how that numbering works, the special role of PID 1 inside
a namespace, and the zombie-reaping responsibility that comes with it.


What You Will Learn


How to create a PID namespace with clone(CLONE_NEWPID) and verify the child is PID 1

How to read the NSpid field in /proc/<pid>/status to see both host and container PIDs
The three special responsibilities of PID 1: signal handling, zombie reaping, and namespace lifetime

Creating a PID Namespace


A new PID namespace is created by passing CLONE_NEWPID to clone().
The child process becomes PID 1 inside the new namespace. All further processes
it creates are numbered 2, 3, 4, … inside the namespace while the host kernel
assigns them real PIDs from its own global table.



#define _GNU_SOURCE
#include <sched.h>
#include <stdio.h>
#include <stdlib.h>
#include <sys/types.h>
#include <sys/wait.h>
#include <unistd.h>

#define STACK_SIZE (1024 * 1024)

static int child_main(void *arg) {
    (void)arg;
    printf("[container] getpid() = %d\n", getpid());
    printf("[container] getppid() = %d\n", getppid());
    /* getppid() returns 0 because the parent is in a different namespace */

    /* Exec a shell; it will be PID 2 if we don't exec */
    char *argv[] = { "/bin/sh", NULL };
    execv("/bin/sh", argv);
    perror("execv");
    return 1;
}

int main(void) {
    char *stack = malloc(STACK_SIZE);
    if (!stack) { perror("malloc"); return 1; }

    pid_t pid = clone(child_main,
                      stack + STACK_SIZE,
                      CLONE_NEWPID | SIGCHLD,
                      NULL);

    if (pid == -1) { perror("clone"); free(stack); return 1; }

    printf("[host] child's host PID = %d\n", pid);

    int status;
    waitpid(pid, &status, 0);
    free(stack);
    return 0;
}



Compile and run as root. You will see the child print PID 1. The host prints the
real kernel PID. The child's getppid() returns 0 because its parent (the
host process) is outside the namespace and therefore invisible to it.


The /proc Visibility Problem


Immediately after entering a new PID namespace, /proc still shows the
host's process table because the /proc filesystem was mounted in the
parent's namespace. To get a namespace-correct view you must remount
/proc inside the new namespace:



/* Inside child_main, after any required setup */
if (mount("proc", "/proc", "proc", 0, NULL) == -1) {
    perror("mount /proc");
    return 1;
}



After this mount, ls /proc shows only the PIDs visible in the new
namespace. The shell inside will show only its own process and any children it
spawns. This is why container runtimes always mount a fresh /proc after
creating the PID namespace.



# Shell inside the container after remounting /proc
ps aux
# Shows only the shell and ps itself

# Compare with the host
# sudo nsenter --pid=/proc/<container-init-pid>/ns/pid -- ps aux


NSpid: The Dual PID


The kernel records all PID namespace memberships for a process in
/proc/<pid>/status under the NSpid field. The values are listed
from the outermost namespace to the innermost:



# On the host, find the container's init process
grep NSpid /proc/$(pgrep -n sh)/status
# NSpid: 48372    1



First column: host PID. Second column: PID inside the container's namespace.
If namespaces are nested three levels deep, there will be three numbers.



#define _GNU_SOURCE
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

void show_nspid(pid_t pid) {
    char path[64];
    char line[256];
    snprintf(path, sizeof(path), "/proc/%d/status", pid);

    FILE *f = fopen(path, "r");
    if (!f) { perror("fopen"); return; }

    while (fgets(line, sizeof(line), f)) {
        if (strncmp(line, "NSpid:", 6) == 0) {
            printf("%s", line);
            break;
        }
    }
    fclose(f);
}


PID 1 and Its Responsibilities


In every PID namespace, PID 1 is special in the same way that init (or
systemd) is special on the host:



Signal handling - signals sent to PID 1 that have no registered handler are silently dropped. SIGKILL and SIGSTOP cannot be caught or ignored, but SIGTERM to a PID-1 process that does not handle it does nothing. Many Docker containers use a minimal init process (tini, dumb-init) for exactly this reason.
Zombie reaping - when any process in the namespace dies and its parent does not call wait(), the child becomes a zombie and is reparented to PID 1. PID 1 must call waitpid(-1, ...)` repeatedly to reap them, or the namespace's process table leaks zombie entries.
Namespace lifetime - if PID 1 exits, the kernel sends SIGKILL to every other process in the namespace and then destroys the namespace.


Zombie Reaping in C


A minimal PID-1 process that reaps zombies in a loop:



#define _GNU_SOURCE
#include <signal.h>
#include <stdio.h>
#include <sys/types.h>
#include <sys/wait.h>
#include <unistd.h>

static volatile int running = 1;

static void sigterm_handler(int sig) {
    (void)sig;
    running = 0;
}

int pid1_main(void) {
    struct sigaction sa = { .sa_handler = sigterm_handler };
    sigaction(SIGTERM, &sa, NULL);
    sigaction(SIGINT,  &sa, NULL);

    /* Fork and exec the actual container workload */
    pid_t workload = fork();
    if (workload == 0) {
        char *argv[] = { "/bin/sh", "-c", "exec sleep 3600", NULL };
        execv("/bin/sh", argv);
        _exit(1);
    }

    /* Reap loop */
    while (running) {
        int status;
        pid_t died = waitpid(-1, &status, WNOHANG);
        if (died == workload) running = 0;
        if (died <= 0) sleep(1);
    }
    return 0;
}


Nested PID Namespaces


PID namespaces can be nested. A process in an inner namespace is visible from
all ancestor namespaces with different PIDs. A process in an outer namespace
cannot directly see inside a deeper namespace - but the outer namespace's PID 1
can send signals to any PID it knows.



The nesting depth is limited by PID_NS_MAX_LEVEL, currently 32 in the
kernel sources. Each level adds a PID entry in the NSpid field.


Practical: Inspecting a Running Container


# Find the PID of a running Docker container's init
CPID=$(docker inspect --format '{{.State.Pid}}' my-container)

# See its namespace memberships
ls -la /proc/$CPID/ns/

# See how it looks from inside its PID namespace
sudo nsenter --pid=/proc/$CPID/ns/pid --mount=/proc/$CPID/ns/mnt \
    /bin/sh -c 'ps aux'

# Read the NSpid field
grep NSpid /proc/$CPID/status


Summary


PID namespaces give a container its own process numbering starting at PID 1.
The kernel records both the host PID and the in-namespace PID in
/proc/<pid>/status under NSpid. The process at PID 1 has special
responsibilities: unhandled signals are silently dropped, orphaned processes are
reparented to it, and when it exits the kernel kills everything else in the
namespace. Always remount /proc after entering a PID namespace, and
consider a minimal init (tini, dumb-init) to handle zombie reaping correctly.
