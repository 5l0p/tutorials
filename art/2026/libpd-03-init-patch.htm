libpd - Part 3: Initialising libpd & Loading a Patch

Before libpd can process audio it needs to be initialised and told about the audio format. Then you load one or more Pd patches and optionally send initial configuration messages before the first audio block arrives. This article covers the correct startup sequence and explains the instance model for applications that need to run multiple independent Pd engines.

  • What libpd_init does and why it must be called exactly once per instance

The Startup Sequence

Every libpd host follows the same four steps at startup:

  1. Call libpd_init() - registers built-in objects, sets up the scheduler.
  2. Call libpd_init_audio() - sets channel count and sample rate.
  3. Call libpd_openfile() - loads a .pd patch from disk.
  4. Send any initial configuration messages via libpd_float() etc.

After these steps, you call libpd_process_float() from your audio callback. Do not call libpd_process_float before all four steps are complete - the engine will produce silence or crash.

libpd_init

#include <z_libpd.h>

int main(void) {
    libpd_init();
    /* ... */
}

Call libpd_init() exactly once per process. It registers all built-in Pd objects (oscillators, filters, arithmetic operators, MIDI, etc.). Calling it twice in one process is safe but redundant. If you use the multiple-instance API, call libpd_init() before creating any instance.

libpd_init_audio

int err = libpd_init_audio(
    int inChannels,   /* number of input channels  (0 = no input) */
    int outChannels,  /* number of output channels (2 = stereo)   */
    int sampleRate    /* e.g. 44100 or 48000                       */
);
if (err != 0) { /* handle error */ }

The channel counts must match the buffers you pass to libpd_process_float. It is fine to call libpd_init_audio again to change the format - for example, when the user changes audio settings in your app - but you must reload the patch afterwards, because changing the audio format tears down the DSP graph.

libpd_openfile

void *patch = libpd_openfile("synth.pd", "/home/user/patches");
if (patch == NULL) {
    fprintf(stderr, "Could not open patch\n");
    return 1;
}

The first argument is the filename (not a full path); the second is the directory. libpd searches for the file in that directory and also in any directories added with libpd_add_to_search_path(). The returned pointer is an opaque handle - keep it until you want to close the patch.

To close a patch:

libpd_closefile(patch);
patch = NULL;

Closing a patch while the audio callback is running is unsafe. Stop audio processing first, or protect access with a mutex.

Adding Search Paths

Pd objects can load abstractions (sub-patches) and external objects from directories on the search path. Add directories before opening any patch:

libpd_add_to_search_path("/usr/lib/pd/extra");
libpd_add_to_search_path("/home/user/patches/abstractions");

Sending Initial Messages

After opening a patch you often need to configure its internal state before audio starts - setting a tempo, choosing a waveform, loading a sample path. The message functions let you do this immediately:

/* Simple scalar */
libpd_float("tempo", 120.0f);

/* A symbol (string) */
libpd_symbol("samplepath", "/sounds/kick.wav");

/* A bang - triggers patch initialization */
libpd_bang("init");

These calls are synchronous: they inject the message directly into the Pd scheduler and are processed before the next audio tick. You can send messages before libpd_process_float is ever called.

Sending Lists and Typed Messages

For messages with multiple values (a Pd list) use the start/add/finish API:

libpd_start_message(3);       /* allocate space for 3 elements */
libpd_add_float(440.0f);      /* element 0: frequency          */
libpd_add_float(0.5f);        /* element 1: amplitude          */
libpd_add_symbol("sine");     /* element 2: waveform name      */
libpd_finish_list("osc-config");

For typed messages (first atom is the selector):

libpd_start_message(2);
libpd_add_float(1.0f);
libpd_add_float(0.0f);
libpd_finish_message("dsp", "state");
/* equivalent to: [; dsp state 1 0 ] in Pd */

Multiple Instances

The default libpd API uses a single global Pd instance. For applications that host multiple independent patches (plugin hosts, DAWs, per-track instruments), libpd provides an instance API:

#include <z_libpd.h>

t_pdinstance *instA = libpd_new_instance();
t_pdinstance *instB = libpd_new_instance();

/* Configure instance A */
libpd_set_instance(instA);
libpd_init_audio(0, 2, 44100);
void *patchA = libpd_openfile("piano.pd", "/patches");

/* Configure instance B */
libpd_set_instance(instB);
libpd_init_audio(0, 2, 44100);
void *patchB = libpd_openfile("drums.pd", "/patches");

Every libpd call after libpd_set_instance() operates on the selected instance until you switch. This is a global setting - it is not thread-safe if two threads call libpd_set_instance concurrently. The recommended pattern is to dedicate one thread per instance and never switch from that thread.

Instance Lifecycle

/* Tear down an instance */
libpd_set_instance(instA);
libpd_closefile(patchA);
libpd_free_instance(instA);
instA = NULL;

Free instances in reverse creation order when possible, though libpd does not strictly require it. Always close the patch before freeing the instance.

Complete Startup Example

#include <z_libpd.h>
#include <stdio.h>

#define SAMPLE_RATE  44100
#define IN_CHANNELS  0
#define OUT_CHANNELS 2

static void *g_patch = NULL;

int audio_init(void) {
    libpd_init();

    if (libpd_init_audio(IN_CHANNELS, OUT_CHANNELS, SAMPLE_RATE) != 0) {
        fprintf(stderr, "libpd_init_audio failed\n");
        return -1;
    }

    libpd_add_to_search_path("patches/abstractions");

    g_patch = libpd_openfile("main.pd", "patches");
    if (g_patch == NULL) {
        fprintf(stderr, "Could not open main.pd\n");
        return -1;
    }

    /* Send initial state */
    libpd_float("master_gain", 0.8f);
    libpd_bang("start");

    return 0;
}

void audio_shutdown(void) {
    if (g_patch) {
        libpd_closefile(g_patch);
        g_patch = NULL;
    }
}

Error Handling

libpd does not throw exceptions. Check return values:

  • libpd_init_audio returns 0 on success, non-zero on failure.
  • libpd_openfile returns NULL on failure (file not found, parse error).
  • libpd_float and libpd_bang return 0 on success, -1 if the receiver is not found (not fatal - the patch may not have subscribed yet).

To capture Pd's own error and print output, set a print hook before calling libpd_init:

static void on_print(const char *msg) {
    fprintf(stderr, "[pd] %s", msg);
}

libpd_set_printhook(on_print);
libpd_init();

Summary

Call libpd_init once at startup, then libpd_init_audio with your channel counts and sample rate. Open a patch with libpd_openfile — it returns a handle you must keep until you call libpd_closefile. Send any initial parameters to the patch as float or bang messages before the first audio block. For multiple independent engines, use libpd_new_instance and libpd_set_instance to switch context; all libpd API calls operate on the currently-bound instance.
