LLM - Part 15: LangChain & LangGraph Concepts

LangChain (Python/JavaScript) is the most widely deployed LLM application framework. You will encounter it in tutorials, job listings, and production codebases. There is no official Go port worth using in production, but understanding LangChain's abstractions makes you a better agent developer: the framework crystallised concepts that were once scattered across papers and blog posts. This article is a conceptual tour - no code that requires LangChain, only Go analogues and comparisons.

What You Will Learn

  • LangChain's core abstractions: Chains, PromptTemplates, OutputParsers, Agents, and Tools
  • How LangGraph extends LangChain with stateful, cyclical execution graphs
  • How each LangChain concept maps to the Go primitives built in this series
  • The LCEL (LangChain Expression Language) pipeline syntax and its Go equivalent
  • When to use a framework and when building your own agent loop is the better choice

What LangChain Provides

LangChain is a collection of abstractions over the common patterns in LLM applications. Its core ideas:

  • Chains - composable pipelines where each step takes input and produces output, with one step feeding the next.
  • Agents - chains that use a model to decide which tool to call at each step, rather than following a fixed sequence.
  • Tools - functions the agent can invoke (identical in concept to what we built in article 03).
  • Memory - components for persisting conversation state (article 10).
  • Retrievers - abstractions over vector stores and other retrieval backends (article 11).
  • Prompt Templates - reusable prompt structures with variable substitution.

These are the same concepts you have implemented from scratch over the past fourteen articles. LangChain provides them as a library with many pre-built connectors (OpenAI, Anthropic, Pinecone, PostgreSQL, etc.). The trade-off is convenience vs transparency.

Chains

A chain is a sequence of operations where the output of one step becomes the input of the next. The simplest chain is a single LLM call. A more complex chain: 

Input text
  → PromptTemplate (format a prompt)
  → LLM (generate a response)
  → OutputParser (extract structured data)
  → Result

In Go (article 16 implements this), a chain is: 

type Chain interface {
    Run(ctx context.Context, input string) (string, error)
}

A sequential chain runs a slice of chains in order, passing each output to the next: 

type SequentialChain struct {
    Steps []Chain
}

func (sc *SequentialChain) Run(ctx context.Context, input string) (string, error) {
    current := input
    for _, step := range sc.Steps {
        var err error
        current, err = step.Run(ctx, current)
        if err != nil {
            return "", err
        }
    }
    return current, nil
}

The Runnable Protocol

LangChain Expression Language (LCEL) extended the chain concept with a pipe operator: prompt | llm | parser. Each component implements a common Runnable interface. The pipe operator composes runnables left to right.

This is analogous to Go function composition or method chaining. The elegance is that anything implementing Runnable can appear anywhere in a pipeline - you can swap an LLM for a retriever, or a parser for a database lookup, without changing surrounding code.

LangGraph: Graphs Instead of Chains

LangChain's chains are linear: step A always leads to step B. LangGraph (built on top of LangChain) replaces the linear sequence with a directed graph where nodes are processing steps and edges can be conditional. A node can emit different edges based on its output, enabling branching, looping, and dynamic routing.

A LangGraph agent is a graph with three node types:

  • Agent node - calls the model and decides what to do. If the model returns tool calls, it emits the "tools" edge. If it produces final text, it emits the "end" edge.
  • Tool node - executes the requested tools. Emits the "agent" edge when done, sending results back to the agent node.
  • END - a terminal node; the graph stops here.

This graph structure is exactly the agent loop from article 04 - but expressed as a graph so it can be visualised, debugged, and extended without touching the loop code. Adding a "human review" node between the agent and tool nodes, for example, requires inserting a new node and redirecting two edges. In a monolithic loop it would require modifying the loop body.

Agent Patterns in LangGraph

LangGraph supports several named patterns:

ReAct (Reason + Act) - the model alternates between reasoning steps ("I need to find out X") and action steps (tool calls). The reasoning is made explicit in the model's output. This is the default pattern for most LangGraph agents and corresponds closely to the agent loop you have already built.

Plan and Execute - a planner agent generates a task list; separate executor agents work through the tasks in order. The planner does not re-evaluate after each task. Good for long, structured tasks. Article 17 implements this in Go.

Reflection - after generating a response, the model critiques its own output and optionally revises it. The reflection loop runs until the model judges the response satisfactory. Good for writing tasks, code generation, and structured output validation.

Multi-Agent - multiple specialised agents collaborate. A supervisor routes tasks to sub-agents based on their expertise. Article 19 covers this pattern.

Prompt Templates

LangChain's PromptTemplate is a string with named placeholders: 

template = "You are a {role}. Answer the following question: {question}"
prompt = template.format(role="Go expert", question="What is a goroutine?")

In Go (implemented in article 16): 

type PromptTemplate struct {
    Template string
}

func (pt *PromptTemplate) Format(vars map[string]string) string {
    result := pt.Template
    for k, v := range vars {
        result = strings.ReplaceAll(result, "{"+k+"}", v)
    }
    return result
}

Templates enforce prompt consistency across an application. When multiple places in your code need to format the same kind of prompt, centralising it in a template prevents drift.

When LangChain Adds Value vs When It Adds Opacity

LangChain is a large framework with many layers of abstraction. The opaqueness is a real cost: stack traces through LangChain are hard to read, behaviour can be surprising when deep defaults interact, and debugging requires understanding framework internals.

Use LangChain when:

  • You need connectors to many external services (Pinecone, Redis, Notion, etc.) and do not want to write them yourself.
  • Your team is already fluent in Python and wants to move fast.
  • You need LangGraph's graph-based orchestration for complex multi-agent flows.

Build your own when:

  • You need to understand exactly what is happening on the wire.
  • Latency matters - every abstraction layer adds overhead.
  • Your use case is narrow and you would use only a small fraction of the framework.
  • You want to run in Go (or any language without a mature LangChain port).

The articles in this series take the "build your own" path. Not because frameworks are bad, but because understanding the primitives makes you better at using frameworks when you do reach for them.

Summary

LangChain formalises four concepts: a Chain (a callable step), a PromptTemplate (an interpolated string), an OutputParser (a structured-output transformer), and an Agent (a chain that loops until it decides to stop). LangGraph adds statefulness and cycles to chains, enabling patterns like self-critique and supervisor delegation. Every LangChain abstraction maps to something in this series — the value of the framework is discoverability and ecosystem integrations, not fundamental new ideas.
