Lua - Part 4: Calling C from Lua
2026-11-25


Part 3 showed how to call Lua from C. This article flips the direction: making C
functions callable from Lua. When you expose a C function to Lua, the scripting layer
can call your audio processing code, your physics engine, your platform I/O - anything
you choose to surface. The mechanism is a single function pointer type,
lua_CFunction, and a small set of API calls to read arguments off the
stack, push return values back onto it, and register the function under a Lua name.
This article covers that mechanism fully, then shows the canonical pattern for building
a named Lua module from a table of C functions.


What You Will Learn


The lua_CFunction signature and why it looks the way it does.
How to read typed arguments with luaL_checkinteger, luaL_checknumber, and luaL_checkstring.
How to push return values and what the integer return means.
How to register individual functions with lua_register.
How to build a module table with lua_createtable, luaL_Reg, and luaL_setfuncs.
How to signal errors from C to Lua with luaL_error.


The lua_CFunction Signature


Every function callable from Lua has the same C type:



typedef int (*lua_CFunction)(lua_State *L);



It takes a lua_State pointer and returns an int. That is the
entire contract. The arguments passed from Lua code are on the stack when the function
is called; return values are pushed onto the stack before returning; and the return value
of the C function is the number of Lua return values it pushed. This is not a mistake -
Lua functions can return any number of values, and the C function explicitly tells the
VM how many it left on the stack.



Consider a C function that adds two numbers:



/* lua_add: called from Lua as math_ext.add(a, b) */
/* stack on entry: [a, b]                          */
static int lua_add(lua_State *L) {
    /* read arguments - indices 1 and 2 are the first and second args */
    lua_Number a = luaL_checknumber(L, 1);   /* stack: [a, b] */
    lua_Number b = luaL_checknumber(L, 2);   /* stack: [a, b] */

    lua_pushnumber(L, a + b);               /* stack: [a, b, result] */
    return 1;                               /* 1 return value on the stack */
}



The argument indices start at 1, not 0. Index 1 is the first argument, index 2 the
second, and so on. The luaL_check* functions both read and validate: if
index 1 is not a number, luaL_checknumber raises a Lua error with a
descriptive message ("bad argument #1 to 'add' (number expected, got string)") - your
function never sees a wrong-typed argument.


Reading Arguments: The luaL_check Family


The luaL_check* functions are the workhorses for reading function arguments.
They raise a Lua error immediately if the argument at the given index is not the expected
type, so the C function body can assume the arguments are valid:



#include <lua.h>
#include <lualib.h>
#include <lauxlib.h>

static int demo_args(lua_State *L) {
    /* luaL_checkinteger: expects an integer at index n, returns lua_Integer */
    lua_Integer i = luaL_checkinteger(L, 1);

    /* luaL_checknumber: expects a number (int or float) at index n, returns lua_Number */
    lua_Number  f = luaL_checknumber(L, 2);

    /* luaL_checkstring: expects a string at index n, returns const char * */
    /* The string is managed by Lua - do not free it, do not store it past the call */
    const char *s = luaL_checkstring(L, 3);

    /* luaL_checkboolean does not exist - use lua_toboolean (no type error raised) */
    /* or check explicitly: */
    luaL_checktype(L, 4, LUA_TBOOLEAN);
    int b = lua_toboolean(L, 4);

    /* Optional argument with default - luaL_optinteger returns the default if nil */
    lua_Integer opt = luaL_optinteger(L, 5, 42);

    printf("i=%lld f=%.2f s=%s b=%d opt=%lld\n",
           (long long)i, (double)f, s, b, (long long)opt);
    return 0;   /* no return values */
}



The luaL_opt* variants accept a default value and return it when the
argument at that index is absent or nil. Use them for optional trailing arguments.



One important restriction on luaL_checkstring: the returned
const char * points into Lua's internal string storage. It is valid only
as long as the string value remains on the stack. If you need to keep the string after
the C function returns, copy it with strdup or into your own buffer.


Pushing Return Values


After doing its work, a C function pushes its return values onto the stack and returns
the count. The values must be pushed in order - first return value first.



/* Returns two values: quotient and remainder */
static int lua_divmod(lua_State *L) {
    lua_Integer a = luaL_checkinteger(L, 1);
    lua_Integer b = luaL_checkinteger(L, 2);

    if (b == 0) {
        /* Raise a Lua error - never returns */
        return luaL_error(L, "divmod: division by zero");
    }

    lua_pushinteger(L, a / b);   /* first return value  */
    lua_pushinteger(L, a % b);   /* second return value */
    return 2;                    /* two values on the stack */
}



-- Lua side:
local q, r = math_ext.divmod(17, 5)
print(q, r)   -- 3   2



Returning 0 means "no return values" - the caller gets nil for any variables it tries
to receive into. There is no constraint on the number; you can return 0, 1, or many
values from a single C function.


Registering with lua_register


The simplest way to expose a single C function as a global Lua name is
lua_register:



/* lua_register(L, name, func) is a macro that expands to:
 *   lua_pushcfunction(L, func);
 *   lua_setglobal(L, name);
 */

lua_register(L, "divmod", lua_divmod);



-- Now callable from Lua as a global:
local q, r = divmod(100, 7)



This works but pollutes the global namespace. For any real module, you want all your
functions namespaced under a table: mylib.func() not func().


Building a Module Table with luaL_Reg and luaL_setfuncs


The canonical pattern for a C module is: define a null-terminated array of
luaL_Reg structs (name + function pointer pairs), create a table, and
register all functions into it at once with luaL_setfuncs. This is the
same pattern used by Lua's own standard libraries.



/* luaL_Reg is just: struct { const char *name; lua_CFunction func; } */

static const luaL_Reg mathext_funcs[] = {
    { "add",         lua_add        },
    { "divmod",      lua_divmod     },
    { "clamp",       lua_clamp      },
    { "lerp",        lua_lerp       },
    { "db_to_linear",lua_db_to_linear },
    { NULL, NULL }   /* sentinel - marks end of the array */
};

/* luaopen_mathext: the module loader function */
/* Convention: name it luaopen_<modulename> */
int luaopen_mathext(lua_State *L) {
    lua_createtable(L, 0, 5);          /* stack: [table] - 0 array slots, 5 hash slots */
    luaL_setfuncs(L, mathext_funcs, 0); /* register all functions into table on top */
                                        /* stack: [table] - functions added as fields */
    return 1;                           /* leave the table as the return value */
}



To make this module available from Lua via require("mathext"), register the
loader with luaL_requiref:



/* In your C initialisation code, after luaL_newstate() */
luaL_requiref(L, "mathext", luaopen_mathext, 1);
lua_pop(L, 1);   /* pop the module table */



After this, Lua code can call require("mathext") or use it directly if
you passed 1 as the third argument (which sets the global mathext).



The third argument to luaL_setfuncs is the number of "upvalues" - shared
values that all functions in the module can access. Upvalues implement closures for C
functions. Pass 0 unless you need shared state (covered in a later article on userdata).


Signalling Errors with luaL_error


When a C function detects a problem - wrong argument values, a failed operation - it
should raise a Lua error rather than returning a success/failure code. This is what
the luaL_check* functions do for type errors; for semantic errors you call
luaL_error yourself:



static int lua_clamp(lua_State *L) {
    lua_Number value = luaL_checknumber(L, 1);
    lua_Number lo    = luaL_checknumber(L, 2);
    lua_Number hi    = luaL_checknumber(L, 3);

    if (lo > hi) {
        /* luaL_error raises a Lua error - it never returns to the caller */
        return luaL_error(L, "clamp: lo (%g) must be <= hi (%g)", lo, hi);
    }

    lua_Number result = value < lo ? lo : (value > hi ? hi : value);
    lua_pushnumber(L, result);
    return 1;
}



luaL_error formats a message with printf-style format specifiers,
raises a Lua error, and never returns. The error is caught by whatever lua_pcall
is on the C call stack above. If there is none - which should not happen in correctly
written embedding code - it calls abort(). The %g format specifier
for lua_Number (a double) is the same as in standard C printf.



For errors that should include a Lua-level traceback, push an error object and call
lua_error(L) directly. luaL_error is the convenience wrapper
that constructs the message for you.


Complete Example: C Math Module


A complete C file implementing a mathext module with clamp,
lerp, and db_to_linear:



/* mathext.c - C math module for Lua 5.4 */
#include <lua.h>
#include <lualib.h>
#include <lauxlib.h>
#include <math.h>

/* clamp(value, lo, hi) -> number */
static int lua_clamp(lua_State *L) {
    lua_Number v  = luaL_checknumber(L, 1);
    lua_Number lo = luaL_checknumber(L, 2);
    lua_Number hi = luaL_checknumber(L, 3);
    if (lo > hi)
        return luaL_error(L, "clamp: lo (%g) > hi (%g)", lo, hi);
    lua_pushnumber(L, v < lo ? lo : (v > hi ? hi : v));
    return 1;
}

/* lerp(a, b, t) -> number  (linear interpolation, t in [0,1]) */
static int lua_lerp(lua_State *L) {
    lua_Number a = luaL_checknumber(L, 1);
    lua_Number b = luaL_checknumber(L, 2);
    lua_Number t = luaL_checknumber(L, 3);
    lua_pushnumber(L, a + (b - a) * t);
    return 1;
}

/* db_to_linear(db) -> number  (dB to linear amplitude, 0 dB = 1.0) */
static int lua_db_to_linear(lua_State *L) {
    lua_Number db = luaL_checknumber(L, 1);
    lua_pushnumber(L, pow(10.0, db / 20.0));
    return 1;
}

/* divmod(a, b) -> quotient, remainder */
static int lua_divmod(lua_State *L) {
    lua_Integer a = luaL_checkinteger(L, 1);
    lua_Integer b = luaL_checkinteger(L, 2);
    if (b == 0)
        return luaL_error(L, "divmod: division by zero");
    lua_pushinteger(L, a / b);
    lua_pushinteger(L, a % b);
    return 2;
}

/* Function table - NULL-terminated sentinel required */
static const luaL_Reg mathext_funcs[] = {
    { "clamp",        lua_clamp        },
    { "lerp",         lua_lerp         },
    { "db_to_linear", lua_db_to_linear },
    { "divmod",       lua_divmod       },
    { NULL, NULL }
};

/* luaopen_mathext: called when Lua does require("mathext") */
int luaopen_mathext(lua_State *L) {
    lua_createtable(L, 0, 4);           /* stack: [{}]          */
    luaL_setfuncs(L, mathext_funcs, 0); /* stack: [{functions}] */
    return 1;                           /* return the table      */
}



A host that embeds this module:



/* host.c */
#include <lua.h>
#include <lualib.h>
#include <lauxlib.h>
#include <stdio.h>

/* Declare the module loader from mathext.c */
extern int luaopen_mathext(lua_State *L);

int main(void) {
    lua_State *L = luaL_newstate();
    luaL_openlibs(L);

    /* Register mathext as a preloaded module */
    luaL_requiref(L, "mathext", luaopen_mathext, 1);
    lua_pop(L, 1);

    /* Run a test script */
    const char *script =
        "local m = require('mathext')\n"
        "print(m.clamp(1.5, 0, 1))        -- 1.0\n"
        "print(m.lerp(0, 100, 0.25))      -- 25.0\n"
        "print(m.db_to_linear(-6))        -- ~0.501\n"
        "local q, r = m.divmod(17, 5)\n"
        "print(q, r)                      -- 3   2\n";

    if (luaL_dostring(L, script) != LUA_OK) {
        fprintf(stderr, "error: %s\n", lua_tostring(L, -1));
        lua_pop(L, 1);
    }

    lua_close(L);
    return 0;
}



gcc -o host host.c mathext.c $(pkg-config --cflags --libs lua5.4) -lm
./host
# 1.0
# 25.0
# 0.50118723362727
# 3   2


Naming Conventions and Module Layout


A few conventions worth following when writing C modules for Lua:



Name the C functions with a consistent prefix that will not collide with anything in
your codebase: lua_clamp, not clamp. Declare them
static - they are implementation details of the module, not public C API.



Name the loader function luaopen_<modulename>. This is the convention
Lua's dynamic loader uses when loading .so/.dll files via require. If you
pre-register with luaL_requiref the name must match the first argument:



/* The name passed to luaL_requiref and the field added to package.loaded */
luaL_requiref(L, "mathext", luaopen_mathext, 1);
/*                 ^^^^^^^^ must match the module name Lua sees */



Keep the luaL_Reg array close to the loader function, not scattered through
the file. It is the index of your module's public API; a reader should be able to find
it immediately.


Summary


Every C function callable from Lua has the signature int f(lua_State *L).
Arguments arrive on the stack starting at index 1; luaL_check* reads them
with automatic type checking. Return values are pushed onto the stack; the C function
returns the count. luaL_error raises a Lua error and never returns.



For modules, define a luaL_Reg array terminated with { NULL, NULL },
create a table with lua_createtable, and register everything at once with
luaL_setfuncs. Use luaL_requiref to pre-register the module so
Lua code can find it with require.
