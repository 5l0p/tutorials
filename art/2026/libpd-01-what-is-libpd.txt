libpd - Part 1: What Is libpd?
2026-06-09

Pure Data (Pd) is a visual dataflow programming language designed for real-time audio and signal processing. libpd strips away the graphical interface and packages the DSP engine as a portable C library, letting you run Pd patches inside any application - games, plugins, servers, or embedded devices - without launching the Pd GUI. This article explains what libpd is, what it exposes through its C API, and where it fits in an audio software stack.

What Is Pure Data?

Pure Data was created by Miller Puckette in the 1990s as a real-time computer music environment. Programs in Pd are called patches: boxes called objects are wired together with patch cables that carry either audio signals (signal rate, processed in 64-sample blocks) or control messages (bang, float, symbol, list). The engine computes an audio graph each block, mixing DSP processing with dynamic message routing.

Pd handles oscillators, filters, envelopes, samplers, reverbs, granular synthesis, and arbitrary signal-processing algorithms - all defined in the patch, not compiled into the host. This separation is the key insight behind libpd: the patch is data, the engine is a library.

What libpd Provides

libpd (created by Peter Brinkmann, Hans-Christoph Steiner, and contributors) is a refactoring of the Pure Data source that compiles to a static or shared library. It provides:


The complete Pd DSP engine and object set (including the standard set of audio objects).
A C API for initialising the engine, loading patches, processing audio, and exchanging messages.
Optional C++ and Java/Android wrappers.
An "extra" collection of community objects enabled with a CMake flag.


What it does not provide is any GUI, windowing system, or audio device driver. Your application owns the audio I/O; libpd just fills buffers.

The Core C API Surface

The entire API lives in a handful of headers. The most important functions are:


#include <z_libpd.h>

/* Initialise the engine once per process */
void  libpd_init(void);

/* Set up audio: input channels, output channels, sample rate */
int   libpd_init_audio(int inChannels, int outChannels, int sampleRate);

/* Load a patch from disk; returns a handle (void*) */
void *libpd_openfile(const char *basename, const char *dirname);

/* Close a previously opened patch */
void  libpd_closefile(void *p);

/* Process one or more 64-sample ticks.
   inBuffer and outBuffer are interleaved float PCM.
   ticks = blockSize / 64  */
int   libpd_process_float(int ticks,
                          const float *inBuffer,
                          float       *outBuffer);

/* Send a bang, float, or symbol to a named receiver in the patch */
int   libpd_bang(const char *recv);
int   libpd_float(const char *recv, float x);
int   libpd_symbol(const char *recv, const char *sym);


These six categories - init, audio setup, patch loading, audio processing, message sending, and hooks for receiving messages back - cover most embedding use cases. Later articles in this series go deep on each area.

The 64-Sample Block Constraint

Pd processes audio in blocks of exactly 64 samples. This is hardwired into the engine. If your audio driver delivers blocks that are not a multiple of 64 (e.g., 128, 256, 512 - all fine; 100 - not fine), you must buffer and rebatch. The ticks parameter to libpd_process_float tells the engine how many 64-sample ticks to compute in one call:


int ticks = blockSize / 64;   /* blockSize must be a multiple of 64 */
libpd_process_float(ticks, inputBuffer, outputBuffer);


For a 256-sample block, ticks = 4. The engine runs the graph four times, advancing the signal by 64 samples each tick.

Multiple Instances

By default, libpd uses a single global instance. For applications that need multiple independent Pd instances (e.g., one per plugin slot), libpd provides an instance API:


t_pdinstance *inst = libpd_new_instance();
libpd_set_instance(inst);
/* All subsequent libpd calls operate on inst */
libpd_init_audio(0, 2, 44100);
void *patch = libpd_openfile("synth.pd", "/patches");


Instances are not thread-safe with each other by default; you need to serialise access or dedicate one thread per instance.

Use Cases


Games: A Pd patch generates adaptive music or procedural sound effects. The game engine calls libpd_float("speed", velocity) to drive the patch in real time.
VST/AU plugins: The entire DSP algorithm lives in a Pd patch. The plugin host calls the audio callback; the plugin calls libpd_process_float. Users can swap patches without recompiling.
Mobile apps: libpd compiles for iOS and Android. Projects like MobMuPlat and AudioKit have shipped apps this way.
Server-side audio: Render audio offline or in real time on a server, then stream it. No audio device needed (see Series 6, article 11).
Embedded devices: Run Pd patches on a Raspberry Pi or similar Linux board, driving audio hardware directly.


What Gets Embedded

When you link libpd into your application, you pull in the Pd scheduler, the signal chain, the message passing system, and all built-in objects. The compiled library is roughly 1–2 MB stripped. Pure Data patches (.pd files) are text files and remain external to the binary - you can ship them alongside your application or generate them at runtime.

The key architectural boundary: your code drives the clock. There is no internal thread or timer inside libpd. You call libpd_process_float from your audio callback, and libpd computes the next block synchronously before returning. This makes integration deterministic and predictable.

Minimal "Hello, Audio" Example


#include <z_libpd.h>
#include <string.h>

#define SAMPLE_RATE 44100
#define BLOCK_SIZE  512
#define TICKS       (BLOCK_SIZE / 64)

int main(void) {
    float out[BLOCK_SIZE * 2]; /* stereo */

    libpd_init();
    libpd_init_audio(0, 2, SAMPLE_RATE);

    void *patch = libpd_openfile("hello.pd", ".");
    if (!patch) { return 1; }

    /* Simulate a few blocks without a real audio device */
    for (int i = 0; i < 10; i++) {
        memset(out, 0, sizeof(out));
        libpd_process_float(TICKS, NULL, out);
        /* out[] now contains rendered stereo audio */
    }

    libpd_closefile(patch);
    return 0;
}


Real applications replace the loop with a platform audio callback (PortAudio, ALSA, SDL2, CoreAudio). The rest of the code is the same.

Summary


libpd packages the Pure Data DSP engine as a C library: your application calls
libpd_init, configures channel counts and sample rate, opens a .pd
patch file, and then drives audio by calling libpd_process_float from its
audio callback. The library computes 64 samples per tick synchronously and returns
immediately — there is no internal thread. Control messages travel in both directions:
host-to-patch via send functions, patch-to-host via registered hook callbacks.
