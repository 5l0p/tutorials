Audio Compression - Part 2: Differential PCM

mu-law companding, covered in the previous article, compresses each sample independently - it never looks at its neighbours. But adjacent audio samples are rarely independent: at a 44.1 kHz sample rate even a 5 kHz tone only moves roughly 1/9 of the way through a cycle between samples, so consecutive sample values are usually close together. Differential Pulse Code Modulation (DPCM) exploits exactly this: instead of storing each sample's absolute value, store the difference from the previous sample. Differences cluster near zero far more tightly than the original waveform does, so they need fewer bits to represent to the same perceptual accuracy - and unlike mu-law, DPCM can be made fully lossless if you want it to be.

The Idea: Predict, Then Encode the Error

The general DPCM architecture has three pieces:

  1. Predictor: guesses the next sample from ones already seen. The simplest possible predictor is "the next sample equals the last one" - a zeroth-order predictor.
  2. Difference: subtract the prediction from the actual sample. This residual is what gets encoded.
  3. Quantizer (optional): reduces the precision of the residual to save bits. Skip this step and DPCM is lossless; include it and DPCM becomes lossy but higher-compression.

The decoder mirrors this: it decodes the residual, adds back the same prediction the encoder used, and reconstructs the sample. The predictor at decode time must compute the identical prediction the encoder used, from data the decoder has already reconstructed - not from the original signal, which the decoder never sees.

Lossless Delta Coding

With a zeroth-order predictor and no quantizer, DPCM is just delta coding: store the difference between each sample and the last one. 

#include <stdint.h>
#include <stdio.h>

typedef struct {
    int16_t prev;
} DeltaEncoder;

void delta_encoder_init(DeltaEncoder *e) {
    e->prev = 0;
}

/* Returns the delta. Because both operands are int16_t, the true
   mathematical difference can overflow int16_t range (e.g. 32767 - (-32768)),
   so we compute in a wider type and let the caller decide how to store it. */
int32_t delta_encode(DeltaEncoder *e, int16_t sample) {
    int32_t delta = (int32_t)sample - (int32_t)e->prev;
    e->prev = sample;
    return delta;
}

typedef struct {
    int16_t prev;
} DeltaDecoder;

void delta_decoder_init(DeltaDecoder *d) {
    d->prev = 0;
}

int16_t delta_decode(DeltaDecoder *d, int32_t delta) {
    int16_t sample = (int16_t)((int32_t)d->prev + delta);
    d->prev = sample;
    return sample;
}

This round-trips exactly - no loss at all - but notice the delta is stored as a 32-bit (or at least 17-bit) value to cover the worst case, which means naive lossless delta coding alone saves nothing; it can even cost more space than the original signal in the worst case. The saving only shows up once you feed the deltas into an entropy coder (Huffman, Golomb-Rice, range coding) that assigns short codes to the small values that dominate real audio - lossless codecs like FLAC and Shorten are built exactly this way: predictor + entropy coder, no quantizer, bit-exact reconstruction. Building that entropy stage is outside the scope of this article, but knowing the split (prediction reduces the *entropy* of the data; the entropy coder is what actually shrinks the *bit count*) tells you where lossless compressors get their gain.

Lossy DPCM: Adding a Quantizer

To get a compression ratio without an entropy coder, quantize the delta down to fewer bits before storing it. A 4-bit quantizer represents the delta as one of 16 fixed step values, immediately giving 4:1 compression versus 16-bit linear PCM: 

#include <stdint.h>

#define DPCM_STEP 128  /* size of one quantization step, in linear PCM units */

typedef struct {
    int16_t predicted;  /* what the decoder will also have */
} DpcmEncoder;

void dpcm_encoder_init(DpcmEncoder *e) {
    e->predicted = 0;
}

/* Encodes one sample to a signed 4-bit code (range -8..7). */
int8_t dpcm_encode(DpcmEncoder *e, int16_t sample) {
    int32_t diff = (int32_t)sample - (int32_t)e->predicted;

    int32_t code = diff / DPCM_STEP;
    if (code > 7)  code = 7;
    if (code < -8) code = -8;

    /* Critical: reconstruct the same quantized value the decoder will
       produce, and predict from THAT - not from the original sample.
       This is what keeps encoder and decoder in sync. */
    int32_t quantized_diff = code * DPCM_STEP;
    e->predicted = (int16_t)((int32_t)e->predicted + quantized_diff);

    return (int8_t)code;
}

typedef struct {
    int16_t predicted;
} DpcmDecoder;

void dpcm_decoder_init(DpcmDecoder *d) {
    d->predicted = 0;
}

int16_t dpcm_decode(DpcmDecoder *d, int8_t code) {
    int32_t quantized_diff = (int32_t)code * DPCM_STEP;
    d->predicted = (int16_t)((int32_t)d->predicted + quantized_diff);
    return d->predicted;
}

The comment in the middle is the single most important line in this article to understand. It is tempting to write e->predicted = sample; - after all, the encoder knows the true sample value. But the decoder never sees the true sample, only the 4-bit code. If the encoder predicts from the exact original sample while the decoder predicts from the quantized reconstruction, the two predictors see different histories and the reconstruction error compounds every sample instead of staying bounded. This technique - having the encoder run its own local copy of the decoder to predict from what the decoder will actually have - is called a closed-loop or backward-adaptive predictor, and every production DPCM and ADPCM codec uses it for exactly this reason.

Round-Tripping and Measuring Error

#include <stdio.h>
#include <math.h>

int main(void) {
    int16_t samples[] = { 0, 500, 1000, 1200, 1100, 800, 300, -200, -600 };
    int n = sizeof(samples) / sizeof(samples[0]);

    DpcmEncoder enc; dpcm_encoder_init(&enc);
    DpcmDecoder dec; dpcm_decoder_init(&dec);

    for (int i = 0; i < n; i++) {
        int8_t code = dpcm_encode(&enc, samples[i]);
        int16_t recon = dpcm_decode(&dec, code);
        printf("orig=%5d  code=%3d  recon=%5d  error=%4d\n",
               samples[i], code, recon, recon - samples[i]);
    }
    return 0;
}

The Slope Overload Problem

Look at what happens when the signal changes quickly (sample 3 to sample 4: a jump of -100 that the quantizer, stepping in units of 128, cannot represent exactly) versus slowly. With a fixed step size you are stuck between two failure modes:

  • Step size too large: quiet, slowly-changing passages get coarse, unnecessarily large steps - audible as granular noise or "idle noise" even in near-silence.
  • Step size too small: loud, fast-changing passages exceed the maximum step the coder can take per sample, so the reconstruction lags behind the real signal and audibly distorts. This specific failure is called slope overload, and you can see the seeds of it above - the coder is one step behind the direction the signal is heading whenever the signal moves faster than DPCM_STEP per sample.

There is no single fixed step size that avoids both problems on real audio, because real audio's local dynamics vary constantly between quiet and loud passages. The fix is to let the step size adapt sample-by-sample based on recent signal behaviour - widen it when the signal is moving fast (avoiding slope overload), narrow it when the signal is calm (avoiding granular noise). That adaptive step size is exactly what turns DPCM into ADPCM, the subject of the next article.

Summary

DPCM exploits the fact that adjacent audio samples are usually close in value by encoding the difference from a prediction rather than the sample itself. Without quantization this is lossless and only pays off once paired with an entropy coder; with a fixed quantizer it gives a direct, guaranteed compression ratio (4:1 for a 4-bit code) at the cost of introducing quantization error. The one rule that must never be broken is that the encoder's predictor has to be driven by the same quantized reconstruction the decoder will produce, not by the true input - otherwise the two sides diverge. A fixed step size, though, forces a compromise between granular noise on quiet material and slope overload on loud, fast-changing material - which is the problem adaptive step sizes exist to solve.
