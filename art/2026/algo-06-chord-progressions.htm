Algorithmic Composition - Part 6: Generating Chord Progressions

The Markov chain in Part 4 generated melodies by transitioning between scale degrees. The same technique applies to harmony, but there is a level of abstraction that makes it more powerful: instead of transitioning between specific chords (I → IV → V → I), transition between harmonic functions (tonic → subdominant → dominant → tonic) and then choose which specific chord to play within each function.

This separation means the grammar of tension and resolution is independent of the specific chords used. A tonic function can be expressed as I, iii, or vi; a dominant can be V or vii°. The same function-level Markov chain generates progressions that vary in their specific chord choices while always following the logic of tonal motion.

The Function-Level Markov Chain

Three harmonic functions: T (tonic), S (subdominant), D (dominant). The transition matrix encodes which function is likely to follow which: 

-- Function indices: 1=Tonic, 2=Subdominant, 3=Dominant
-- Each row: weights for {T, S, D} as the next function.

local function_chain = {
    [1] = {1, 4, 4},   -- Tonic → usually departs to S or D (equal probability)
    [2] = {1, 1, 7},   -- Subdominant → strongly to Dominant
    [3] = {8, 0, 2},   -- Dominant → usually resolves to Tonic; sometimes stays
}

The chain encodes the asymmetry of tonal motion:

  • T → S or D: the tonic can depart in either direction with similar probability.
  • S → D: subdominant almost always moves to dominant. S → T is the "plagal cadence" (amen cadence) — valid but far less common, hence the low weight of 1.
  • D → T: the dominant's strong pull toward resolution. D → D extends tension; D → S is the "deceptive cadence" (V → vi), encoded here as D→T followed by choosing vi as the tonic chord.

Chord Choices Per Function

Map each function to the degrees that can express it. Use the chord_db table from Part 5, which includes both diatonic chords and borrowed chords: 

-- Chord database indices: 1=I, 2=ii, 3=iii, 4=IV, 5=V, 6=vi, 7=vii°,
--                         8=bVII (borrowed), 9=bVI (borrowed), 10=iv (borrowed)
local chord_db = {
    [1]  = {root_offset=0,  type="major",      function_="T"},
    [2]  = {root_offset=2,  type="minor",      function_="S"},
    [3]  = {root_offset=4,  type="minor",      function_="T"},
    [4]  = {root_offset=5,  type="major",      function_="S"},
    [5]  = {root_offset=7,  type="major",      function_="D"},
    [6]  = {root_offset=9,  type="minor",      function_="T"},
    [7]  = {root_offset=11, type="diminished", function_="D"},
    [8]  = {root_offset=10, type="major",      function_="D", borrowed=true},
    [9]  = {root_offset=8,  type="major",      function_="S", borrowed=true},
    [10] = {root_offset=5,  type="minor",      function_="S", borrowed=true},
}

-- Weighted list of chord indices per function
-- Higher weight = more likely to be chosen for that function
local function_chords = {
    T = { {id=1, w=6}, {id=3, w=2}, {id=6, w=4} },           -- I, iii, vi
    S = { {id=4, w=5}, {id=2, w=4}, {id=9, w=2}, {id=10, w=1} }, -- IV, ii, bVI, iv
    D = { {id=5, w=7}, {id=7, w=2}, {id=8,  w=2} },           -- V, vii°, bVII
}

local function_names = {[1]="T", [2]="S", [3]="D"}

The borrowed chords (bVI, iv, bVII) have lower weights than their diatonic equivalents, so they appear occasionally for colour rather than dominating the output.

Generating a Progression

local pitch = require("pitch")

local triad_offsets = {
    major      = {0, 4, 7},
    minor      = {0, 3, 7},
    diminished = {0, 3, 6},
    augmented  = {0, 4, 8},
}

local function sample(items)
    local total = 0
    for _, item in ipairs(items) do total = total + item.w end
    local r = math.random() * total
    local cum = 0
    for _, item in ipairs(items) do
        cum = cum + item.w
        if r <= cum then return item.id end
    end
    return items[#items].id
end

local function sample_weights(weights)
    local total = 0
    for _, w in ipairs(weights) do total = total + w end
    local r = math.random() * total
    local cum = 0
    for i, w in ipairs(weights) do
        cum = cum + w
        if r <= cum then return i end
    end
    return #weights
end

-- Build MIDI pitches for one chord
local function build_chord(chord_id, root)
    local entry = chord_db[chord_id]
    local chord_root = root + entry.root_offset
    local offsets = triad_offsets[entry.type]
    return {chord_root + offsets[1], chord_root + offsets[2], chord_root + offsets[3]}
end

-- Generate n chords, starting on tonic function
local function generate_progression(n, root)
    local func_state = 1   -- start on Tonic
    local progression = {}

    for i = 1, n do
        local func_name  = function_names[func_state]
        local chord_id   = sample(function_chords[func_name])
        local chord_midi = build_chord(chord_id, root)
        progression[i]   = {func=func_name, id=chord_id, notes=chord_midi}
        func_state = sample_weights(function_chain[func_state])
    end

    return progression
end

-- Example: 8-chord progression in C major, root C4 = 60
math.randomseed(7)
local prog = generate_progression(8, 60)

local chord_names = {"I","ii","iii","IV","V","vi","vii°","bVII","bVI","iv"}
for i, chord in ipairs(prog) do
    print(string.format("[%s] %s: %s %s %s",
        chord.func, chord_names[chord.id],
        pitch.name(chord.notes[1]),
        pitch.name(chord.notes[2]),
        pitch.name(chord.notes[3])))
end

Sample output: 

[T] I:    C4 E4 G4
[D] V:    G4 B4 D5
[T] vi:   A4 C5 E5
[S] IV:   F4 A4 C5
[D] V:    G4 B4 D5
[T] I:    C4 E4 G4
[S] ii:   D4 F4 A4
[D] bVII: Bb4 D5 F5

The function logic is correct throughout: every S is followed by D, every D resolves to T. The bVII at position 8 is a borrowed chord that creates unexpected colour while still performing a dominant function.

Locking Progressions to a Period

A 4-bar progression should start and end on a tonic chord to sound complete. The simplest approach: generate the progression until the chain reaches tonic, then stop if you have at least 3 chords; repeat if the chain stays on tonic too long or never returns.

A more reliable approach: generate a fixed 4-chord block, force the first chord to tonic function, force the last chord to tonic function (V→I cadence), and allow the middle two to vary: 

local function four_bar_progression(root)
    -- chord 1: forced tonic
    local c1 = build_chord(sample(function_chords.T), root)
    -- chord 2: S or D
    local f2  = ({1,2,3,2,3,3,1})[math.random(7)]  -- bias toward S
    local fn2 = function_names[f2] == "T" and "S" or function_names[f2]
    fn2 = (math.random(2) == 1) and "S" or "D"
    local c2  = build_chord(sample(function_chords[fn2]), root)
    -- chord 3: D (pre-cadential dominant)
    local c3  = build_chord(sample(function_chords.D), root)
    -- chord 4: forced tonic (resolution)
    local c4  = build_chord(1, root)  -- always I for clarity

    return {
        {notes=c1, func="T"},
        {notes=c2, func=fn2},
        {notes=c3, func="D"},
        {notes=c4, func="T"},
    }
end

Sending Chords to Pure Data

Send each chord as a three-float list to a Pd receiver. The Pd patch routes each float to one voice in the Monomachine SuperWave engine: 

-- In Lua:
local function send_chord(notes)
    -- Pd [list] message: "chord pitch1 pitch2 pitch3"
    pd.send("algo-chord", notes[1], notes[2], notes[3])
end

-- In Pure Data:
-- [receive algo-chord]
-- |
-- [unpack f f f]
-- |  |  |
-- [mtof] [mtof] [mtof]
-- voice1 voice2 voice3

Summary

Operating on harmonic function (T, S, D) rather than specific chord degrees produces progressions that follow tonal logic while varying in their specific chord choices. The function-level Markov chain encodes asymmetric motion: S almost always goes to D, D almost always resolves to T. Within each function, a weighted list chooses which chord to play — diatonic chords with high weight, borrowed chords with low weight. The result is a 4-bar progression generator that sounds coherent in any key by changing only the root parameter. Chord tones go to Pure Data as list messages, one float per voice.
