openFrameworks - Part 21: SDL2 as a Lighter Alternative

openFrameworks is powerful but not lightweight. It pulls in FreeImage, FreeType, GLFW, Boost filesystem, RTAudio, and a pile of its own abstractions. For a command-line tool that just needs a preview window, or a small utility with a fixed GUI, that weight is a liability. SDL2 is the opposite: a single, stable C library that gives you a window, an OpenGL or software renderer, keyboard/mouse input, and audio, with a total size measured in hundreds of kilobytes. This article shows when SDL2 is the right choice, how it covers the same ground as oF's core, and how to connect SDL2's audio callback to the DSP oscillator and filter code from Series 4.

What You Will Learn

  • How to create an SDL_Window and an OpenGL context or SDL renderer.
  • The difference between SDL_Surface (CPU) and SDL_Texture (GPU) in the 2D renderer.
  • How to handle the SDL2 event loop for keyboard and mouse input.
  • How to use SDL_ttf for text rendering without FreeType setup.
  • How to connect SDL2's audio callback to the Series 4 DSP code.
  • When to choose SDL2 over openFrameworks.

Installing SDL2

# Debian/Ubuntu:
sudo apt install libsdl2-dev libsdl2-ttf-dev

# macOS (Homebrew):
brew install sdl2 sdl2_ttf

# Windows (vcpkg):
vcpkg install sdl2 sdl2-ttf

Compile with: 

g++ -std=c++17 main.cpp $(sdl2-config --cflags --libs) -lSDL2_ttf -o myapp

SDL_Window and the Event Loop

The SDL2 lifecycle is: SDL_Init, create window and renderer (or GL context), enter event loop, clean up. No base class, no setup/update/draw virtuals - just a while loop you control. 

#include <SDL2/SDL.h>
#include <cstdio>

int main(int argc, char* argv[]) {
    if (SDL_Init(SDL_INIT_VIDEO | SDL_INIT_AUDIO) != 0) {
        SDL_Log("SDL_Init failed: %s", SDL_GetError());
        return 1;
    }

    SDL_Window *window = SDL_CreateWindow(
        "SDL2 Demo",
        SDL_WINDOWPOS_CENTERED, SDL_WINDOWPOS_CENTERED,
        800, 600,
        SDL_WINDOW_SHOWN | SDL_WINDOW_RESIZABLE
    );

    SDL_Renderer *renderer = SDL_CreateRenderer(
        window, -1,
        SDL_RENDERER_ACCELERATED | SDL_RENDERER_PRESENTVSYNC
    );

    bool running = true;
    while (running) {
        // --- Event phase ---
        SDL_Event e;
        while (SDL_PollEvent(&e)) {
            if (e.type == SDL_QUIT) running = false;
            if (e.type == SDL_KEYDOWN && e.key.keysym.sym == SDLK_ESCAPE) running = false;
        }

        // --- Draw phase ---
        SDL_SetRenderDrawColor(renderer, 20, 20, 30, 255);
        SDL_RenderClear(renderer);

        // Draw a filled rectangle
        SDL_SetRenderDrawColor(renderer, 200, 100, 50, 255);
        SDL_Rect rect = {300, 200, 200, 150};
        SDL_RenderFillRect(renderer, &rect);

        SDL_RenderPresent(renderer);
    }

    SDL_DestroyRenderer(renderer);
    SDL_DestroyWindow(window);
    SDL_Quit();
    return 0;
}

SDL_Renderer vs OpenGL Context

SDL2 provides two rendering paths:

SDL_Renderer is a hardware-accelerated 2D API: rectangles, lines, points, and textures. It targets the GPU via Metal, Direct3D, or OpenGL under the hood but exposes none of that complexity. It is the right choice for 2D tools, visualisers, and simple games.

OpenGL context: request SDL_WINDOW_OPENGL on the window and call SDL_GL_CreateContext(). SDL2 handles the platform-specific context creation; after that you use raw OpenGL or a loader like GLEW/GLAD. This is the path for 3D work, shaders, and FBOs. 

// OpenGL path
SDL_GL_SetAttribute(SDL_GL_CONTEXT_MAJOR_VERSION, 3);
SDL_GL_SetAttribute(SDL_GL_CONTEXT_MINOR_VERSION, 3);
SDL_GL_SetAttribute(SDL_GL_CONTEXT_PROFILE_MASK, SDL_GL_CONTEXT_PROFILE_CORE);

SDL_Window *window = SDL_CreateWindow("GL Window", SDL_WINDOWPOS_CENTERED,
    SDL_WINDOWPOS_CENTERED, 800, 600, SDL_WINDOW_OPENGL | SDL_WINDOW_SHOWN);

SDL_GLContext ctx = SDL_GL_CreateContext(window);
SDL_GL_SetSwapInterval(1);   // vsync

// ... OpenGL calls here ...

SDL_GL_SwapWindow(window);   // present frame
SDL_GL_DeleteContext(ctx);

SDL_Surface vs SDL_Texture

An SDL_Surface is a pixel buffer in CPU memory. Operations on it (blitting, pixel access) run on the CPU. A SDL_Texture is a GPU-side image; it can only be drawn by an SDL_Renderer, not accessed pixel-by-pixel directly.

The typical flow for a dynamically updated visualisation: 

// Allocate a texture for CPU-updated content
SDL_Texture *waveformTex = SDL_CreateTexture(
    renderer,
    SDL_PIXELFORMAT_RGBA8888,
    SDL_TEXTUREACCESS_STREAMING,   // updated each frame
    512, 128
);

// Each frame: lock, write pixels, unlock, render
void* pixels;
int pitch;
SDL_LockTexture(waveformTex, nullptr, &pixels, &pitch);
    uint32_t *px = static_cast<uint32_t*>(pixels);
    for (int x = 0; x < 512; x++) {
        // Draw a waveform: white pixel at y = 64 + sample * 64
        int y = 64 + (int)(waveformData[x] * 60.0f);
        y = std::max(0, std::min(127, y));
        for (int row = 0; row < 128; row++) {
            px[row * (pitch / 4) + x] = (row == y) ? 0xFFFFFFFF : 0x00000000;
        }
    }
SDL_UnlockTexture(waveformTex);

SDL_Rect dst = {20, 400, 512, 128};
SDL_RenderCopy(renderer, waveformTex, nullptr, &dst);

Keyboard and Mouse Input

// In the event loop:
SDL_Event e;
while (SDL_PollEvent(&e)) {
    switch (e.type) {
    case SDL_QUIT:
        running = false;
        break;
    case SDL_KEYDOWN:
        if (e.key.keysym.sym == SDLK_SPACE) togglePlay();
        if (e.key.keysym.sym == SDLK_UP)    freq += 10.0f;
        if (e.key.keysym.sym == SDLK_DOWN)  freq -= 10.0f;
        break;
    case SDL_MOUSEBUTTONDOWN:
        if (e.button.button == SDL_BUTTON_LEFT) {
            int x = e.button.x;
            int y = e.button.y;
            // handle click at (x, y)
        }
        break;
    case SDL_MOUSEMOTION:
        if (e.motion.state & SDL_BUTTON_LMASK) {
            // left-drag
        }
        break;
    case SDL_MOUSEWHEEL:
        freq += e.wheel.y * 5.0f;
        break;
    }
}

SDL_ttf for Text

SDL2 itself does not render text; that requires SDL_ttf. It wraps FreeType and produces SDL_Surface glyphs you upload to a texture. 

#include <SDL2/SDL_ttf.h>

TTF_Init();
TTF_Font *font = TTF_OpenFont(ofToDataPath("fonts/Mono.ttf").c_str(), 16);

// Each frame (or cache the texture if the string is static):
SDL_Color white = {255, 255, 255, 255};
SDL_Surface *surf = TTF_RenderUTF8_Blended(font, "Hello, World", white);
SDL_Texture *tex  = SDL_CreateTextureFromSurface(renderer, surf);
SDL_FreeSurface(surf);

SDL_Rect dst;
SDL_QueryTexture(tex, nullptr, nullptr, &dst.w, &dst.h);
dst.x = 20;
dst.y = 20;
SDL_RenderCopy(renderer, tex, nullptr, &dst);
SDL_DestroyTexture(tex);   // destroy each frame, or cache by string

TTF_CloseFont(font);
TTF_Quit();

For high-frequency text updates, cache the texture and only recreate it when the string changes. Creating a texture from a surface on every frame is expensive.

Connecting SDL2 Audio to Series 4 DSP Code

SDL2's audio API uses a callback model identical to oF's ofSoundStream. The Series 4 DSP objects (oscillators, biquad filters, compressors) are plain C++ classes with no oF dependency - they work unchanged inside an SDL2 audio callback. 

#include <SDL2/SDL.h>
// Series 4 DSP headers - no oF dependency required
#include "Oscillator.h"   // from dsp-03 / dsp-04
#include "BiquadFilter.h" // from dsp-05

Oscillator osc;
BiquadFilter lpf;

void audioCallback(void* userdata, Uint8* stream, int len) {
    float* out    = reinterpret_cast<float*>(stream);
    int numFrames = len / (2 * sizeof(float));  // stereo, float32

    for (int i = 0; i < numFrames; i++) {
        float s = osc.tick();          // generate oscillator sample
        s = lpf.process(s);            // run through low-pass filter
        out[i * 2]     = s;            // left
        out[i * 2 + 1] = s;            // right
    }
}

int main(int argc, char* argv[]) {
    SDL_Init(SDL_INIT_AUDIO | SDL_INIT_VIDEO);

    osc.setSampleRate(44100);
    osc.setFrequency(220.0f);
    lpf.setLowPass(44100, 800.0f, 0.707f);

    SDL_AudioSpec want, have;
    SDL_zero(want);
    want.freq     = 44100;
    want.format   = AUDIO_F32;
    want.channels = 2;
    want.samples  = 512;
    want.callback = audioCallback;

    SDL_AudioDeviceID dev = SDL_OpenAudioDevice(nullptr, 0, &want, &have, 0);
    SDL_PauseAudioDevice(dev, 0);   // start playing

    // ... event loop ...

    SDL_CloseAudioDevice(dev);
    SDL_Quit();
    return 0;
}

The callback runs on a dedicated audio thread. Use SDL_LockAudioDevice(dev) / SDL_UnlockAudioDevice(dev) around parameter changes from the main thread, or use lock-free atomics as described in the Series 4 articles.

When to Choose SDL2 over openFrameworks

SDL2 is the better choice when:

  • You need a minimal footprint - a small utility, a CI test harness, or an embedded target where oF's dependencies are impractical.
  • You want explicit control over the event loop and rendering pipeline without a framework's opinions.
  • You are writing portable code that will also be used by projects not using oF (the DSP code from Series 4 is a good example).
  • You only need 2D hardware-accelerated drawing plus text and input - SDL_Renderer covers this with less complexity than oF.

openFrameworks is the better choice when:

  • You need ofMesh, ofEasyCam, ofLight, or the shader pipeline.
  • You want the addon ecosystem (ofxCV, ofxOSC, ofxGui, etc.).
  • You are doing live-coding or prototyping where oF's integrated utilities save time.
  • You need cross-platform audio routing beyond a simple callback (JACK, per-device selection).

Summary

SDL2 covers window creation, hardware-accelerated 2D rendering, input, and audio in a single small C library with minimal dependencies. SDL_Renderer is the 2D path; SDL_GL_CreateContext gives you a raw OpenGL context for 3D work. The audio callback matches the pattern from oF's ofSoundStream, so Series 4 DSP objects plug in without modification. Choose SDL2 when you need a lightweight, low-dependency tool; choose oF when you need the full 3D pipeline, shaders, and the addon ecosystem.
