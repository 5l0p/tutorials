Containers - Part 15: cgroups v1
2026-08-13


Namespaces restrict what a process can see. They say nothing about what it can
consume. A container in its own PID namespace can still fork ten thousand
processes, allocate every byte of RAM on the host, and saturate every disk and
CPU. cgroups - control groups - are the kernel mechanism that enforces hard
resource limits. They have been in the mainline kernel since 2.6.24 (2008),
and every container runtime relies on them. This article covers cgroups v1: how
the virtual filesystem at /sys/fs/cgroup/ works, the most important
resource controllers, how to attach a process from C and from Go, and how to
watch a memory-limited process be killed by the kernel's out-of-memory handler.


What You Will Learn


How the cgroupfs virtual filesystem exposes cgroup control to userspace
The purpose of the most important v1 controllers: memory, cpu, pids, blkio
How to create a cgroup, set limits, and attach a process in shell
How to do the same from C using only filesystem operations
How to do the same from Go
How to trigger and observe an OOM kill


The cgroupfs Model


cgroups v1 exposes its entire API through a virtual filesystem. There is no
special syscall for "create a cgroup" or "set a memory limit". Instead, the
kernel mounts a pseudo-filesystem of type cgroup - one mount per controller
- under /sys/fs/cgroup/. Each mount supports a different resource type.



mount | grep cgroup
# tmpfs on /sys/fs/cgroup type tmpfs ...
# cgroup on /sys/fs/cgroup/memory type cgroup (rw,...,memory)
# cgroup on /sys/fs/cgroup/cpu,cpuacct type cgroup (rw,...,cpu,cpuacct)
# cgroup on /sys/fs/cgroup/pids type cgroup (rw,...,pids)
# cgroup on /sys/fs/cgroup/blkio type cgroup (rw,...,blkio)
# ... (8–12 mounts total on a typical system)



Each mounted hierarchy is a tree of directories. The root of the tree
represents all processes on the system. Creating a subdirectory creates a new
cgroup. The kernel automatically populates that directory with interface files.
Moving a PID into cgroup.procs attaches the process to that cgroup. Deleting
the directory (when it has no processes and no children) destroys the cgroup.



ls /sys/fs/cgroup/memory/
# cgroup.clone_children  memory.failcnt         memory.memsw.limit_in_bytes
# cgroup.event_control   memory.force_empty     memory.memsw.max_usage_in_bytes
# cgroup.procs           memory.kmem.limit_in_bytes  ...
# memory.limit_in_bytes  memory.max_usage_in_bytes   tasks


The Memory Controller


The memory controller is the most important for containers. Its key files are:



memory.limit_in_bytes - write the maximum bytes of user-space memory. The kernel OOM-kills the cgroup's processes when this is exceeded (by default).
memory.memsw.limit_in_bytes - limit on memory + swap combined. Set equal to memory.limit_in_bytes to disable swap for this cgroup.
memory.max_usage_in_bytes - read-only high-water mark since the cgroup was created.
memory.usage_in_bytes - current memory usage in bytes.
memory.failcnt - number of times the memory limit was hit.
memory.oom_control - write oom_kill_disable 1 to pause rather than kill on OOM. Useful for debugging; not for production.



# Create a cgroup limited to 64 MB
sudo mkdir /sys/fs/cgroup/memory/demo
sudo bash -c 'echo 67108864 > /sys/fs/cgroup/memory/demo/memory.limit_in_bytes'
# Disable swap (optional but common for containers)
sudo bash -c 'echo 67108864 > /sys/fs/cgroup/memory/demo/memory.memsw.limit_in_bytes'

# Attach the current shell
sudo bash -c 'echo $$ > /sys/fs/cgroup/memory/demo/cgroup.procs'

# Any child processes are also in the cgroup
cat /sys/fs/cgroup/memory/demo/memory.usage_in_bytes


The CPU Controller


The cpu controller uses a Completely Fair Scheduler (CFS) bandwidth system.
Two files matter:



cpu.cfs_period_us - the accounting period in microseconds. Default 100000 (100 ms).
cpu.cfs_quota_us - the maximum CPU time the cgroup may use per period. A value of 50000 with a period of 100000 means 50% of one CPU core. A value of 200000 means 200% - two full cores. The value -1 (default) means unlimited.
cpu.shares - a relative weight used when CPU is contended. Default 1024. A cgroup with 512 shares gets half as much CPU as one with 1024 when both are competing.



# Limit to 50% of one CPU core
sudo mkdir /sys/fs/cgroup/cpu/demo
sudo bash -c 'echo 50000 > /sys/fs/cgroup/cpu/demo/cpu.cfs_quota_us'

# Attach a CPU-intensive process
stress --cpu 1 &
sudo bash -c "echo $! > /sys/fs/cgroup/cpu/demo/cgroup.procs"

# Observe with top - the stress process should show ~50% CPU


The pids Controller


The pids controller limits the number of processes (and threads) that can exist
in a cgroup at once. This prevents fork bombs.



pids.max - write the maximum number of tasks. Write max to remove the limit.
pids.current - read-only current count of tasks in the cgroup and its descendants.



sudo mkdir /sys/fs/cgroup/pids/demo
sudo bash -c 'echo 10 > /sys/fs/cgroup/pids/demo/pids.max'
sudo bash -c 'echo $$ > /sys/fs/cgroup/pids/demo/cgroup.procs'

# Attempting to fork more than 10 processes total will fail with EAGAIN
# The kernel returns -EAGAIN from clone() when the limit is reached


Attaching a Process from C


Because the entire API is filesystem-based, there is no special syscall needed.
Creating a cgroup is mkdir(); writing a limit is write(); attaching is
writing a PID to cgroup.procs.



#define _GNU_SOURCE
#include <fcntl.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/stat.h>
#include <sys/types.h>
#include <unistd.h>

#define MEM_LIMIT   (64 * 1024 * 1024)   /* 64 MB */
#define PIDS_LIMIT  20

static int write_file(const char *path, const char *val) {
    int fd = open(path, O_WRONLY);
    if (fd == -1) { perror(path); return -1; }
    int r = write(fd, val, strlen(val));
    close(fd);
    return r == -1 ? -1 : 0;
}

static int write_pid(const char *cgroup_procs, pid_t pid) {
    char buf[32];
    snprintf(buf, sizeof(buf), "%d", pid);
    return write_file(cgroup_procs, buf);
}

int setup_cgroups(pid_t pid) {
    /* --- memory --- */
    mkdir("/sys/fs/cgroup/memory/mycontainer", 0755);

    char limit_str[32];
    snprintf(limit_str, sizeof(limit_str), "%d", MEM_LIMIT);
    write_file("/sys/fs/cgroup/memory/mycontainer/memory.limit_in_bytes",
               limit_str);
    write_file("/sys/fs/cgroup/memory/mycontainer/memory.memsw.limit_in_bytes",
               limit_str);
    write_pid("/sys/fs/cgroup/memory/mycontainer/cgroup.procs", pid);

    /* --- pids --- */
    mkdir("/sys/fs/cgroup/pids/mycontainer", 0755);

    char pids_str[32];
    snprintf(pids_str, sizeof(pids_str), "%d", PIDS_LIMIT);
    write_file("/sys/fs/cgroup/pids/mycontainer/pids.max", pids_str);
    write_pid("/sys/fs/cgroup/pids/mycontainer/cgroup.procs", pid);

    return 0;
}

int main(void) {
    pid_t pid = fork();
    if (pid == 0) {
        /* Child: pause until parent sets up cgroups */
        pause();
        _exit(0);
    }

    printf("setting up cgroups for PID %d\n", pid);
    if (setup_cgroups(pid) != 0) {
        fprintf(stderr, "cgroup setup failed\n");
        kill(pid, SIGKILL);
        return 1;
    }

    printf("cgroups configured. child is constrained.\n");

    /* Let the child run */
    kill(pid, SIGCONT);
    waitpid(pid, NULL, 0);

    /* Cleanup: rmdir after the cgroup is empty */
    rmdir("/sys/fs/cgroup/memory/mycontainer");
    rmdir("/sys/fs/cgroup/pids/mycontainer");
    return 0;
}



Compile and run as root: gcc -o cgdemo cgdemo.c && sudo ./cgdemo.


Attaching a Process from Go


package main

import (
    "fmt"
    "log"
    "os"
    "path/filepath"
    "strconv"
)

const (
    cgMemBase  = "/sys/fs/cgroup/memory/gocontainer"
    cgPidsBase = "/sys/fs/cgroup/pids/gocontainer"
    memLimit   = 64 * 1024 * 1024 // 64 MB
    pidsLimit  = 20
)

func writeFile(path, value string) error {
    return os.WriteFile(path, []byte(value), 0644)
}

func setupCgroup(pid int) error {
    // Memory cgroup
    if err := os.MkdirAll(cgMemBase, 0755); err != nil {
        return fmt.Errorf("mkdir memory cgroup: %w", err)
    }
    limitStr := strconv.Itoa(memLimit)
    if err := writeFile(filepath.Join(cgMemBase, "memory.limit_in_bytes"), limitStr); err != nil {
        return err
    }
    if err := writeFile(filepath.Join(cgMemBase, "memory.memsw.limit_in_bytes"), limitStr); err != nil {
        return err
    }
    if err := writeFile(filepath.Join(cgMemBase, "cgroup.procs"), strconv.Itoa(pid)); err != nil {
        return err
    }

    // Pids cgroup
    if err := os.MkdirAll(cgPidsBase, 0755); err != nil {
        return fmt.Errorf("mkdir pids cgroup: %w", err)
    }
    if err := writeFile(filepath.Join(cgPidsBase, "pids.max"), strconv.Itoa(pidsLimit)); err != nil {
        return err
    }
    if err := writeFile(filepath.Join(cgPidsBase, "cgroup.procs"), strconv.Itoa(pid)); err != nil {
        return err
    }

    return nil
}

func teardown() {
    os.Remove(cgMemBase)
    os.Remove(cgPidsBase)
}

func main() {
    if os.Getuid() != 0 {
        log.Fatal("must run as root")
    }
    defer teardown()

    pid := os.Getpid()
    fmt.Printf("constraining PID %d to 64 MB and 20 processes\n", pid)

    if err := setupCgroup(pid); err != nil {
        log.Fatal(err)
    }

    // Read back current usage
    usage, _ := os.ReadFile(filepath.Join(cgMemBase, "memory.usage_in_bytes"))
    fmt.Printf("current memory usage: %s", usage)

    fmt.Println("cgroup limits applied successfully")
}


Triggering an OOM Kill


The most visceral demonstration of the memory controller is watching it kill a
process that tries to allocate beyond its limit. The kernel OOM handler sends
SIGKILL to the task that triggered the limit.



# Create a cgroup limited to 32 MB
sudo mkdir /sys/fs/cgroup/memory/oom-demo
sudo bash -c 'echo 33554432 > /sys/fs/cgroup/memory/oom-demo/memory.limit_in_bytes'
sudo bash -c 'echo 33554432 > /sys/fs/cgroup/memory/oom-demo/memory.memsw.limit_in_bytes'

# Run a Python one-liner that allocates 100 MB inside the cgroup
sudo cgexec -g memory:oom-demo \
    python3 -c "x = bytearray(100 * 1024 * 1024); print('allocated')"

# Output: Killed
# Exit code: 137 (128 + SIGKILL)

# Confirm the OOM event was recorded
cat /sys/fs/cgroup/memory/oom-demo/memory.failcnt
# e.g. 1

# The kernel also logs it
dmesg | tail -5
# [12345.678] oom_reaper: reaped process NNNN (python3), now anon-rss:0kB ...



cgexec (from the cgroup-tools package) is a convenience wrapper that
creates a process attached to a named cgroup. The kernel's OOM log in
dmesg shows which process was killed, how much memory it held, and what the
cgroup's limit was.


Reading Usage


# Memory usage and high watermark
cat /sys/fs/cgroup/memory/mycontainer/memory.usage_in_bytes
cat /sys/fs/cgroup/memory/mycontainer/memory.max_usage_in_bytes

# Detailed per-category breakdown
cat /sys/fs/cgroup/memory/mycontainer/memory.stat

# How many times the limit was hit
cat /sys/fs/cgroup/memory/mycontainer/memory.failcnt

# CPU accounting (if cpu,cpuacct controller is mounted)
cat /sys/fs/cgroup/cpu,cpuacct/mycontainer/cpuacct.usage
# nanoseconds of CPU time used

# Current pids count
cat /sys/fs/cgroup/pids/mycontainer/pids.current


Cleaning Up Cgroups


A cgroup directory cannot be removed while it contains processes or has child
cgroups. The standard cleanup sequence is:



Wait for all processes in the cgroup to exit (or move them to the root cgroup by writing their PIDs to /sys/fs/cgroup/memory/cgroup.procs).
Recursively remove child cgroups bottom-up.
Remove the parent cgroup directory with rmdir().



# Move any remaining processes to root cgroup
for pid in $(cat /sys/fs/cgroup/memory/mycontainer/cgroup.procs); do
    echo $pid | sudo tee /sys/fs/cgroup/memory/cgroup.procs
done

# Remove
sudo rmdir /sys/fs/cgroup/memory/mycontainer
sudo rmdir /sys/fs/cgroup/pids/mycontainer


Summary


cgroups v1 exposes resource control through a virtual filesystem. Each
controller type is mounted separately under /sys/fs/cgroup/. Creating a
cgroup is mkdir; setting a limit is write; attaching a process is
writing its PID to cgroup.procs. The memory controller enforces hard memory
limits and OOM-kills processes that exceed them. The cpu controller throttles
CPU using CFS bandwidth controls. The pids controller prevents fork bombs.
All of this works identically from shell, C, and Go because it is all just
filesystem I/O.



The next article covers cgroups v2: the unified hierarchy that consolidates all
those separate mounts into a single tree, with a cleaner interface and better
delegation semantics.
