Go - Part 15: Code Generation with go generate
2026-07-18


Every large codebase contains repetitive code that cannot reasonably be abstracted away:
string representations for enumerations, marshalling glue, mock implementations, lookup
tables. In C, the traditional answer is macros - powerful but unreadable, undebuggable,
and invisible to the type system. In C# the modern answer is source generators, which are
Roslyn-based plug-ins that run during compilation. Go's answer is different from both:
go generate runs external programs as ordinary build steps, and the generated
code is checked in alongside hand-written code. The approach is less magic but more
transparent - you can read, debug, and diff the generated output just like any other Go file.


What You Will Learn


How the //go:generate directive works and when to use it.
Using stringer to generate String() methods for enum-like types.
Writing a code generator with text/template.
Inspecting Go source code with go/ast and go/parser.
How Go code generation compares to C macros and C# source generators.
Best practices: where to put generated files, how to mark them, and when to regenerate.


The //go:generate Directive


A //go:generate comment placed anywhere in a Go source file records a command
to run when go generate is invoked for that package. The comment must start
at column zero with no space between // and go:generate:



//go:generate stringer -type=Direction



Running go generate ./... finds every such comment in the package tree and
executes the recorded command with the shell. The working directory is set to the directory
containing the source file. The command receives the following environment variables:



$GOFILE   - the source file containing the directive
$GOPACKAGE - the package name
$GOARCH   - target architecture
$GOOS     - target OS
$GOROOT   - Go root directory
$GOPATH   - GOPATH



This is the complete mechanism. There is no phase integration with the compiler, no plugin
API - just "run this command". The simplicity is intentional. Compare this with C macros,
which are expanded by the preprocessor during compilation, invisible to debuggers and most
tooling. Compare with C# source generators, which are Roslyn extensions that run inside the
compiler pipeline and can only emit, not replace, source. Go generators can do anything an
external program can do.



The trade-off: unlike C# source generators, Go generators must be re-run manually (or in
your Makefile/CI pipeline). The output is checked in. This means a stale generated file
will compile but may be wrong. The convention is to add a CI check:



go generate ./...
git diff --exit-code   # fail if generate changed anything


stringer: String Methods for Enumerations


Go has no built-in enum keyword. The idiomatic substitute is a named integer
type and a block of const values using iota. The weakness is that
printing one of these constants shows the underlying integer, not a name. The
stringer tool generates a String() string method to fix that.



// direction.go
package main

//go:generate stringer -type=Direction

type Direction int

const (
    North Direction = iota
    South
    East
    West
)



Install and run:



go install golang.org/x/tools/cmd/stringer@latest
go generate .



This produces direction_string.go (the name follows the -type
argument). Open it and you will find a generated String() method that uses a
baked-in string containing all names concatenated and an index array - an efficient
O(1) lookup with no heap allocation:



// Code generated by "stringer -type=Direction"; DO NOT EDIT.

package main

import "strconv"

func _() {
    // Compile-time guard: fails if the value of North changes
    var x [1]struct{}
    _ = x[North-0]
    _ = x[South-1]
    _ = x[East-2]
    _ = x[West-3]
}

const _Direction_name = "NorthSouthEastWest"

var _Direction_index = [...]uint8{0, 5, 10, 14, 18}

func (i Direction) String() string {
    if i < 0 || i >= Direction(len(_Direction_index)-1) {
        return "Direction(" + strconv.FormatInt(int64(i), 10) + ")"
    }
    return _Direction_name[_Direction_index[i]:_Direction_index[i+1]]
}



The compile-time guard at the top is a key technique: if you add a new constant and forget
to regenerate, the blank identifier assignment will fail to compile because the constant
value will not match the expected index. This is how Go generators remain safe without
deep compiler integration.



In C# you would implement this with a switch expression or let the debugger
show the enum name. Go's approach makes the printed value match the source name in all
contexts - log files, JSON output, error messages - without runtime reflection.


Writing a Generator with text/template


stringer covers one specific problem. For custom generation, you write a Go
program. The standard approach is to use text/template to turn a data
description into Go source code.



Suppose you have a configuration struct that grows over time and you want to generate a
Merge(other Config) Config method that copies only the non-zero fields. Writing
this by hand is tedious and error-prone. A generator makes it trivial to keep up with
struct changes.



First, define the generator as a standalone program, conventionally in a
cmd/gen-merge/ directory:



// cmd/gen-merge/main.go
package main

import (
    "os"
    "text/template"
)

// Field describes one field to merge.
type Field struct {
    Name string
    Type string
}

var fields = []Field{
    {"Host", "string"},
    {"Port", "int"},
    {"Timeout", "time.Duration"},
    {"Debug", "bool"},
}

var tmpl = template.Must(template.New("merge").Parse(`// Code generated by gen-merge; DO NOT EDIT.

package config

import "time"

// Merge returns a new Config with non-zero fields from other overriding c.
func (c Config) Merge(other Config) Config {
    out := c
{{- range .}}
    if other.{{.Name}} != {{zeroOf .Type}} {
        out.{{.Name}} = other.{{.Name}}
    }
{{- end}}
    return out
}
`))

func main() {
    // In a real generator you would parse flags for the output file path.
    if err := tmpl.Execute(os.Stdout, fields); err != nil {
        panic(err)
    }
}



The {{- range .}} iterates over the slice; {{.Name}} inserts the
field name. Template actions (the {{...}} parts) are evaluated; everything
else is copied verbatim into the output. After piping the output through
gofmt, you have clean, formatted Go code.



In practice, add a helper function map to the template for things like zeroOf:



var tmpl = template.Must(template.New("merge").Funcs(template.FuncMap{
    "zeroOf": func(t string) string {
        switch t {
        case "string":
            return `""`
        case "bool":
            return "false"
        case "time.Duration", "int", "int64":
            return "0"
        default:
            return "nil"
        }
    },
}).Parse(rawTemplate))



Wire this into the package that consumes it:



// config/config.go
package config

import "time"

//go:generate go run ../cmd/gen-merge/main.go | gofmt > merge_gen.go

type Config struct {
    Host    string
    Port    int
    Timeout time.Duration
    Debug   bool
}



The convention for generated file names is to end with _gen.go or
_string.go. Always include the // Code generated ... DO NOT EDIT.
header - gofmt, go vet, and many editors recognise this and
suppress lint warnings on the file.


Inspecting Code with go/ast


Hardcoding the field list in the generator defeats the purpose - you still need to update
two files. The proper approach is to parse the Go source and extract the information
automatically using go/ast and go/parser.



go/ast gives you the Abstract Syntax Tree of a Go file as a data structure you
can walk. This is what tools like stringer, mockgen, and
protoc-gen-go do internally.



package main

import (
    "fmt"
    "go/ast"
    "go/parser"
    "go/token"
)

func main() {
    fset := token.NewFileSet()
    // Parse the source file
    f, err := parser.ParseFile(fset, "config/config.go", nil, 0)
    if err != nil {
        panic(err)
    }

    // Walk declarations looking for the Config struct
    for _, decl := range f.Decls {
        genDecl, ok := decl.(*ast.GenDecl)
        if !ok {
            continue
        }
        for _, spec := range genDecl.Specs {
            typeSpec, ok := spec.(*ast.TypeSpec)
            if !ok || typeSpec.Name.Name != "Config" {
                continue
            }
            structType, ok := typeSpec.Type.(*ast.StructType)
            if !ok {
                continue
            }
            // Print each field name and type
            for _, field := range structType.Fields.List {
                for _, name := range field.Names {
                    fmt.Printf("Field: %s\n", name.Name)
                }
            }
        }
    }
}



The AST types mirror the language syntax exactly: *ast.GenDecl is a
type, const, var, or import block;
*ast.TypeSpec is a single type declaration; *ast.StructType is
a struct literal. Once you have the StructType, you can extract field names,
types, and struct tags.



For larger generators that need full type information - resolving imported types, checking
interface satisfaction - use go/types and golang.org/x/tools/go/packages
instead of the raw parser. These packages load a full package graph, not just a single file.



In C, there is no equivalent - the preprocessor sees tokens, not a typed AST, so genuine
code inspection requires external tools like libclang. In C#, source generators have access
to the full Roslyn semantic model, which is similar in power to go/types but
runs inside the compiler rather than as an external program.


Practical Patterns


A few conventions the Go community has settled on:



Name generated files with a _gen.go suffix or a tool-specific suffix like _string.go. Do not use generic names like generated.go - when you have multiple generators they will collide.
Always include the header // Code generated by TOOL; DO NOT EDIT. on the first line. The exact wording matters - the Go toolchain and many editors check for this string.
Check generated files into version control. This means diffs show both the input change and the generated output change together, making code review coherent.
Run generators in CI with a subsequent git diff --exit-code to detect stale generated files.
Keep generators in cmd/ or as separate modules if they are only needed for development, not at runtime.


Comparison: C Macros, C# Source Generators, Go Generate


C macros operate at the token level before parsing. They are Turing-complete but produce
code invisible to debuggers and type checkers. X-macros - a C pattern where a macro is
defined and undefined repeatedly to generate parallel code - can generate enum values,
string tables, and switch cases from a single list. It works, but the output is unreadable.



C# source generators (Roslyn, C# 9+) run inside the compiler. They receive the compilation
as input and can add files to it but cannot modify existing files. The advantage is
tight IDE integration - generated code is immediately available to IntelliSense. The
disadvantage is complexity: writing a source generator requires knowing the Roslyn API,
and debugging one is painful.



Go go generate runs external programs. The output is ordinary Go source. It is
the most transparent approach: you can cat the generated file, step through it
in a debugger, and diff it in code review. The trade-off is that you must run the generator
explicitly and check the output in. For most teams this is a small cost for a large gain in
clarity.


Summary


go generate is a lightweight wrapper around external commands - its power comes
from the tools you invoke, not from any magic in the directive itself. For enum string
methods, stringer is the standard tool. For custom generation, combine
text/template (to emit code) with go/ast (to read existing code)
and write a small standalone Go program. Generated files should always carry the
// Code generated header, be checked into version control, and be validated in CI.



The approach contrasts with C macros (invisible to tooling) and C# source generators
(deep compiler integration) in favour of transparency: the generated code is just code.
Coming from the tooling article (Part 14), you can see how go generate fits
into the broader Go automation philosophy - automate the boring parts, but keep the result
readable.
