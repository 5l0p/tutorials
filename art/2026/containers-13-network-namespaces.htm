Containers - Part 13: Network Namespaces

When you run a Docker container and it has its own IP address, its own routing table, and its own iptables rules completely separate from the host, that isolation comes from a single kernel primitive: the network namespace. A network namespace is a full copy of the networking stack - interfaces, routes, neighbour tables, netfilter rules, sockets - confined to one namespace. Processes inside see only the interfaces that belong to their namespace. This article builds up that isolation step by step: creating a namespace, connecting it to the host with a virtual Ethernet pair, bridging multiple containers together, and NATting outbound traffic so the container can reach the internet. Every step is implemented first with shell commands and then programmatically from Go using golang.org/x/sys/unix and the netlink library.

What You Will Learn

  • What a network namespace contains and how the kernel isolates network stacks
  • How to create a network namespace with ip netns and with clone()/unshare()
  • How virtual Ethernet pairs (veth) connect two namespaces
  • How a Linux bridge aggregates multiple container interfaces
  • How to NAT container traffic through the host with iptables MASQUERADE
  • How to do all of the above programmatically from Go using github.com/vishvananda/netlink

What a Network Namespace Contains

The kernel defines a network namespace as an isolated instance of the network stack. Specifically, each namespace has its own:

  • Network interfaces (each interface belongs to exactly one namespace at a time)
  • IPv4 and IPv6 routing tables
  • ARP and neighbour tables
  • netfilter (iptables/nftables) rules
  • The loopback device (lo, which starts down and must be brought up separately)
  • Socket table - so ss -tulpn inside a namespace only shows that namespace's sockets
  • Network sysctls under /proc/sys/net/

When a process is in the initial network namespace it sees all the host's interfaces. When it enters a new network namespace it starts with nothing but a down loopback interface. This is why a freshly created container has no network access until the runtime wires it up.

You can inspect the namespace of any process at /proc/<pid>/ns/net. Two processes sharing the same inode number share the same network namespace. 

ls -la /proc/self/ns/net
# lrwxrwxrwx ... net -> net:[4026531992]

# Create a named network namespace (persists in /var/run/netns/)
sudo ip netns add myns

# List existing named namespaces
ip netns list
# myns

# Run a command inside the namespace
sudo ip netns exec myns ip link list
# 1: lo: <LOOPBACK> mtu 65536 qdisc noop state DOWN mode DEFAULT

Named network namespaces created by ip netns add are bind-mounted under /var/run/netns/. This keeps the namespace alive even when no process is using it, just as bind-mounting a namespace fd to a file does for any namespace type. Delete it with sudo ip netns del myns.

Virtual Ethernet Pairs

A veth pair is a pair of virtual network interfaces connected back-to-back: a packet written to one end immediately appears on the other, exactly like a physical crossover cable. The kernel creates them atomically as a pair; moving one end into a different network namespace forms a link between the two namespaces.

This is the fundamental building block of container networking. The runtime creates a veth pair, leaves one end (usually named vethXXXXXX) in the host namespace, moves the other (usually named eth0) into the container's network namespace, assigns IP addresses to both ends, and the container can now communicate with the host. 

# Create a veth pair: veth0 <--> veth1
sudo ip link add veth0 type veth peer name veth1

# Both are in the host namespace now
ip link show veth0
ip link show veth1

# Move veth1 into the named namespace
sudo ip link set veth1 netns myns

# Assign addresses and bring interfaces up
sudo ip addr add 10.200.0.1/24 dev veth0
sudo ip link set veth0 up

sudo ip netns exec myns ip addr add 10.200.0.2/24 dev veth1
sudo ip netns exec myns ip link set veth1 up
sudo ip netns exec myns ip link set lo up

# Test
sudo ip netns exec myns ping -c 2 10.200.0.1
# PING 10.200.0.1: 2 packets transmitted, 2 received

At this point the container can reach the host's veth address. It cannot yet reach anything else because it has no default route and the host does not NAT for it. We will fix both shortly.

Connecting Multiple Containers with a Bridge

A Linux bridge is a virtual Layer 2 switch. Attaching multiple veth host-ends to the bridge makes all those container namespaces able to communicate with each other at Ethernet level, exactly as if they were on the same LAN segment. Docker creates a bridge called docker0 by default; all container veth pairs are attached to it. 

# Create a bridge
sudo ip link add br0 type bridge
sudo ip addr add 10.201.0.1/24 dev br0
sudo ip link set br0 up

# Create two namespaces and connect them via the bridge
sudo ip netns add ns1
sudo ip netns add ns2

# First veth pair
sudo ip link add veth-ns1 type veth peer name veth-ns1-br
sudo ip link set veth-ns1 netns ns1
sudo ip link set veth-ns1-br master br0
sudo ip link set veth-ns1-br up

sudo ip netns exec ns1 ip addr add 10.201.0.2/24 dev veth-ns1
sudo ip netns exec ns1 ip link set veth-ns1 up
sudo ip netns exec ns1 ip link set lo up
sudo ip netns exec ns1 ip route add default via 10.201.0.1

# Second veth pair
sudo ip link add veth-ns2 type veth peer name veth-ns2-br
sudo ip link set veth-ns2 netns ns2
sudo ip link set veth-ns2-br master br0
sudo ip link set veth-ns2-br up

sudo ip netns exec ns2 ip addr add 10.201.0.3/24 dev veth-ns2
sudo ip netns exec ns2 ip link set veth-ns2 up
sudo ip netns exec ns2 ip link set lo up
sudo ip netns exec ns2 ip route add default via 10.201.0.1

# ns1 can now reach ns2 and the bridge address
sudo ip netns exec ns1 ping -c 2 10.201.0.3

NAT with iptables MASQUERADE

To give containers internet access, the host must act as a NAT router: packets from the container's private address range are rewritten so they appear to come from the host's public interface, and response packets are tracked and rewritten back. This uses the netfilter MASQUERADE target, which automatically uses the outgoing interface's address (unlike SNAT, which requires a static IP). 

# Enable IP forwarding (containers need the host to forward packets)
sudo sysctl -w net.ipv4.ip_forward=1

# NAT all traffic leaving from the container subnet via the host's eth0
sudo iptables -t nat -A POSTROUTING \
    -s 10.201.0.0/24 ! -o br0 -j MASQUERADE

# Allow forwarding between br0 and the outside
sudo iptables -A FORWARD -i br0 -j ACCEPT
sudo iptables -A FORWARD -o br0 -j ACCEPT

# Test from inside ns1
sudo ip netns exec ns1 ping -c 2 8.8.8.8

The ! -o br0 clause prevents the rule from matching packets that stay within the bridge (container-to-container traffic), which would break their source addresses unnecessarily.

To make IP forwarding persistent across reboots, add net.ipv4.ip_forward = 1 to /etc/sysctl.conf or a file under /etc/sysctl.d/.

Creating a Network Namespace from C

To give a process its own network namespace at creation time, pass CLONE_NEWNET to clone(). The child starts with only a loopback interface (in the DOWN state). The parent then has to wire up the networking while the child waits - a common pattern using a pipe as a synchronisation primitive. 

#define _GNU_SOURCE
#include <fcntl.h>
#include <sched.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/types.h>
#include <sys/wait.h>
#include <unistd.h>

#define STACK_SIZE (1024 * 1024)

/* Pipe used to synchronise parent and child */
static int sync_pipe[2];

static int child_main(void *arg) {
    (void)arg;
    char buf[1];

    /* Wait for parent to configure networking */
    read(sync_pipe[0], buf, 1);
    close(sync_pipe[0]);

    printf("[container] network interfaces:\n");
    system("ip link list");

    printf("[container] routes:\n");
    system("ip route list");

    return 0;
}

int main(void) {
    if (pipe(sync_pipe) == -1) { perror("pipe"); return 1; }

    char *stack = malloc(STACK_SIZE);
    if (!stack) { perror("malloc"); return 1; }

    pid_t pid = clone(child_main,
                      stack + STACK_SIZE,
                      CLONE_NEWNET | CLONE_NEWPID | SIGCHLD,
                      NULL);
    if (pid == -1) { perror("clone"); return 1; }

    printf("[host] child PID = %d\n", pid);

    /*
     * At this point we would use netlink or shell commands to:
     *   1. Create a veth pair.
     *   2. Move one end into /proc//ns/net.
     *   3. Assign addresses.
     * For brevity we just signal the child immediately.
     */
    write(sync_pipe[1], "x", 1);
    close(sync_pipe[1]);

    waitpid(pid, NULL, 0);
    free(stack);
    return 0;
}

Compile: gcc -o netns_demo netns_demo.c && sudo ./netns_demo. The child will show only a loopback interface in the DOWN state, confirming complete network isolation.

Doing It All from Go with netlink

Shell commands and C are good for understanding. In a real runtime you want programmatic control. The github.com/vishvananda/netlink package provides a Go API over the kernel's netlink socket interface - the same mechanism that ip uses internally. 

// go.mod: require github.com/vishvananda/netlink v1.2.1
// Also requires: golang.org/x/sys
// Build: go build -o netns_go ./...
// Run as root: sudo ./netns_go

package main

import (
    "fmt"
    "log"
    "net"
    "os"
    "os/exec"
    "runtime"

    "github.com/vishvananda/netlink"
    "golang.org/x/sys/unix"
)

func main() {
    if os.Getuid() != 0 {
        log.Fatal("must run as root")
    }

    // Save the host network namespace fd so we can return to it later.
    hostNS, err := os.Open("/proc/self/ns/net")
    if err != nil {
        log.Fatal(err)
    }
    defer hostNS.Close()

    // Create a named network namespace (bind-mounts it under /var/run/netns/).
    if err := exec.Command("ip", "netns", "add", "go-demo").Run(); err != nil {
        log.Fatalf("ip netns add: %v", err)
    }
    defer exec.Command("ip", "netns", "del", "go-demo").Run()

    // Open the new namespace fd.
    containerNS, err := os.Open("/var/run/netns/go-demo")
    if err != nil {
        log.Fatal(err)
    }
    defer containerNS.Close()

    // Create a veth pair in the host namespace.
    veth := &netlink.Veth{
        LinkAttrs: netlink.LinkAttrs{Name: "veth-host"},
        PeerName:  "veth-cont",
    }
    if err := netlink.LinkAdd(veth); err != nil {
        log.Fatalf("LinkAdd veth: %v", err)
    }
    defer netlink.LinkDel(veth)

    // Move the peer end into the container namespace.
    peer, err := netlink.LinkByName("veth-cont")
    if err != nil {
        log.Fatalf("LinkByName veth-cont: %v", err)
    }
    if err := netlink.LinkSetNsFd(peer, int(containerNS.Fd())); err != nil {
        log.Fatalf("LinkSetNsFd: %v", err)
    }

    // Configure the host end.
    hostAddr := &netlink.Addr{
        IPNet: &net.IPNet{
            IP:   net.ParseIP("10.202.0.1"),
            Mask: net.CIDRMask(24, 32),
        },
    }
    host, _ := netlink.LinkByName("veth-host")
    netlink.AddrAdd(host, hostAddr)
    netlink.LinkSetUp(host)

    // Switch into the container namespace to configure the peer end.
    // LockOSThread ensures the goroutine stays on this OS thread while we
    // manipulate thread-local state (the current network namespace).
    runtime.LockOSThread()
    defer runtime.UnlockOSThread()

    if err := unix.Setns(int(containerNS.Fd()), unix.CLONE_NEWNET); err != nil {
        log.Fatalf("setns: %v", err)
    }

    cont, err := netlink.LinkByName("veth-cont")
    if err != nil {
        log.Fatalf("LinkByName in container ns: %v", err)
    }
    contAddr := &netlink.Addr{
        IPNet: &net.IPNet{
            IP:   net.ParseIP("10.202.0.2"),
            Mask: net.CIDRMask(24, 32),
        },
    }
    netlink.AddrAdd(cont, contAddr)
    netlink.LinkSetUp(cont)

    lo, _ := netlink.LinkByName("lo")
    netlink.LinkSetUp(lo)

    // Add a default route via the host end.
    gw := net.ParseIP("10.202.0.1")
    netlink.RouteAdd(&netlink.Route{
        Scope:   netlink.SCOPE_UNIVERSE,
        Gw:      gw,
    })

    links, _ := netlink.LinkList()
    for _, l := range links {
        fmt.Printf("[container ns] interface: %s\n", l.Attrs().Name)
    }

    // Return to the host namespace.
    if err := unix.Setns(int(hostNS.Fd()), unix.CLONE_NEWNET); err != nil {
        log.Fatalf("setns back to host: %v", err)
    }

    fmt.Println("Network namespace configured successfully.")
    fmt.Println("From host: ping 10.202.0.2 to reach the container veth.")
}

The key insight is runtime.LockOSThread(). Network namespace membership is a per-thread property in the Linux kernel (stored in the thread's task_struct). Go's scheduler can move goroutines between OS threads. Locking the goroutine to one OS thread guarantees that the setns call and all subsequent netlink calls happen on the same thread and therefore see the same namespace.

How Docker and runc Use Network Namespaces

When you run docker run, the Docker daemon delegates container creation to runc via the OCI runtime spec. runc creates a new network namespace with CLONE_NEWNET, then signals the network plugin (by default the CNI plugin bridge). The CNI plugin creates a bridge if one does not exist, creates a veth pair, moves one end into the container namespace, assigns addresses from a configured IPAM pool, and sets up the default route. On container exit the CNI plugin cleans up.

You can inspect what Docker has set up: 

# Find the container's init PID
CPID=$(docker inspect --format '{{.State.Pid}}' my-container)

# Enter its network namespace
sudo nsenter --net=/proc/$CPID/ns/net -- ip link list
sudo nsenter --net=/proc/$CPID/ns/net -- ip route list
sudo nsenter --net=/proc/$CPID/ns/net -- ss -tulpn

# See which iptables rules Docker added
sudo iptables -t nat -L DOCKER -n -v
sudo iptables -L DOCKER-ISOLATION-STAGE-1 -n -v

Port Forwarding

When you use docker run -p 8080:80, Docker adds a DNAT rule in the nat table's PREROUTING chain: packets arriving at the host on port 8080 have their destination rewritten to the container's IP on port 80. The connection tracking module (conntrack) handles the reverse translation automatically. 

# Equivalent manual rule for a container at 10.201.0.2
sudo iptables -t nat -A PREROUTING \
    -p tcp --dport 8080 \
    -j DNAT --to-destination 10.201.0.2:80

# Also needed for access from the host itself (OUTPUT chain)
sudo iptables -t nat -A OUTPUT \
    -p tcp --dport 8080 \
    -j DNAT --to-destination 10.201.0.2:80

Cleaning Up

# Remove iptables rules
sudo iptables -t nat -D POSTROUTING -s 10.201.0.0/24 ! -o br0 -j MASQUERADE
sudo iptables -D FORWARD -i br0 -j ACCEPT
sudo iptables -D FORWARD -o br0 -j ACCEPT

# Delete namespaces
sudo ip netns del ns1
sudo ip netns del ns2

# Delete the bridge (veth pairs attached to it are automatically cleaned up)
sudo ip link del br0

Summary

A network namespace gives a process its own complete, isolated copy of the Linux networking stack. Veth pairs are the cables that connect namespaces to each other or to a bridge. A bridge aggregates multiple namespaces at Layer 2. NAT with iptables MASQUERADE gives containers outbound internet access. Port forwarding with DNAT gives external hosts access to services inside containers.

All of this is achievable from Go using github.com/vishvananda/netlink, which sends netlink messages to the kernel - the same messages that the ip command sends. The critical gotcha when working with namespaces in Go is runtime.LockOSThread(): namespace membership is per-thread, not per-goroutine, and failing to lock the thread leads to subtle races where network operations silently land in the wrong namespace.
