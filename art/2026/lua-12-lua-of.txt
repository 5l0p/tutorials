Lua - Part 12: Lua in openFrameworks
2026-12-03


openFrameworks is a C++ creative coding framework - its DNA is about rapid prototyping
of audio-visual ideas. Lua fits this context perfectly: you write the slow-changing
structural logic in Lua (which nodes exist, what parameters they have, how they connect),
and openFrameworks handles the heavy lifting in C++ (OpenGL, audio buffers, OS windows).
With ofxLua installed and a file-change watcher in update(), editing a Lua
sketch and seeing the result update in a running window takes under a second. This
article covers the complete workflow from addon installation to live-reloading a custom
C++ binding, and closes with a series retrospective on what Lua embedding gives you
across the domains this series has covered.


What You Will Learn


How to add the ofxLua addon and hook it into the openFrameworks project generator.
Calling Lua setup, update, and draw from C++ lifecycle methods.
Binding oF draw calls (ofDrawCircle, ofSetColor) to Lua.
Live-reloading on file change with a stat() mtime check in update().
Writing your own ofxLua binding for a custom C++ class.
A complete audio-visual sketch described entirely in Lua.


Adding ofxLua


ofxLua is a community addon that wraps a Lua state and provides convenience methods for
the openFrameworks lifecycle. Install it alongside your project:



# From your openFrameworks addons directory
cd OF_ROOT/addons
git clone https://github.com/danomatika/ofxLua.git

# Then use the openFrameworks project generator to add it to your project,
# or add manually to addons.make:
echo "ofxLua" >> path/to/your/project/addons.make



ofxLua uses Lua 5.4 (recent versions) or can be configured for LuaJIT. It handles
the lua_State lifecycle internally and exposes a higher-level C++ API.
You can also bypass it and manage the lua_State * directly using
everything you have learned in this series - ofxLua is a convenience wrapper, not a
requirement.


Project Structure


A live-scriptable openFrameworks project has this layout:



mySketch/
    src/
        ofApp.h          -- C++ app class
        ofApp.cpp        -- lifecycle + Lua bridge
        LuaBindings.cpp  -- register C++ functions with Lua
    bin/data/
        sketch.lua       -- the live-editable Lua sketch



The bin/data/ directory is where openFrameworks looks for assets. Putting
the Lua script there means it is accessible by filename without an absolute path, and
your text editor can modify it while the application runs.


The C++ App Class


// ofApp.h
#pragma once
#include "ofMain.h"
extern "C" {
#include <lua.h>
#include <lualib.h>
#include <lauxlib.h>
}
#include <sys/stat.h>

class ofApp : public ofBaseApp {
public:
    void setup();
    void update();
    void draw();

private:
    void loadScript(const string &path);
    void callLua(const char *fn);
    bool scriptChanged();

    lua_State   *L        = nullptr;
    string       scriptPath;
    time_t       lastMtime = 0;
};



// ofApp.cpp
#include "ofApp.h"

/* Forward declaration - defined in LuaBindings.cpp */
void registerBindings(lua_State *L);

void ofApp::setup() {
    ofSetWindowTitle("Lua Sketch");
    ofSetFrameRate(60);

    scriptPath = ofToDataPath("sketch.lua", true);

    L = luaL_newstate();
    luaL_openlibs(L);
    registerBindings(L);  /* register oF draw calls as Lua globals */
    loadScript(scriptPath);
}

void ofApp::update() {
    if (scriptChanged()) {
        ofLogNotice("ofApp") << "Reloading " << scriptPath;
        loadScript(scriptPath);
    }
    callLua("update");
}

void ofApp::draw() {
    callLua("draw");
}

void ofApp::loadScript(const string &path) {
    /* Clear previous globals set by the script */
    /* (selective: only what the script defines) */

    if (luaL_loadfile(L, path.c_str()) != LUA_OK) {
        ofLogError("ofApp") << "Load error: " << lua_tostring(L, -1);
        lua_pop(L, 1);
        return;
    }
    if (lua_pcall(L, 0, 0, 0) != LUA_OK) {
        ofLogError("ofApp") << "Runtime error: " << lua_tostring(L, -1);
        lua_pop(L, 1);
        return;
    }

    /* Call Lua setup() after loading */
    callLua("setup");

    /* Update mtime so we do not immediately re-trigger */
    struct stat st;
    if (stat(path.c_str(), &st) == 0)
        lastMtime = st.st_mtime;
}

void ofApp::callLua(const char *fn) {
    lua_getglobal(L, fn);
    if (!lua_isfunction(L, -1)) {
        lua_pop(L, 1);
        return;   /* function not defined - silently skip */
    }
    if (lua_pcall(L, 0, 0, 0) != LUA_OK) {
        ofLogError("ofApp") << fn << "(): " << lua_tostring(L, -1);
        lua_pop(L, 1);
    }
}

bool ofApp::scriptChanged() {
    struct stat st;
    if (stat(scriptPath.c_str(), &st) != 0) return false;
    return st.st_mtime != lastMtime;
}


Binding oF Draw Calls to Lua


Expose a minimal set of openFrameworks draw calls as Lua functions. Each wrapper is a
lua_CFunction (as covered in Part 4):



// LuaBindings.cpp
extern "C" {
#include <lua.h>
#include <lualib.h>
#include <lauxlib.h>
}
#include "ofMain.h"

/* of.background(r, g, b) */
static int lua_of_background(lua_State *L) {
    float r = (float)luaL_checknumber(L, 1);
    float g = (float)luaL_checknumber(L, 2);
    float b = (float)luaL_checknumber(L, 3);
    ofBackground(r, g, b);
    return 0;
}

/* of.set_color(r, g, b [, a]) */
static int lua_of_set_color(lua_State *L) {
    float r = (float)luaL_checknumber(L, 1);
    float g = (float)luaL_checknumber(L, 2);
    float b = (float)luaL_checknumber(L, 3);
    float a = (float)luaL_optnumber(L, 4, 255.0);
    ofSetColor(r, g, b, a);
    return 0;
}

/* of.draw_circle(x, y, radius) */
static int lua_of_draw_circle(lua_State *L) {
    float x = (float)luaL_checknumber(L, 1);
    float y = (float)luaL_checknumber(L, 2);
    float r = (float)luaL_checknumber(L, 3);
    ofDrawCircle(x, y, r);
    return 0;
}

/* of.draw_rect(x, y, w, h) */
static int lua_of_draw_rect(lua_State *L) {
    float x = (float)luaL_checknumber(L, 1);
    float y = (float)luaL_checknumber(L, 2);
    float w = (float)luaL_checknumber(L, 3);
    float h = (float)luaL_checknumber(L, 4);
    ofDrawRectangle(x, y, w, h);
    return 0;
}

/* of.elapsed_time() -> seconds since app start */
static int lua_of_elapsed_time(lua_State *L) {
    lua_pushnumber(L, ofGetElapsedTimef());
    return 1;
}

/* of.width(), of.height() */
static int lua_of_width(lua_State *L)  { lua_pushinteger(L, ofGetWidth());  return 1; }
static int lua_of_height(lua_State *L) { lua_pushinteger(L, ofGetHeight()); return 1; }

/* of.draw_string(text, x, y) */
static int lua_of_draw_string(lua_State *L) {
    const char *text = luaL_checkstring(L, 1);
    float x = (float)luaL_checknumber(L, 2);
    float y = (float)luaL_checknumber(L, 3);
    ofDrawBitmapString(text, x, y);
    return 0;
}

static const luaL_Reg of_funcs[] = {
    { "background",   lua_of_background   },
    { "set_color",    lua_of_set_color    },
    { "draw_circle",  lua_of_draw_circle  },
    { "draw_rect",    lua_of_draw_rect    },
    { "elapsed_time", lua_of_elapsed_time },
    { "width",        lua_of_width        },
    { "height",       lua_of_height       },
    { "draw_string",  lua_of_draw_string  },
    { NULL, NULL }
};

void registerBindings(lua_State *L) {
    lua_createtable(L, 0, 8);
    luaL_setfuncs(L, of_funcs, 0);
    lua_setglobal(L, "of");   /* available as of.background(), etc. */
}


The Lua Sketch


With the bindings in place, the entire visual logic lives in Lua:



-- bin/data/sketch.lua
-- Audio-visual sketch: bouncing circles with audio-reactive sizes
-- Edit this file and save; the running sketch updates immediately.

local circles = {}
local NUM = 12

function setup()
    circles = {}
    local W = of.width()
    local H = of.height()
    for i = 1, NUM do
        local angle = (i - 1) * (2 * math.pi / NUM)
        circles[i] = {
            x     = W/2 + math.cos(angle) * 180,
            y     = H/2 + math.sin(angle) * 180,
            base_r = 20 + i * 4,
            phase = angle,
            r     = 180 + math.floor(i * 255 / NUM),
            g     = 80,
            b     = 200 - math.floor(i * 150 / NUM),
        }
    end
end

function update()
    -- nothing heavy in update for this sketch
end

function draw()
    local t = of.elapsed_time()
    of.background(10, 10, 20)

    local W = of.width()
    local H = of.height()

    for i, c in ipairs(circles) do
        -- size pulses with time
        local pulse = 1.0 + 0.4 * math.sin(t * 2.0 + c.phase)
        local radius = c.base_r * pulse

        of.set_color(c.r, c.g, c.b, 200)
        of.draw_circle(c.x, c.y, radius)

        -- connecting lines to centre
        of.set_color(c.r, c.g, c.b, 60)
        -- ofDrawLine not bound here but could be added the same way
    end

    -- HUD
    of.set_color(200, 200, 200, 180)
    of.draw_string(string.format("t=%.2f  circles=%d", t, #circles), 12, 20)
    of.draw_string("Edit sketch.lua to live-reload", 12, 38)
end


Binding a Custom C++ Class


Suppose you have a C++ audio analyser that computes RMS levels per frame. Expose it
as a Lua userdata (Part 7 pattern, translated to C++):



// AudioAnalyser.h
class AudioAnalyser {
public:
    void push_sample(float s)   { sum_ += s*s; count_++; }
    float rms() const           { return count_ ? sqrtf(sum_/count_) : 0; }
    void reset()                { sum_ = 0; count_ = 0; }
private:
    float sum_ = 0;
    int   count_ = 0;
};



// In LuaBindings.cpp - add after the existing bindings

#define ANALYSER_MT "AudioAnalyser"

static int lua_analyser_new(lua_State *L) {
    AudioAnalyser *a = new(lua_newuserdata(L, sizeof(AudioAnalyser))) AudioAnalyser();
    luaL_getmetatable(L, ANALYSER_MT);
    lua_setmetatable(L, -2);
    return 1;
}
static int lua_analyser_rms(lua_State *L) {
    auto *a = static_cast<AudioAnalyser *>(luaL_checkudata(L, 1, ANALYSER_MT));
    lua_pushnumber(L, a->rms());
    return 1;
}
static int lua_analyser_reset(lua_State *L) {
    auto *a = static_cast<AudioAnalyser *>(luaL_checkudata(L, 1, ANALYSER_MT));
    a->reset();
    return 0;
}
static int lua_analyser_gc(lua_State *L) {
    auto *a = static_cast<AudioAnalyser *>(luaL_checkudata(L, 1, ANALYSER_MT));
    a->~AudioAnalyser();   /* call destructor; Lua frees the memory */
    return 0;
}

static const luaL_Reg analyser_methods[] = {
    { "rms",   lua_analyser_rms   },
    { "reset", lua_analyser_reset },
    { "__gc",  lua_analyser_gc    },
    { NULL, NULL }
};
static const luaL_Reg analyser_module[] = {
    { "new", lua_analyser_new },
    { NULL, NULL }
};

/* Add this to registerBindings(), after the 'of' table: */
static void register_analyser(lua_State *L) {
    luaL_newmetatable(L, ANALYSER_MT);
    lua_pushvalue(L, -1);
    lua_setfield(L, -2, "__index");
    luaL_setfuncs(L, analyser_methods, 0);
    lua_pop(L, 1);

    lua_createtable(L, 0, 1);
    luaL_setfuncs(L, analyser_module, 0);
    lua_setglobal(L, "AudioAnalyser");
}



Note the destructor call in __gc. Because the memory was allocated with
placement new (new(lua_newuserdata(...))), you must explicitly call the
destructor. The memory itself is freed by Lua's allocator after __gc returns.
For trivially destructible types like plain structs, the destructor call can be omitted.


Live Reload in Practice


The mtime check in update() fires every frame (60 times per second). On
Linux and macOS, stat() on a local file takes under 1 microsecond - well
within budget. On network filesystems it may be slower; poll less frequently in that
case with a frame counter.



When the script is reloaded, setup() runs again. For a visual sketch this
means state is reset: positions, counters, random seeds. For a persistent-state sketch
(one that accumulates data across frames), save state to globals before reload and
restore it in setup():



-- sketch.lua - preserving state across hot-reloads
local state = state or {}   -- 'state' persists if already defined
state.frame_count = state.frame_count or 0

function setup()
    -- Only reset what should change when parameters change
    -- state.frame_count continues from where it was
end

function update()
    state.frame_count = state.frame_count + 1
end



The state = state or {} idiom works because re-running the script
re-executes the top-level code, which evaluates state or {}. If
state is already a global (from the previous load), it is reused.
If it is nil (first load), a new table is created.


Series Retrospective


Across twelve articles this series has covered the complete Lua embedding toolkit:



The foundation (Parts 1-3): why Lua is the right embeddable language, the stack
model that makes the C API memory-safe, and the complete C-to-Lua call workflow.



The C-Lua interface (Parts 4-7): exposing C functions as Lua modules, reading
structured config tables, declaring typed component graphs with hot-reload, and wrapping
C structs as first-class Lua objects with metatables and automatic cleanup.



Advanced embedding (Parts 8-10): coroutines for sequential scripting in a
C event loop, sandboxing for untrusted scripts with selective libraries and instruction
limits, and LuaJIT for near-C-speed execution and zero-cost FFI calls to C functions.



Applications (Parts 11-12): a live-editable DSP pipeline and a live-editable
openFrameworks sketch - the pattern in full, end to end.



The common thread: C owns performance and resources; Lua describes structure and
behaviour. The interface is defined entirely in C. Scripts can only do what the C code
permits. Hot-reload is a property of the architecture, not a feature bolted on:
luaL_loadfile + lua_pcall again, reconstruct the C objects.
That is the entire mechanism for every example in the series.


Summary


ofxLua adds a Lua state to an openFrameworks project. The C++ app class calls Lua
setup, update, and draw via lua_pcall.
oF draw calls are exposed as a Lua of module using the lua_CFunction
pattern from Part 4. A stat() mtime check in update() triggers
live reload in under a frame. Custom C++ classes are bound using the userdata pattern
from Part 7, with placement new and explicit destructor call in __gc.



The full Lua embedding toolkit is now in hand. The remaining decisions are architectural:
which parts of your application should Lua describe, how much API surface to expose,
and how strictly to sandbox. Those decisions are yours; the mechanism to implement them
is fully covered.
