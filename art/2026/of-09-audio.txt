openFrameworks - Part 9: Audio I/O with ofSoundStream
2026-06-01


Real-time audio programming is one of the most demanding tasks you can give a CPU. The audio
callback runs on a dedicated high-priority thread; your code must produce or consume exactly
bufferSize samples before the driver's deadline, or you get a glitch (called an
underrun or xrun). openFrameworks abstracts the platform audio APIs - ALSA and JACK on Linux,
CoreAudio on macOS, WASAPI on Windows - behind a uniform interface built around
ofSoundStream and ofSoundBuffer. This article shows how to open the
stream, write audio processing code correctly, and avoid common latency and threading pitfalls.


What You Will Learn


How to configure ofSoundStreamSettings with sample rate, buffer size, and channel counts
How to open output and input streams and implement audioOut() / audioIn() callbacks
The golden rule of audio callbacks: no blocking, no allocation, no locks
Using a lock-free ring buffer to pass data between the audio thread and the main thread

When to use ofSoundPlayer for file playback vs ofSoundStream for custom DSP


The Audio Callback Model


Unlike the graphics loop, audio does not run on your main thread. The driver owns a real-time
thread that calls your callback at precise intervals. Each call provides a buffer of samples to
fill (output) or consume (input). The size of that buffer - the buffer size - is a
trade-off: smaller buffers mean lower latency but more CPU overhead; larger buffers mean
higher latency but more headroom to do computation.



A typical buffer size for interactive audio is 256 or 512 samples. At 44100 Hz sample rate,
256 samples corresponds to about 5.8 ms of audio. That is your processing deadline.


ofSoundStreamSettings and ofSoundStream


// ofApp.h
#pragma once
#include <ofMain.h>

class ofApp : public ofBaseApp {
public:
    void setup();
    void update();
    void draw();
    void exit();

    void audioIn(ofSoundBuffer &buffer);
    void audioOut(ofSoundBuffer &buffer);

private:
    ofSoundStream soundStream;
    float         inputLevel  = 0.0f;   // shared between audio + main thread
    float         sinePhase   = 0.0f;
};



// ofApp.cpp
#include "ofApp.h"

void ofApp::setup() {
    ofSetFrameRate(60);

    ofSoundStreamSettings settings;

    // Enumerate devices (optional - default device is used if you skip this)
    auto devices = soundStream.getDeviceList();
    for (auto &d : devices) {
        ofLogNotice("Audio") << d.deviceID << ": " << d.name
                             << " in=" << d.inputChannels
                             << " out=" << d.outputChannels;
    }

    settings.setInListener(this);    // this->audioIn will be called
    settings.setOutListener(this);   // this->audioOut will be called
    settings.sampleRate    = 44100;
    settings.numInputChannels  = 1;
    settings.numOutputChannels = 2;
    settings.bufferSize    = 256;
    settings.numBuffers    = 4;      // number of buffers in the queue

    soundStream.setup(settings);
}

void ofApp::exit() {
    soundStream.close();
}


The audioOut Callback: Generating Sound


audioOut is called when the driver needs more samples. Fill
buffer with the next chunk of audio. The buffer is interleaved: for stereo,
samples alternate L, R, L, R, &#8230;



void ofApp::audioOut(ofSoundBuffer &buffer) {
    float sampleRate = buffer.getSampleRate();
    int   numFrames  = buffer.getNumFrames();
    int   numChans   = buffer.getNumChannels();
    float freq = 440.0f;

    for (int i = 0; i < numFrames; i++) {
        float sample = sin(sinePhase) * 0.3f;
        sinePhase += TWO_PI * freq / sampleRate;
        if (sinePhase >= TWO_PI) sinePhase -= TWO_PI;

        for (int c = 0; c < numChans; c++) {
            buffer[i * numChans + c] = sample;
        }
    }
}



ofSoundBuffer overloads operator[] for indexed access to interleaved
samples. You can also use buffer.getSample(frame, channel) and
buffer.setSample(frame, channel, value) for more readable code.


The audioIn Callback: Capturing Sound


audioIn is called when the driver has a new buffer of captured samples ready.



void ofApp::audioIn(ofSoundBuffer &buffer) {
    int numFrames = buffer.getNumFrames();

    // Compute RMS level (root-mean-square = perceived loudness)
    float sum = 0.0f;
    for (int i = 0; i < numFrames; i++) {
        float s = buffer.getSample(i, 0);   // channel 0
        sum += s * s;
    }
    float rms = sqrt(sum / numFrames);

    // Write to a variable that draw() reads.
    // ofSoundBuffer callbacks run on the audio thread, so access to
    // shared data should be protected. For a single float, an atomic
    // is the cleanest option.
    inputLevel = rms;   // float assignment is atomic on x86 in practice,
                        // but use std::atomic<float> for correctness.
}


Thread Safety


The audio callback runs on a different thread from update() and
draw(). Reading and writing shared state without synchronisation is a data race -
undefined behaviour in C++. In practice the most common pattern is to copy audio data into a
ring buffer or a local snapshot at the start of update():



// ofApp.h
#include <atomic>
#include <mutex>

class ofApp : public ofBaseApp {
    // ...
    std::atomic<float>       inputLevel{0.0f};  // safe for single float
    std::mutex               bufferMutex;
    std::vector<float>       audioCopy;          // snapshot for draw thread
    std::vector<float>       liveBuffer;         // written by audio thread
};



void ofApp::audioIn(ofSoundBuffer &buffer) {
    std::lock_guard<std::mutex> lock(bufferMutex);
    liveBuffer.resize(buffer.getNumFrames());
    for (int i = 0; i < (int)buffer.getNumFrames(); i++) {
        liveBuffer[i] = buffer.getSample(i, 0);
    }
}

void ofApp::update() {
    {
        std::lock_guard<std::mutex> lock(bufferMutex);
        audioCopy = liveBuffer;
    }
    // Now use audioCopy freely in update() and draw() without holding the lock
}


Connecting DSP Logic


For anything more complex than a sine oscillator, build a DSP object that holds state
(oscillator phase, filter coefficients, delay line) and has a process() method
that the audio callback invokes:



// Sawtooth oscillator
struct SawOsc {
    float phase     = 0.0f;
    float sampleRate = 44100.0f;

    float next(float freq) {
        float out = phase * 2.0f - 1.0f;   // range [-1, 1]
        phase += freq / sampleRate;
        if (phase >= 1.0f) phase -= 1.0f;
        return out;
    }
};

// One-pole low-pass filter
struct LowPass {
    float prev = 0.0f;
    float process(float in, float cutoff01) {
        // cutoff01 in [0,1]: 0 = DC, 1 = no filtering
        prev = prev + cutoff01 * (in - prev);
        return prev;
    }
};



// In ofApp.h
SawOsc osc;
LowPass lp;

// In audioOut:
void ofApp::audioOut(ofSoundBuffer &buffer) {
    float freq   = 220.0f;
    float cutoff = 0.1f;
    for (int i = 0; i < (int)buffer.getNumFrames(); i++) {
        float s = lp.process(osc.next(freq), cutoff) * 0.5f;
        buffer.setSample(i, 0, s);
        buffer.setSample(i, 1, s);
    }
}


Buffer Size and Latency


The round-trip latency from microphone input to speaker output is approximately:



latency_ms = (bufferSize / sampleRate) * 1000 * numBuffers



At 44100 Hz, 256 samples, 4 buffers:



latency_ms = (256 / 44100) * 1000 * 4 = ~23 ms



For a live visualiser (audio drives graphics) 23 ms is imperceptible. For a musical
instrument where you are monitoring your own voice, anything above 10 ms is annoying.
Reduce buffer size and/or use JACK (Linux) or CoreAudio low-latency mode (macOS) for
performance-critical applications.


Choosing the Audio Backend


On Linux, oF supports ALSA (default), PulseAudio, and JACK. To select JACK, build with:



make AUDIO_LIBS="jack"



Or set settings.api = ofSoundDevice::JACK before calling
soundStream.setup(settings). JACK offers the lowest latency and integrates
with professional audio tools (Ardour, Pure Data). ALSA is simpler to set up and works
out of the box on most systems.


ofSoundPlayer: Simple Playback


For simple one-shot sound effects, ofSoundPlayer is easier than
ofSoundStream:



ofSoundPlayer click;

void ofApp::setup() {
    click.load("sounds/click.wav");
    click.setVolume(0.8f);
}

void ofApp::mousePressed(int x, int y, int button) {
    click.play();
}



ofSoundPlayer handles loading, decoding, and playback internally. It does not
give you access to the sample buffer, so you cannot do DSP on it. Use
ofSoundStream when you need that access.


Summary


Configure an ofSoundStreamSettings with sample rate, buffer size, and channel
counts, then call soundStream.setup(this, settings) in setup().
Override audioOut(ofSoundBuffer& buffer) to fill output samples and
audioIn(ofSoundBuffer& buffer) to consume input. The callback runs on a
high-priority audio thread — never allocate, block, or acquire locks inside it. Use a
lock-free ring buffer to transfer data to the main thread for visualisation. Use
ofSoundPlayer for simple file playback where DSP access is not needed.
