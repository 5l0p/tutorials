BSD Jails: Isolation Without Linux

On PID visibility: Linux PID namespaces give a process a completely private PID space — PID 1 inside the namespace is a different kernel task_struct entry than PID 1 on the host. FreeBSD jails do not remap PIDs. Process IDs are global. The sysctl security.bsd.see_other_jails=0 (the default) prevents processes in one jail from seeing processes in another, but the kernel data structures share the same PID space. This matters for kill(2): a jailed process cannot signal a process in another jail, but if a PID is somehow known, the kernel checks the jail boundary before allowing the signal.

On user namespaces: This is the sharpest difference. Linux user namespaces allow an unprivileged user to create a new user namespace in which they have uid 0, enabling rootless containers. FreeBSD has no equivalent. Root inside a jail is root — uid 0 with a reduced privilege set, but not a remapped identity. There is no mechanism for an unprivileged user on a FreeBSD host to create a jail.

On Capsicum vs seccomp: FreeBSD's Capsicum is a capability-based sandboxing system for processes, but it operates at a different level than jails. A process opts into capability mode with cap_enter(2), after which it can only use file descriptors it already holds and cannot open new paths. Capsicum is for hardening individual applications, not for container-style isolation. It is closer in spirit to OpenBSD's pledge(2) than to Linux seccomp, though seccomp's BPF filter mode and Capsicum both restrict which syscalls can be made — they just use different policy languages. Jails and Capsicum are orthogonal and can be combined.

What jails get right: The unified model is operationally simpler. There is no namespace escape class of vulnerability caused by leaving one dimension unconfigured. The kernel's jail checks are concentrated, auditable code paths rather than the sprawling namespace checks scattered across every Linux syscall. FreeBSD's security track record on jail escapes is strong.

What Linux namespaces get right: Flexibility. You can combine namespaces in ways that have no jail equivalent — a shared network namespace with separate mount namespaces, rootless containers via user namespaces, namespace file descriptors (/proc/self/ns/*) that can be passed over Unix sockets and entered later. The eBPF ecosystem built on top of namespaces and cgroups (for network policy, observability, and syscall filtering) has no FreeBSD counterpart at comparable maturity.

Rootless and Nested Jails

Nested jails — jails within jails — are supported since FreeBSD 8 via the children.max parameter. A jail with children.max=5 may create up to five child jails. The child jail's parameters must be a subset of the parent's: it cannot claim IP addresses the parent does not own, cannot have a path outside the parent's path, and cannot exceed the parent's resource limits.

# /etc/jail.conf

parent_jail {
    path          = "/jails/parent";
    ip4.addr      = "192.168.100.0/24";
    children.max  = 10;
    allow.mount;
    allow.mount.nullfs;
    enforce_statfs = 1;
}

From inside the parent jail (as root):

# jail -c name=child path=/jails/parent/child \
       host.hostname=child.internal \
       ip4.addr=192.168.100.20 \
       persist

This is the mechanism used by jail management frameworks (like iocage and bastille) to implement jail templates, thick jails, and thin jails from a shared base.

There is no rootless jail equivalent on FreeBSD. Creating a jail always requires host root privilege, and there is no mechanism analogous to Linux user namespaces that would allow a non-root user to set up the isolation boundary. This is a deliberate design decision: jails are an administrative primitive, not a user-level one.

For workloads that need jail-like isolation without root, FreeBSD offers Capsicum (application-level sandboxing, described above) and, in more recent work, mac_do(4), which allows specific privilege transitions under MAC framework control. Neither is a substitute for the full jail model.

Jail management tools: while jail(8) and jail.conf are the kernel interface, production deployments often use higher-level tools:

  • bastille(8): template-based jail manager, the closest thing to a Docker-style workflow on FreeBSD. Supports thin jails (shared base via nullfs), ZFS cloning, and a Bastillefile format similar to a Dockerfile.
  • iocage: Python-based, ZFS-native jail manager with a richer feature set. Less actively maintained than Bastille as of 2025.
  • cbsd: a comprehensive framework supporting both FreeBSD jails and bhyve VMs with a web UI.
  • Podman's FreeBSD backend: experimental support for running OCI containers on FreeBSD using jails as the isolation mechanism rather than Linux namespaces.

Summary

BSD jails are a unified, kernel-enforced isolation primitive that trades Linux's compositional flexibility for a simpler, more auditable security model. A jail gives a process tree an isolated filesystem root, a restricted IP set, a private hostname, and a fixed reduction in root's privileges — all enforced through a single struct prison attached to every credential in the jail.

You create jails with jail(8) and /etc/jail.conf, which invoke the jail(2)/jail_set(2) syscall family. The kernel allocates a prison structure, validates parameters, and fires subsystem init hooks. Processes enter an existing jail with jail_attach(2), the call behind jexec(8).

Networking is either shared-stack with IP aliasing or fully isolated with VNET — a complete per-jail network stack equivalent to a Linux network namespace, connected to the outside world via epair(4) pairs (the FreeBSD veth). Resource limits come from rctl(8) (hard caps on memory, CPU percentage, process count, and more) and CPU affinity from cpuset(1), together covering most of what cgroups provide, minus I/O control and the v2 hierarchy model.

The key differences from Linux containers: no PID remapping, no user namespace, no rootless path, but also no namespace misconfiguration attack surface and a substantially smaller, more auditable kernel code path for the security boundary. Nested jails work through the children.max parameter and strict subset inheritance. Modern tools like Bastille provide a Docker-adjacent workflow on top of the raw jail primitives.
