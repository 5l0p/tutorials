Elektron Synthesis - Part 1: Machinedrum and Monomachine Architecture
2026-06-01


The Elektron Machinedrum (2001) and Monomachine (2004) are hardware synthesisers that
organise their audio generation into discrete, swappable machines — synthesis
algorithms that can be assigned independently to each track. Understanding what algorithm
each machine type implements is the prerequisite for recreating it in Pure Data or as
a Pd external. This series reverse-engineers each machine family from the synthesis
architecture up.

The Machinedrum


The Machinedrum SPS-1 is a 16-track drum synthesiser with a step sequencer. Each of
the 16 tracks runs an independently selectable machine — a synthesis algorithm tuned
for a specific class of drum sound. The machine type determines which parameters
appear on the track's eight knobs and what sound is produced on trigger.



Four synthesis families are available:

TRXTurbo Resonant eXtendedAnalog-modeled subtractive synthesisParts 2–5
EFMExtended FMMulti-operator FM synthesisPart 6
E12Emulation 12Vector/PCM emulation of classic machinesPart 7
PIPitched InstrumentsPhysical modelling / resonant synthesisPart 8



Within each family, machines are organised by instrument role: BD (bass drum), SD (snare),
HH (hi-hat closed), XT (hi-hat open), CY (cymbal), CB (cowbell), CP (clap), RS (rimshot).
A TRX-BD and an EFM-BD produce a kick drum sound using entirely different synthesis
algorithms — you choose the family that matches the character you want.


The Monomachine


The Monomachine SFX60 is a 6-track synthesiser — not a drum machine, but a
general-purpose synthesiser with monophonic tracks. Each track runs one machine from
one of four synthesis engines, plus the universal filter, amplifier, and effects chain.

SIDSID machinesMOS6581/8580 chip emulation: tri/saw/pulse/noise, ring mod, sync, 4-pole filterPart 9
VOVirtual Operator2-operator FM: carrier, modulator, index ADSR, feedbackPart 10

Unlike the Machinedrum, all Monomachine tracks share the same downstream architecture:
a 2-pole multimode filter (LP/BP/HP), an amplifier with ADSR envelope, and a per-track
send to the effects bus. The machine type controls only the oscillator section.


Parameter Locks


Both instruments use parameter locks (p-locks): any parameter on a track can
be set differently for each sequencer step. This is the key to the Elektron sequencer's
expressive power — a kick can be tuned to a different pitch on each step, a hi-hat can
open on some steps and close on others, all without automation.



For Pd recreations, this maps to: all synthesis parameters must be settable at message
rate (not just at patch load time). Every parameter should be reachable by a
[receive] object so the sequencer (in Lua or another Pd patch) can lock
any parameter to any step value.


Common Pd Architecture for This Series


Every Pd recreation in this series follows the same top-level structure:



; Voice template:
[r $0-trig]   → trigger bang
[r $0-vel]    → velocity 0–1
[r $0-pN]     → parameter N (message rate, p-lockable)

; Synthesis section (machine-specific)

; Filter section (shared for MNM, absent for MD):
[r $0-cutoff] → VCF cutoff (0–1)
[r $0-res]    → VCF resonance (0–1)
[r $0-fenv]   → filter envelope amount (±1)

; Amp section:
[r $0-vol]    → output level
→ [vline~] ADSR → [*~] → [outlet~]



The synthesis section varies per machine type. The filter and amplifier are identical
across all Monomachine recreations.


Pd Patch vs Pd External


A Pd patch reconstruction uses only built-in Pd objects — [osc~],
[noise~], [lop~], [vline~] etc. This is fully
portable and requires no compilation.



A Pd external is a compiled C or C++ object loaded as a .pd_linux
or .dll. An external is appropriate when:



The algorithm requires sample-accurate state that is awkward to express in Pd dataflow
Performance matters (e.g. a many-oscillator FM machine where C is 10–50× faster than Pd)
The algorithm requires non-standard memory layouts (lookup tables, per-sample feedback)



The EFM (4-operator FM) and E12 (table-based emulation) articles use a Pd external
approach. The TRX, PI, GND, SFX, VO, and DIGI articles use vanilla Pd patches.


Parameter Naming Convention


The articles use abbreviated parameter names matching the Machinedrum and Monomachine
front panel abbreviations. Full parameter names are given at first mention. Where
the original parameter range is 0–127 (MIDI-style), the Pd implementation maps to 0–1
for consistency with the rest of this site.

Summary


The Machinedrum organises drum synthesis into four machine families — TRX
(analog-modeled subtractive), EFM (multi-operator FM), E12 (vector/PCM emulation),
and PI (physical modelling) — while the Monomachine offers a SID-chip emulation, two
FM engines (VO and FM+), a detuned-unison SuperWave engine, a phase-distortion and
wavetable DigiPro engine, and a built-in Percussion toolbox. Every machine type in this
series maps onto one common Pd voice template: message-rate, p-lockable parameter
receivers feeding a machine-specific synthesis section, an optional shared filter (on the
Monomachine), and a [vline~] amplitude envelope. Once you have that template,
each of the following articles is really just the synthesis section that plugs into it.
