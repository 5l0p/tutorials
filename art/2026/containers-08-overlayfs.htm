Containers - Part 8: OverlayFS & Image Layers

When you pull a Docker image, you are downloading a stack of tarballs. When you run a container, those tarballs are not extracted into a single directory. They are stacked using OverlayFS - a Linux kernel filesystem that presents a unified view of multiple directories without copying data. Writes go into a writable upper layer while the lower layers remain untouched. This article explains the kernel mechanics, shows how to create a layered filesystem from scratch, and maps Docker's layer model to OverlayFS mount options.

What You Will Learn

  • The four OverlayFS components: lowerdir, upperdir, workdir, and merged
  • How to create a layered filesystem from scratch using only mount
  • How copy-on-write works: modifications to lower-layer files appear in upperdir
  • How whiteout files implement deletion from read-only lower layers
  • How opaque directories handle directory deletion and recreation
  • How Docker maps image layers to OverlayFS lowerdir entries in /var/lib/docker/overlay2/
  • How to mount an OverlayFS programmatically in C and Go

OverlayFS Concepts

An OverlayFS mount has four components:

  • lowerdir - one or more read-only directories, stacked bottom to top. Files in higher layers shadow files in lower layers.
  • upperdir - a writable directory. All new files and modifications to lower-layer files are written here.
  • workdir - a scratch directory on the same filesystem as upperdir. Used internally by the kernel for atomic operations. Must be empty and on the same filesystem as upperdir.
  • merged - the unified view presented to the user. Reading from here shows the combined view; writes go to upperdir.

OverlayFS was merged into the mainline kernel in version 3.18 (December 2014). Before that, AUFS was the common alternative used by Docker.

Creating a Layered Filesystem from Scratch

No Docker required - just mount: 

#!/bin/sh
set -e

BASE=/tmp/overlay-demo
mkdir -p $BASE/{lower1,lower2,upper,work,merged}

# Populate the lower layers
echo "from lower1" > $BASE/lower1/fileA.txt
echo "from lower1" > $BASE/lower1/shared.txt
echo "from lower2" > $BASE/lower2/fileB.txt
echo "lower2 overrides lower1" > $BASE/lower2/shared.txt

# Mount the overlay
# lower layers are listed colon-separated, highest priority last→first
mount -t overlay overlay \
  -o lowerdir=$BASE/lower2:$BASE/lower1,\
upperdir=$BASE/upper,\
workdir=$BASE/work \
  $BASE/merged

echo "=== merged view ==="
ls $BASE/merged/
cat $BASE/merged/shared.txt   # shows lower2 version
cat $BASE/merged/fileA.txt    # shows lower1 version
cat $BASE/merged/fileB.txt    # shows lower2 version

echo "=== write a new file ==="
echo "new content" > $BASE/merged/newfile.txt
ls $BASE/upper/   # newfile.txt appears here; lower layers unchanged

echo "=== modify a lower-layer file ==="
echo "modified" > $BASE/merged/fileA.txt
ls $BASE/upper/   # fileA.txt also appears here (copy-on-write)
cat $BASE/lower1/fileA.txt  # original still "from lower1"

# Cleanup
umount $BASE/merged

The key observation: modifications go to upper; the lower directories are never written to. The merged view is the overlay of all layers.

Whiteout Files: Deleting from Lower Layers

When you delete a file that exists only in a lower layer, OverlayFS cannot remove it from the read-only lower dir. Instead it creates a whiteout entry in upperdir: a character device with major:minor 0:0. The kernel uses this to hide the lower-layer file from the merged view. 

# Delete fileA.txt through the merged layer
rm $BASE/merged/fileA.txt

# A whiteout file appears in upper
ls -la $BASE/upper/
# c--------- 1 root root 0, 0 ... fileA.txt

# fileA.txt is hidden in the merged view
ls $BASE/merged/  # no fileA.txt

# But the original is still in lower1
cat $BASE/lower1/fileA.txt  # still "from lower1"

Opaque Directories

When a directory is deleted and recreated through the overlay, a regular whiteout is insufficient (directories can have children). OverlayFS uses an opaque directory: a directory in upperdir with the extended attribute trusted.overlay.opaque=y. When the kernel sees this attribute, it does not look into the corresponding lower-layer directory. 

# Create a directory in merged, then delete and recreate it
mkdir $BASE/merged/testdir
touch $BASE/merged/testdir/lowerfile
rm -rf $BASE/merged/testdir
mkdir $BASE/merged/testdir

# Inspect the upper
getfattr -n trusted.overlay.opaque $BASE/upper/testdir
# trusted.overlay.opaque = "y"

How Docker Maps to OverlayFS

Docker (and containerd) stores image layers in /var/lib/docker/overlay2/. Each layer is a directory with diff/ (the layer content), merged/, work/, and a lower file that lists ancestor layer IDs. 

# Inspect a running container's OverlayFS mount
docker run -d --name test alpine sleep 3600
docker inspect test | grep -A 10 GraphDriver
# "Data": {
#   "LowerDir": "/var/lib/docker/overlay2/<id>/diff:...",
#   "MergedDir": "/var/lib/docker/overlay2/<id>/merged",
#   "UpperDir": "/var/lib/docker/overlay2/<id>/diff",
#   "WorkDir": "/var/lib/docker/overlay2/<id>/work"
# }

# See the actual mount
mount | grep overlay

Each Docker image layer is one entry in lowerdir (colon-separated). The container's writable layer is upperdir. When you commit a container to an image, Docker tarballs upperdir and stores it as a new layer.

Multiple Lower Layers: Ordering

The lowerdir option accepts a colon-separated list. The kernel applies them in order: the first entry in the list has the highest priority (is checked first for a given path). So: 

lowerdir=/layer3:/layer2:/layer1

Here /layer3 shadows /layer2 which shadows /layer1. Docker builds image layers oldest-first, so the newest Dockerfile instructions (higher priority) come first in the list.

Mounting OverlayFS Programmatically in C

#define _GNU_SOURCE
#include <stdio.h>
#include <sys/mount.h>

int mount_overlay(const char *lower, const char *upper,
                  const char *work,  const char *merged) {
    char opts[1024];
    snprintf(opts, sizeof(opts),
             "lowerdir=%s,upperdir=%s,workdir=%s",
             lower, upper, work);

    if (mount("overlay", merged, "overlay", 0, opts) == -1) {
        perror("mount overlay");
        return -1;
    }
    return 0;
}

Multiple lower layers with colon separation: 

snprintf(opts, sizeof(opts),
         "lowerdir=%s:%s:%s,upperdir=%s,workdir=%s",
         layer3, layer2, layer1, upper, work);

In Go

package main

import (
    "fmt"
    "golang.org/x/sys/unix"
)

func mountOverlay(lower, upper, work, merged string) error {
    opts := fmt.Sprintf(
        "lowerdir=%s,upperdir=%s,workdir=%s",
        lower, upper, work,
    )
    return unix.Mount("overlay", merged, "overlay", 0, opts)
}

Kernel Version Requirements

OverlayFS was mainlined in Linux 3.18. Multiple lower directories (the colon syntax) require Linux 4.0. Index mode (for NFS export support) requires 4.13. Most production systems running Linux 5.x or later have full support. Check with: 

grep overlay /proc/filesystems
# nodev   overlay

Summary

OverlayFS stacks read-only lower directories under a writable upperdir and presents a unified merged view. Reads walk down through layers; writes go to upperdir. Deletions create whiteout entries (char device 0:0). Docker stores each image layer as a separate directory and mounts them as a colon-separated lowerdir list — newest layers first. The container's writable layer is upperdir. Committing a container tarballs upperdir into a new image layer.
