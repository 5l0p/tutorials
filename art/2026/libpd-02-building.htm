libpd - Part 2: Building libpd from Source

libpd ships as a CMake project and compiles cleanly on Linux, macOS, and Windows. This article walks through cloning the repository, building it with the recommended flags, and wiring it into a host application via CMake. Understanding the build structure saves time when you hit linker errors or need the optional "extra" objects.

What You Will Learn

  • How to clone the libpd repository and initialise its pure-data submodule
  • The CMake configuration flags: static vs shared, threading model, "extra" externals
  • How to build libpd on Linux, macOS, and Windows
  • How to wire the built library into your own CMake project with find_package or add_subdirectory
  • What libpd replaces in the Pd source (s_audio.c stub) and what runs unchanged

Cloning the Repository

libpd's repository contains the pure-data upstream Pd source as a Git submodule. Always initialise submodules after cloning:

git clone https://github.com/libpd/libpd.git
cd libpd
git submodule update --init --recursive

The resulting tree looks like:

libpd/
  CMakeLists.txt
  libpd_wrapper/      <-- libpd C API layer
  pure-data/          <-- upstream Pd source (submodule)
    src/              <-- Pd scheduler, objects, DSP core
  cpp/                <-- optional C++ wrapper (PdBase.h)
  java/               <-- Android/Java wrapper
  csharp/             <-- C# wrapper
  samples/            <-- minimal usage examples

The libpd_wrapper/ directory contains the glue that turns the Pd internals into the stable public API exposed through z_libpd.h.

CMake Build Options

The main flags you need to know:

  • LIBPD_EXTRA=ON - include the extra/ object library (zexy, cyclone-style objects, etc.).
  • LIBPD_SHARED=ON - build a shared library (.so/.dylib/.dll) instead of the default static library.
  • LIBPD_SETLOCALE=OFF - do not let libpd call setlocale; important when embedding in a host that manages locale itself.
  • PD_FLOATSIZE=64 - use 64-bit float throughout Pd's signal chain (default is 32-bit). Rarely needed.

A typical release build targeting a static library:

cmake -B build -DLIBPD_EXTRA=ON -DLIBPD_SHARED=OFF \
      -DCMAKE_BUILD_TYPE=Release
cmake --build build --parallel

After the build, you find:

build/libpd.a          (Linux/macOS static)
build/libpd.lib        (Windows static)

Header Layout

After building, add libpd/libpd_wrapper and libpd/pure-data/src to your include path. The public headers are:

z_libpd.h       <-- main API
z_hooks.h       <-- callback hook types
m_pd.h          <-- Pd internals (rarely needed directly)

Only include z_libpd.h and z_hooks.h in application code. Including m_pd.h directly pulls in Pd internals and can cause symbol conflicts.

Integrating into Your CMake Project

The cleanest approach is to add libpd as a subdirectory or use FetchContent. Here is a complete CMakeLists.txt for a host application using add_subdirectory:

cmake_minimum_required(VERSION 3.21)
project(MyAudioApp C CXX)

set(CMAKE_C_STANDARD 11)
set(CMAKE_CXX_STANDARD 17)

# --- libpd ---
set(LIBPD_EXTRA ON  CACHE BOOL "" FORCE)
set(LIBPD_SHARED OFF CACHE BOOL "" FORCE)
add_subdirectory(deps/libpd)

# --- host application ---
add_executable(myapp
    src/main.cpp
    src/audio.cpp
)

target_link_libraries(myapp PRIVATE libpdstatic)

target_include_directories(myapp PRIVATE
    deps/libpd/libpd_wrapper
    deps/libpd/pure-data/src
)

The CMake target name is libpdstatic (static) or libpd (shared). Check libpd's own CMakeLists.txt for the exact target name in the version you are using - it has changed across versions.

Using FetchContent

If you prefer not to vendor libpd as a submodule, CMake's FetchContent can download it at configure time:

include(FetchContent)

FetchContent_Declare(
    libpd
    GIT_REPOSITORY https://github.com/libpd/libpd.git
    GIT_TAG        main
    GIT_SUBMODULES "pure-data"
)

set(LIBPD_EXTRA ON CACHE BOOL "" FORCE)
FetchContent_MakeAvailable(libpd)

target_link_libraries(myapp PRIVATE libpdstatic)
target_include_directories(myapp PRIVATE
    ${libpd_SOURCE_DIR}/libpd_wrapper
    ${libpd_SOURCE_DIR}/pure-data/src
)

Static vs Shared Linking

Choose static when distributing a standalone binary. The libpd code is baked in; no separate .so to deploy. This is the right choice for VST plugins and embedded applications.

Choose shared when multiple processes on the same machine use libpd (saves RAM by sharing one copy) or when you want to update the Pd engine independently of the host binary. For a mobile app or a single-binary tool, static is almost always simpler.

On Linux, static libpd requires linking against -lm -lpthread -ldl as well:

target_link_libraries(myapp PRIVATE libpdstatic m pthread dl)

Windows-Specific Notes

On Windows, the Pd sources use WIN32-specific code paths. Build with Visual Studio 2019+ or MinGW. The CMake build handles this automatically. When using static linking on Windows, add ws2_32 to the linker libraries (some Pd objects use socket APIs):

if(WIN32)
    target_link_libraries(myapp PRIVATE ws2_32)
endif()

macOS Universal Binaries

To build a fat binary for Apple Silicon + Intel when targeting macOS:

cmake -B build \
    -DCMAKE_OSX_ARCHITECTURES="arm64;x86_64" \
    -DLIBPD_EXTRA=ON \
    -DCMAKE_BUILD_TYPE=Release

Verifying the Build

The samples/ directory contains a minimal test program. Build and run it to confirm everything links:

cd build
./samples/pdtest

If it exits cleanly, the library is correctly built. If it prints "error opening patch", check that the sample .pd file path is correct.

The pure-data Submodule Structure

The pure-data/src/ directory is the heart of the engine:

m_pd.h          <-- object/message type definitions
m_sched.c       <-- scheduler: drives tick computation
d_*.c           <-- DSP (signal) objects: filters, osc, arithmetic
g_*.c           <-- graphical objects (used headlessly by libpd)
s_audio.c       <-- audio system (stubbed out by libpd)
x_*.c           <-- control objects: math, logic, MIDI, etc.

libpd replaces s_audio.c with its own stub that routes audio through the libpd_process_float API instead of talking to a sound device. Everything else runs unchanged from desktop Pd.

Summary

Clone libpd with --recurse-submodules to pull the embedded pure-data source. Configure with CMake: set PD_MULTI_INSTANCE if you need multiple engines, PD_EXTRA to include community externals, and BUILD_SHARED_LIBS to choose static vs shared. Link your CMake target against libpd and add the include directory for libpd.h. Everything else in the Pd source compiles unchanged — only s_audio.c is replaced with the headless stub.
